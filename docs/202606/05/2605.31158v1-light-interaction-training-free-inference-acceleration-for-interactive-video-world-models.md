## 摘要
交互式视频世界模型能够响应用户控制的相机运动，逐块生成视频，从而支持实时游戏仿真、虚拟场景导航和具身 AI 训练等应用。然而，由于上下文内存不断增长、二次复杂度的注意力计算以及重复的去噪步骤，扩展到长交互轨迹的成本极其高昂。我们提出 Light Interaction，这是一种面向交互式视频世界模型的免训练推理加速框架。我们的核心洞察是，交互天然支持依赖于轨迹的自适应计算：在探索新区域时可以丢弃检索到的空间记忆，可以根据局部潜在动态调整时间上下文，并且当相机重访熟悉区域时可以复用早期步骤的模型输出。基于这一洞察，Light Interaction 结合了自适应上下文管理、去噪缓存加速，以及软硬件协同设计的 3D 块稀疏注意力和融合 Triton 内核。在 HY-WorldPlay 和 Matrix-Game-3.0 上的评估表明，Light Interaction 在无需模型重训练的情况下实现了最高 2.59 倍的加速，同时保持了具有竞争力的视觉质量。

## Abstract
Interactive video world models generate video chunk by chunk in response to user-controlled camera movements, enabling applications such as real-time game simulation, virtual scene navigation, and embodied AI training. However, scaling to long interactive trajectories is prohibitively expensive due to growing context memory, quadratic attention complexity, and repeated denoising steps. We present Light Interaction, a training-free inference acceleration framework for interactive video world models. Our key insight is that interaction naturally enables trajectory-dependent adaptive computation: retrieved spatial memory can be discarded during novel exploration, temporal context can be adjusted according to local latent dynamics, and early-step model outputs can be reused when the camera revisits familiar regions. Based on this insight, Light Interaction combines adaptive context management, denoising cache acceleration, and hardware-software co-designed 3D block sparse attention with fused Triton kernels. Evaluated on HY-WorldPlay and Matrix-Game-3.0, Light Interaction achieves up to 2.59x speedup without model retraining while maintaining competitive visual quality.

## 2. 方法论：核心思想、关键技术细节与流程

Light Interaction 包含三个主要模块：

1. **自适应上下文管理**；
2. **轻量级去噪缓存加速**；
3. **软硬件协同设计的 3D 块稀疏注意力**。

整体目标是在不训练、不修改模型参数的前提下，减少推理阶段的冗余计算。

---

### 2.1 自适应上下文管理

交互式视频世界模型中的上下文主要包括两类：

- **时间上下文**：最近生成的视频帧或 chunk，用于维持短期运动连续性；
- **空间检索记忆**：从较长历史中根据相机位姿相似性检索出的历史视角，用于维持长程几何一致性。

论文认为，这两类上下文并非始终有用，应根据当前状态动态调整。

---

#### 2.1.1 时间上下文自适应机制

论文用历史 latent 表征之间的差异估计局部动态强度。

设第 \(t\) 个历史单元的 latent 表示为 \(z_t\)，则局部动态定义为：

\[
D_t = MSE(z_{t-1}, z_{t-2})
\]

其中 MSE 在所有 latent 维度上求平均。

为了减少短期波动，论文使用指数滑动平均：

\[
\bar{D}_t = \alpha D_t + (1-\alpha)\bar{D}_{t-1}
\]

然后根据动态强度决定保留的时间窗口长度：

\[
L_t = clip\left(
\left\lfloor
L_m \cdot \frac{\kappa}{\bar{D}_t + \kappa}
\right\rfloor,
1,
L_m
\right)
\]

含义是：

- 当局部动态较大时，历史上下文可能快速失效，因此缩短时间窗口；
- 当局部动态较小时，历史上下文更稳定，因此扩大时间窗口。

不过附录中说明，在实际实现里：

- HY-WorldPlay 使用了简化版本，通常只保留最近一个时间单元；
- Matrix-Game-3.0 当前未启用完整参数化时间窗口自适应机制。

---

#### 2.1.2 空间检索记忆自适应机制

空间记忆依赖相机位姿相似度。设当前时间 \(t\) 与第 \(j\) 个历史候选的位姿相似度为：

\[
S_{pose}(t,j)
\]

如果最大相似度低于阈值：

\[
\max_j S_{pose}(t,j) < \tau_{pose}
\]

则判定当前处于**探索新区域**阶段，此时检索到的历史记忆可能并不可靠，因此丢弃空间记忆。

如果：

\[
\max_j S_{pose}(t,j) \geq \tau_{pose}
\]

则判定当前处于**重访已有区域**阶段，保留空间检索记忆用于维持几何一致性。

在 HY-WorldPlay 和 Matrix-Game-3.0 中，论文使用的是基于 FoV 的相似度信号 \(S_{FOV}\)，阈值分别为：

- HY-WorldPlay：0.7；
- Matrix-Game-3.0：0.45。

---

### 2.2 轻量级去噪缓存加速

交互式视频生成模型通常使用很少的去噪步数，例如 \(K \leq 4\)。论文观察到：

- 在探索新区域时，生成结果不确定性更高，不同去噪步之间变化较大；
- 在重访已有区域时，历史空间记忆提供了较强约束，不同步之间的模型输出更稳定。

因此，论文只在“重访”状态下启用去噪缓存复用。

判定条件仍然使用相机位姿相似度：

\[
\max_j S_{pose}(t,j) \geq \tau_{pose}
\]

当满足该条件时，模型执行如下流程：

1. 正常计算第一个去噪步的模型输出；
2. 对中间去噪步复用第一步输出；
3. 最后一个去噪步仍然正常计算，用于修正累计误差。

设第 \(i\) 步模型输出为：

\[
v_\theta(x_i, t_i, c)
\]

其中 \(c\) 表示上下文和相机条件。对于 \(K\) 步去噪过程，中间步骤近似为：

\[
v_\theta(x_i, t_i, c) \approx v_\theta(x_0, t_0, c),
\quad i \in \{1,\dots,K-2\}
\]

如果 \(K \leq 2\)，则没有中间步可以复用，因此不启用该机制。

该设计的关键点在于：

- 不盲目复用所有场景；
- 只在重访区域中复用；
- 保留最后一步完整计算以保证质量。

### 2.3 软硬件协同设计的 3D 块稀疏注意力

普通稀疏注意力方法在交互式自回归视频生成中效果有限，原因包括：

- 自回归生成存在因果约束；
- 当前 query 和历史 KV 长度不对称；
- 稀疏块 gather / scatter 和 layout conversion 的开销可能抵消理论加速收益。

因此，论文提出面向自回归视频生成的 3D 块稀疏注意力实现。

---

#### 2.3.1 稀疏策略

Light Interaction 将视觉 token 按时间、高度、宽度划分为 3D 块：

\[
(B_t, B_h, B_w)
\]

并且只对**历史视觉 KV cache** 做稀疏化。

以下部分始终保留完整注意力：

- 文本条件 token；
- 当前 denoising chunk / frame 的 token。

这样做可以减少质量损失，同时降低历史长上下文带来的注意力开销。

对于每个 query block \(Q_i\)，论文计算 query block 和历史 KV block 的池化表示：

\[
\bar{q}_i = \frac{1}{|Q_i|}\sum_{j \in Q_i}q_j
\]

\[
\bar{k}_m = \frac{1}{|K_m|}\sum_{n \in K_m}k_n
\]

然后通过块级相似度选择 Top-\(r\) 的历史块：

\[
I_i = Top_{\lfloor rM \rfloor}
\left\{
\frac{\bar{q}_i^\top \bar{k}_m}{\sqrt{d}}
\right\}_{m=1}^{M}
\]

最终每个 query block 的注意力上下文为：

\[
C_i = T \cup K_{curr} \cup S_i
\]

其中：

- \(T\)：文本 token；
- \(K_{curr}\)：当前 chunk 的 KV；
- \(S_i\)：被选中的历史视觉 KV 块。

---

#### 2.3.2 Triton 融合 Kernel

论文进一步提出三个融合 Triton kernel，减少稀疏注意力周边操作开销：

- **Fused Q-Preparation**
  - 完成 query block 切分；
  - 计算 query block 池化表示；
  - 生成 block-major layout；
  - 避免额外 padding 和 copy。

- **Fused KV-Preparation**
  - 同时读取 K 和 V；
  - 完成 3D block 切分和 layout 转换；
  - 计算 pooled K；
  - 保留文本 token 区域；
  - 将视觉块连续写入缓存。

- **Fused Untile Scatter**
  - 将稀疏注意力输出恢复到原始自回归 token 顺序；
  - 通过 masked store 丢弃边界 padding 位置。

这部分是论文中较强的系统工程贡献，因为它解决了“算法上稀疏但实际不加速”的问题。

- 基线按官方配置适配；
- 稀疏方法尽量匹配 sparse computation volume；
- 在相同模型和相同评估集上比较；
- 同时报告质量和效率指标，避免只看速度。

但也存在潜在偏差：

- 不同稀疏方法适配到自回归场景的工程优化程度可能不同；
- Light Interaction 有专门设计的 Triton 融合 kernel，而部分基线可能没有同等程度系统优化；
- 主结果使用作者选择的阈值和 retained ratio，虽然有超参数实验，但泛化到其他场景仍需验证。

---

## 6. 主要结论与发现

### 6.1 整体性能

在 **HY-WorldPlay** 上，Light Interaction 实现：

- 延迟从 **228.60 s** 降至 **88.24 s**；
- 加速比为 **2.59×**；
- 峰值显存从 **76.57 GB** 降至 **54.66 GB**；
- 相对原始模型：
  - PSNR 为 **24.81**；
  - SSIM 为 **0.6500**；
  - LPIPS 为 **0.1788**；
- VBench 为 **0.8220**，高于原始模型的 0.8190。

在 **Matrix-Game-3.0** 上，Light Interaction 实现：

- 延迟从 **59.70 s** 降至 **37.07 s**；
- 加速比为 **1.61×**；
- 相对原始模型：
  - PSNR 为 **17.76**；
  - SSIM 为 **0.5306**；
  - LPIPS 为 **0.3692**；
- VBench 为 **0.7350**。

### 6.2 与基线比较

论文发现：

- SVG 和 BSA 在普通视频生成中有用，但在自回归交互式视频生成中加速有限，甚至可能变慢；
- TeaCache 在部分指标上质量较好，但速度提升有限，且 VBench 不一定更高；
- Light Interaction 在 HY-WorldPlay 上取得最优质量-效率权衡；
- 在 Matrix-Game-3.0 上取得最快速度，同时保持有竞争力的视频质量。

### 6.3 模块贡献

论文的消融实验显示：

- **时间上下文管理**对降低延迟和显存贡献较大；
- **空间上下文管理**对保持与原始模型的输出一致性更有帮助；
- **去噪缓存**作为轻量补充模块，进一步减少 Transformer 前向次数；
- **3D 稀疏注意力**是主要加速来源之一；
- 三个模块组合后效果最好，说明它们具有互补性。

### 6.4 系统实现发现

论文还发现：

- 稀疏注意力如果没有硬件友好的实现，理论加速可能无法转化为实际加速；
- Triton kernel 融合可以显著降低 Q/KV 准备和输出恢复开销；
- 在 HY-WorldPlay 上，kernel fusion 使 sparse-attention 相关部分获得约 **1.40×** 加速。

---

## 7. 优点与亮点

### 7.1 方法层面的优点

- **无需训练**
  - 不需要重新训练或微调大模型；
  - 容易部署到已有交互式视频世界模型上。

- **利用交互轨迹信息**
  - 将相机位姿相似度用于上下文选择和去噪缓存；
  - 区分探索与重访状态，而不是使用固定策略。

- **上下文管理更精细**
  - 同时处理时间上下文和空间检索记忆；
  - 避免在探索阶段引入无关历史记忆。

- **去噪缓存更稳健**
  - 不盲目复用所有步骤；
  - 只在几何上可靠的重访状态启用；
  - 保留最终去噪步骤做质量修正。

- **稀疏注意力适配自回归场景**
  - 保留文本 token 和当前 chunk token；
  - 只稀疏历史视觉 KV；
  - 更符合交互式视频生成的因果约束。

### 7.2 实验与系统层面的优点

- 覆盖两个代表性开源世界模型；
- 同时报告质量、速度和显存；
- 包含充分的模块消融和超参数分析；
- 提供阶段延迟分解，能够解释速度提升来源；
- 引入 Triton 融合 kernel，强调实际部署效率，而非仅报告理论稀疏度。

---

## 8. 不足与局限

### 8.1 方法依赖相机位姿相关信号

Light Interaction 的核心机制依赖相机位姿或 FoV 相似度。如果目标模型或应用场景中没有可靠的相机外参、位姿估计或几何相似度信号，则需要额外近似。

这会带来：

- 相似度估计误差；
- 错误判断探索 / 重访状态；
- 误删有用空间记忆；
- 在不该复用时复用去噪输出。

---

### 8.2 去噪缓存只验证于短步数模型

论文明确指出，去噪缓存机制主要验证在短步去噪模型上，即：

\[
K \leq 4
\]

对于去噪步数更多的扩散模型，该策略是否仍然稳定尚未充分验证。

---

### 8.3 模型覆盖范围有限

实验仅覆盖两个交互式视频世界模型：

- HY-WorldPlay；
- Matrix-Game-3.0。

虽然它们具有代表性，但还不足以证明该方法能泛化到所有：

- 文生视频模型；
- 图生视频模型；
- 3D-aware 视频模型；
- 具身智能仿真模型；
- 不同注意力架构或记忆机制的世界模型。

---

### 8.4 硬件依赖较强

论文的加速效果部分来自 Triton fused kernel 和特定内存布局优化。因此实际收益可能依赖：

- GPU 架构；
- 显存带宽；
- 模型 token layout；
- KV cache 组织方式；
- 自回归执行框架。

在非 A100 GPU 或消费级显卡上，加速比可能不同。

---

### 8.5 基线优化程度可能不完全对等

Light Interaction 针对交互式自回归世界模型做了专门设计和 kernel 优化，而部分基线方法主要源自普通视频生成或非自回归生成场景。

因此，虽然论文进行了适配和计算量匹配，但仍可能存在：

- 基线没有充分工程优化；
- 稀疏方法在当前框架中额外开销更高；
- 系统实现层面对比不完全对等。

---

### 8.6 评价仍以自动指标为主

论文主要使用：

- PSNR；
- SSIM；
- LPIPS；
- VBench；
- Self-Comparison。

这些指标能够衡量一定程度的图像质量和一致性，但对交互式世界模型而言，还可能需要更多评估，例如：

- 人类主观评价；
- 交互响应体验；
- 几何一致性定量测量；
- 长时间导航稳定性；
- 任务驱动的具身智能性能。

---

## 总结

Light Interaction 是一篇面向交互式视频世界模型实际部署瓶颈的推理加速论文。它的核心贡献在于将交互轨迹中的探索 / 重访状态转化为自适应计算策略，并结合上下文管理、去噪缓存和硬件友好的 3D 稀疏注意力，实现了无需训练的显著加速。

从实验结果看，该方法在 HY-WorldPlay 上最高达到 **2.59×** 加速，并显著降低显存占用；在 Matrix-Game-3.0 上也取得 **1.61×** 加速。整体上，论文方法设计合理、系统实现扎实、实验较完整，尤其适合低延迟交互式视频生成和世界模型部署场景。

其主要局限在于依赖可靠的相机位姿相似度信号、实验模型数量有限、硬件实现依赖较强，以及缺少更广泛的人类交互体验评估。
