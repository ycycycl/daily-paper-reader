## 摘要
视觉-语言-动作（Vision-Language-Action, VLA）模型仍面临推理速度慢、难以执行细粒度运动调整等挑战，限制了其在工业中的广泛应用。尽管实时动作分块（Real-Time Action Chunking, RTAC）算法已被提出用于解决这些瓶颈，但如何弥合从伪代码形式的算法到在低成本机械臂上实现稳定真实部署之间的差距，仍然是一项挑战。在本研究中，我们提出了一种面向低成本机器人操作系统的完整系统级 RTAC 实现。我们在原始高层伪代码的基础上进一步推进，通过优化策略推理与控制流水线的线程实现，降低端到端延迟并提升响应性，同时不修改底层策略。我们在农业产品操作任务上对该系统进行了评估，具体包括大蒜鳞茎和核桃。实验结果表明，与 RTAC 的基础实现相比，我们定制的线程实现显著提升了控制稳定性和速度。

## Abstract
Vision-Language Action (VLA) models continue to face challenges such as slow inference speed and difficulty performing fine-grained motion adjustments, limiting their widespread adoption in industry. While the Real-Time Action Chunking (RTAC) algorithm has been proposed to address these bottlenecks, bridging the gap between the algorithm provided in pseudocode to a stable, real-world deployment on a low-cost robotic arm remains a challenge. In this work, we present a complete system-level implementation of RTAC tailored for a low-cost robotic manipulation system. We advance beyond the original high-level pseudocode by optimizing the threading implementation for the policy inference and control pipeline, reducing end-to-end latency and improving responsiveness without modifying the underlying policy. We evaluate this system on tasks involving the manipulation of agricultural produce, specifically garlic bulbs and walnuts. Experimental results demonstrate that our custom threading implementation significantly improves control stability and speed compared to the base implementation of RTAC.

## 2. 方法论：核心思想、关键技术细节与算法流程

### 2.1 方法核心思想

- 论文提出一种面向低成本农业机械臂的 **优化版 RTAC 系统实现**。
- 核心目标是在**不修改底层 VLA 策略模型**的前提下，通过优化推理与控制之间的线程调度，降低端到端延迟，提高动作执行的连续性和稳定性。
- 系统采用一种生产者-消费者式架构：
  - **推理线程**负责根据视觉、语言和机器人状态生成动作块；
  - **执行线程**负责将动作块拆分为单个动作并持续发送给真实机械臂；
  - **Action Broker**作为中间模块，负责动作块缓存、分发和执行协调。

### 2.2 VLA 动作生成基础

- 本文使用的 VLA 管线基于：
  - **PaliGemma LoRA**：使用作者自制 3D 打印 GELLO 遥操作臂采集的数据进行微调；
  - **π0 Action Expert**：用于生成连续动作块。
- 输入包括：
  - 自然语言指令，例如“Pick up the garlic and put it into the bowl”；
  - 双相机视觉观测；
  - 机器人 7D 状态，即 6 个机械臂关节加 1 个夹爪自由度。
- 输出是一个动作块：

  \[
  A_t = [a_t, ..., a_{t+H-1}]
  \]

  表示未来一段时间内的连续动作序列。

### 2.3 π0 的动作流建模

- π0 将动作生成建模为连续时间概率流，通过学习速度场：

  \[
  v_\theta(A_t^\tau, o_t)
  \]

  将高斯噪声逐步映射为动作轨迹。
- 推理时使用类似 Euler ODE solver 的迭代更新：

  \[
  A_t^{\tau+\delta} = A_t^\tau + \delta v_\theta(A_t^\tau, o_t)
  \]

- 这意味着动作块不是一次性直接输出，而是通过流匹配过程逐步生成。

### 2.4 RTAC 的时间 inpainting 机制

- RTAC 的关键在于：当前动作块正在执行时，系统异步生成下一个动作块。
- 为避免新旧动作块之间不连续，RTAC 使用 guided flow-matching temporal inpainting。
- 新生成的动作块需要在重叠时间步上与当前正在执行的轨迹 \(Y\) 对齐。
- 论文引用的 RTAC 修正速度场形式为：

  \[
  v_{\Pi GDM}(A_t^\tau, o_t, \tau)
  =
  v(A_t^\tau, o_t, \tau)
  +
  \text{guidance term}
  \]

- 其中 guidance term 会将预测动作块在被 mask 的重叠部分拉向当前轨迹 \(Y\)，从而保证动作连续。
- 文中还使用：
  - 近似最终动作块 \( \hat{A}_t^1 \)；
  - 方差缩放调度 \( r^2_{>\tau} \)；
  - 稳定性超参数 \( \beta \)；
  - Jacobian 项用于传播空间约束。

### 2.5 系统级线程优化

- 原始 RTAC 伪代码在真实 Python 机器人系统中可能面临：
  - GIL 导致 Python 字节码无法真正并行执行；
  - I/O、GPU 推理、WebSocket 通信、机器人控制之间互相阻塞；
  - 控制线程等待推理结果，导致机械臂动作中断或抖动。
- 作者提出优化后的双线程架构：
  - **Thread 0：Inference Thread**
    - 负责 guided inference；
    - 接收语言、图像、机器人状态；
    - 调用 JAX JIT 编译后的 inpainting / RTAC 推理；
    - 生成新的动作块。
  - **Thread 1：Execution / Worker Thread**
    - 持续从 Action Broker 获取动作；
    - 如果动作块中仍有动作，则执行下一步；
    - 如果暂时无动作，则短暂 idle，例如 1 ms；
    - 将动作发送给 Fairino FR5 机械臂。
- 系统通过 **concurrent.futures.ThreadPoolExecutor** 管理线程调度。
- 作者认为该设计能够利用以下特性减少阻塞：
  - 控制循环中大量 I/O 操作会释放 GIL；
  - GPU 上的 JAX 推理 kernel 执行期间 Python 线程可继续处理其他任务；
  - ThreadPoolExecutor 能更稳定地管理任务调度；
  - JAX JIT 编译减少推理和 inpainting 的计算开销。
- 此外，策略服务和运行时控制端通过本地 **WebSocket** 通信：
  - `serve_policy.py` 托管策略服务器；
  - `main.py` / `runtime.py` 运行机器人控制逻辑；
  - `pi0_rtac.py` 负责 RTAC 动作块生成。

  - 没有与更多系统级方案比较，例如 multiprocessing、C++ 控制桥、ROS2 实时控制、RTOS、异步进程架构等；
  - 没有详细消融实验，例如分别分析 ThreadPoolExecutor、JAX JIT、Action Broker、WebSocket 本地化等组件的贡献。
- 因此，实验对于“工程可行性展示”是有说服力的，但对于“广泛适用性和统计可靠性证明”仍偏有限。

### 5.3 公平性与客观性

- 对比方法使用相同硬件平台、相同任务、相同底层 VLA 策略，整体上较公平。
- 本文强调不修改底层策略，仅优化线程实现，因此对比重点清晰。
- 但由于缺少重复次数、方差、失败案例和成功率，客观性仍有提升空间。

## 6. 主要结论与发现

- 优化后的 RTAC 线程实现相较标准 π0 和基础 RTAC 都能降低任务执行时间。
- 在大蒜任务中：
  - Standard π0 任务完成时间：36.818 s；
  - Standard RTAC：29.152 s；
  - Optimized RTAC：27.409 s。
- 论文报告 Optimized RTAC：
  - 相比 Standard π0，任务完成时间提升约 25.6%；
  - 相比 Standard RTAC，任务完成时间提升约 6.0%。
- 在大蒜完整 episode 时间上：
  - Standard π0：38.800 s；
  - Standard RTAC：35.548 s；
  - Optimized RTAC：34.764 s。
- 在核桃任务中：
  - Standard π0 任务完成时间：35.268 s；
  - Standard RTAC：33.323 s；
  - Optimized RTAC：31.070 s。
- 论文报告 Optimized RTAC：
  - 相比 Standard π0，任务完成时间提升约 11.9%；
  - 相比 Standard RTAC，任务完成时间提升约 6.8%。
- 在核桃完整 episode 时间上：
  - Standard π0：41.711 s；
  - Standard RTAC：37.001 s；
  - Optimized RTAC：36.331 s。
- 主要发现是：
  - RTAC 本身能够通过异步动作块机制降低 VLA 推理延迟带来的影响；
  - 但在真实部署中，系统级线程实现会进一步影响控制性能；
  - 通过更合理地解耦推理和执行，可以在不改变策略模型的前提下提升机器人操作速度和稳定性。

## 7. 优点与亮点

- **关注真实部署瓶颈**
  - 论文没有停留在算法层面，而是聚焦 VLA 和 RTAC 在真实低成本机械臂上的系统实现问题。

- **不修改底层策略**
  - 方法不依赖重新训练或修改 VLA 模型，而是通过系统架构优化获得性能提升，具有较强工程实用性。

- **低成本硬件平台**
  - 使用 Fairino FR5、RealSense 相机、Jodell 夹爪和 3D 打印 GELLO 遥操作设备，整体系统成本低于许多高端机器人平台，有助于降低 VLA 机器人研究门槛。

- **双线程生产者-消费者设计清晰**
  - 将高延迟 VLA 推理与高频机器人执行解耦，符合实时机器人系统设计原则。

- **真实农业操作验证**
  - 实验不是仿真，而是真实机器人完成大蒜和核桃操作，体现了实际应用价值。

- **对 RTAC 工程实现有补充价值**
  - 原始 RTAC 主要提供理论和伪代码，本文补充了 Python、JAX、WebSocket、Action Broker、线程调度等部署细节。

## 8. 不足与局限

- **实验规模较小**
  - 仅包含两个农业任务，且物体类型只有大蒜和核桃，无法充分说明方法对更多物体、更多操作类型的泛化能力。

- **缺少成功率和统计方差**
  - 论文主要报告单次或汇总时间结果，没有明确给出重复实验次数、均值、标准差、置信区间或统计显著性。

- **缺少系统消融实验**
  - 没有分别验证 ThreadPoolExecutor、JAX JIT、Action Broker、本地 WebSocket 等组件各自带来的收益。

- **对 GIL 绕过的表述仍需更严谨**
  - 文中指出使用 concurrent.futures 和 GPU/JAX 执行期间释放 GIL 来提升并发，但这并不等同于完全消除 Python GIL 的限制。
  - 论文结论中还提到“process-based threading architecture”，与正文主要描述的 ThreadPoolExecutor / threading 架构略有不一致，可能需要进一步澄清。

- **未与更强实时系统方案比较**
  - 没有比较 multiprocessing、C++ 控制层、ROS2 real-time executor、RTOS 或硬件实时控制桥等替代实现。

- **算力与训练细节披露不足**
  - 虽然说明部署使用 RTX 4090 24GB，但没有说明模型训练或 LoRA 微调的数据量、训练时间、训练超参数等。

- **应用范围仍受限**
  - 当前方法依赖特定机械臂、夹爪、相机布局和本地工作站部署。
  - 若迁移到边缘设备、多机器人系统、远程策略服务器或更复杂农业环境，可能会遇到新的延迟和同步问题。

- **评估指标偏单一**
  - 主要关注时间效率，缺少轨迹误差、控制频率、延迟分布、动作抖动量化、安全性指标和失败模式分析。
