## 摘要
具身大语言模型（LLM）正日益被用作机器人控制流水线中的推理模块，以改善人机交互，但其内存占用和生成延迟使实时部署变得困难。剪枝可以降低这些成本，但对于需要经历多个预训练和后训练阶段的控制器而言，关键问题不仅在于剪枝多少，还在于何时进行剪枝。在本文中，我们提出 Before Parc Fermé（BPF），一种在强化学习（RL）过程中执行的剪枝策略，可在具身 LLM 控制器仍在针对闭环行为进行优化时对其进行压缩。这使得剪枝决策能够考虑塑造最终控制器的任务特定监督和闭环反馈。我们提出两个变体：BPF-RL，即在 RL 过程中按预定义训练间隔移除模型的一部分来进行迭代剪枝；以及 BPF-SFT/RL，即先在监督微调（SFT）阶段剪除部分模型结构，然后在 RL 过程中采用与 BPF-RL 相同的迭代策略进一步压缩，直至达到目标剪枝率。我们在 RobotxR1（一种基于 LLM 的自动驾驶控制流水线）上评估 BPF，使用成熟的 LLM 剪枝框架（LLM-Pruner），并将其与后训练剪枝、带 RL 恢复的后训练剪枝、SFT 阶段剪枝以及同一模型族中的较小稠密模型进行比较。我们的结果表明，在所考察的剪枝策略中，BPF 在任务性能与内存及吞吐量之间提供了最佳权衡。在压缩较大的 RobotxR1 模型时，BPF-SFT/RL 在规模—端到端性能权衡方面比直接选择同一模型族中的较小稠密模型提高了 $1.69\times$，该指标以每损失一个百分点控制适应性所移除的参数量衡量。在安装于目标机器人平台上的 Jetson AGX Orin 上，紧凑模型将解码吞吐量最高提升了 $27\%$。

## Abstract
Embodied Large Language Models (LLMs) are increasingly used as reasoning modules in robotic control pipelines to improve human-robot interaction, but their memory and generation latency make real-time deployment difficult. Pruning can reduce these costs, but for controllers that undergo multiple pre- and post-training phases, the crucial question is not only how much to prune, but when pruning should occur.   In this work, we propose Before Parc Fermé (BPF), a pruning strategy performed during RL that compresses embodied LLM controllers while they are still being optimized for closed-loop behavior. This allows pruning decisions to account for the task-specific supervision and closed-loop feedback that shape the final controller. We propose two variants: BPF-RL, which performs iterative pruning during RL by removing part of the model at predefined training intervals, and BPF-SFT/RL, which first prunes part of the model structure during SFT and then further compresses it during RL using the same iterative strategy as BPF-RL until the target pruning ratio is reached. We evaluate BPF on RobotxR1, an LLM-based autonomous-driving control pipeline, using an established LLM pruning framework (LLM-Pruner), and compare it against post-training pruning, post-training pruning with RL recovery, SFT-stage pruning, and smaller dense models from the same family.   Our results show that BPF provides the best task-performance vs. memory and throughput trade-off among the considered pruning strategies. When compressing the larger RobotxR1 models, BPF-SFT/RL achieves a $1.69\times$ better size-end-to-end performance trade-off than directly selecting a smaller dense model from the same family, measured as removed parameters per lost percentage point of control adaptability. On the Jetson AGX Orin mounted on the target robotic platform, the compact models improve decode throughput by up to $27\%$.

## 2. 方法论：核心思想、关键技术细节与流程

### 2.1 核心思想：Before Parc Fermé（BPF）

- 论文提出 **Before Parc Fermé（BPF）** 剪枝策略。
- “Parc Fermé” 原指 F1 赛车中赛前车辆被封存、不可再修改的阶段。
- 在本文中，它比喻模型训练完成后结构固定的阶段。
- 因此，“Before Parc Fermé” 的含义是：

> 在模型结构最终锁定之前，也就是模型仍在通过 RL 进行闭环行为优化时进行剪枝。

- BPF 的核心假设是：
  - 如果在训练完成后才剪枝，模型已经收敛，剩余权重难以充分适应被移除结构；
  - 如果在 SFT 后、RL 前剪枝，剪枝过程无法利用闭环任务反馈；
  - 如果在 RL 过程中逐步剪枝，模型既已经学会任务格式，又仍能通过闭环奖励适应压缩后的结构。

### 2.2 被剪枝的系统：RobotxR1 流水线

- RobotxR1 由两个 LLM 模块级联组成：

#### DecisionxR1

- 输入：
  - 用户自然语言驾驶指令；
  - 近期车辆状态历史；
  - RAG 检索上下文与格式约束。
- 输出：
  - 判断当前驾驶行为是否符合用户指令。

可抽象表示为：

\[
y_D = f_{\theta_D}(c, h_t, z_D)
\]

其中：

- \(c\)：用户指令；
- \(h_t\)：车辆状态历史；
- \(z_D\)：检索上下文和提示信息；
- \(\theta_D\)：DecisionxR1 参数。

#### MPCxR1

- 输入：
  - 用户指令；
  - 车辆状态；
  - DecisionxR1 输出；
  - MPC 公式、约束和上下文。
- 输出：
  - 文本形式的 MPC 参数更新。

可表示为：

\[
y_M = f_{\theta_M}(c, h_t, y_D, z_M)
\]

其中：

- \(y_D\)：上游 DecisionxR1 输出；
- \(z_M\)：MPC 相关上下文；
- \(y_M\)：生成的控制器参数更新。

### 2.3 四类剪枝时机对比

论文系统比较了不同剪枝时机。

#### 1. Post-Training Pruning（PTP）

- 先完整完成 SFT 和 RL；
- 得到最终模型 \(\theta_{RL}\)；
- 然后一次性剪枝。
- 这是传统后训练剪枝方式。

缺点：

- 剪枝发生太晚；
- 模型已经收敛，剩余结构难以适应被删除部分。

#### 2. PTP + RL Recovery

- 在 PTP 之后，再额外进行 RL 训练恢复性能。
- 用于检验“训练后剪枝 + 恢复”是否能弥补损失。

#### 3. SFT-stage Pruning

- 模型在 SFT 阶段先 warm-up；
- 然后进行一次剪枝；
- 剩余 SFT 步骤在压缩模型上继续训练；
- 之后再进入 RL。

缺点：

- 剪枝只利用监督微调数据；
- 尚未接触闭环控制反馈。

#### 4. BPF-RL

- 先完整进行 SFT；
- 然后在 RL 阶段进行迭代剪枝。
- 设最终目标剪枝率为 \(\rho\)，RL 阶段剪枝事件数为 \(J_{RL}\)。
- 每次剪枝增加累计剪枝率：

\[
\frac{\rho}{J_{RL}}
\]

- 流程为：
  1. 从 SFT 后模型 \(\theta_{SFT}\) 开始；
  2. RL warm-up 若干步；
  3. 每隔 \(\Delta t_{RL}\) 步重新计算剪枝重要性；
  4. 移除最低重要性的参数组；
  5. 继续 RL，让剩余参数适应结构变化；
  6. 重复直到达到目标剪枝率。

#### 5. BPF-SFT/RL

- 这是论文主推的最佳策略。
- 其思想是：
  - SFT 阶段可用于移除明显冗余结构；
  - RL 阶段再根据闭环反馈进一步剪枝。
- 设最终剪枝率为 \(\rho\)。
- 先在 SFT 中剪到中间比例 \(\rho_{SFT}\)，且：

\[
\rho_{SFT} < \rho
\]

- 然后 RL 阶段继续剪剩余比例：

\[
\rho - \rho_{SFT}
\]

- 如果 RL 中有 \(J_{RL}\) 次剪枝事件，则每次增加：

\[
\frac{\rho - \rho_{SFT}}{J_{RL}}
\]

- 实验设置中，BPF-SFT/RL 将目标剪枝率平均分到两阶段：
  - SFT 阶段剪 \(\rho / 2\)；
  - RL 阶段再逐步剪 \(\rho / 2\)。

### 2.4 剪枝准则：基于 Taylor 重要性的结构化剪枝

- 论文采用 **LLM-Pruner** 框架。
- 主实验使用 **结构化剪枝**，原因是：
  - 能直接得到更小的稠密模型结构；
  - 更容易在标准推理框架和硬件上获得实际加速；
  - 相比非结构化稀疏，更适合 Jetson、llama.cpp 等部署环境。

- 对每个权重 \(w_i\)，使用一阶 Taylor 近似评估重要性：

\[
I(w_i) = w_i \frac{\partial L(D;\theta)}{\partial w_i}
\]

其中：

- \(D\)：校准数据；
- \(L(D;\theta)\)：当前模型在校准数据上的损失；
- \(\theta\)：当前模型参数。

- 对于一组结构相关参数 \(G_k\)，其重要性为组内参数重要性之和：

\[
S(G_k) = \sum_{w_i \in G_k} I(w_i)
\]

- 剪枝时移除得分最低的参数组，直到达到当前目标剪枝率。

### 3.4 Benchmark / 评估指标

主要评估指标是 RobotxR1 中的 **control adaptability**，即相对于默认 MPC 控制器的平均改进百分比。

- 评估四类任务误差：
  - \(E_c\)：中心线跟踪误差；
  - \(E_v\)：目标速度跟踪误差；
  - \(E_r\)：倒车任务误差；
  - \(E_s\)：平滑驾驶误差，以加速度接近 0 作为 proxy。

- 平均提升定义为：

\[
AvgImprove = \frac{100}{4} \sum_{E \in \{E_c,E_v,E_r,E_s\}} \frac{E_{MPC} - E_{LLM}}{E_{MPC}}
\]

其中：

- \(E_{MPC}\)：默认 MPC 误差；
- \(E_{LLM}\)：LLM 生成参数更新后的误差。

  - RL 剪枝事件数：3 次；
  - 剪枝间隔：100 RL steps。

- 但论文正文和附录中**没有明确报告训练所用 GPU 型号、GPU 数量、训练总时长或总算力成本**。
- 因此无法准确判断训练阶段的计算开销。

### 4.2 部署资源

论文较详细地报告了部署硬件与推理性能。

#### Jetson AGX Orin

- 目标机器人平台搭载：
  - **NVIDIA Jetson AGX Orin**；
  - 作为车载 LLM 推理设备。

- 部署流程：
  - 使用 BPF-SFT/RL 剪枝模型；

  - 导出为 **GGUF**；
  - 使用 **llama.cpp** runtime；
  - 为适配 llama.cpp，对 MLP 权重输出维度进行 zero-padding，使其为 256 的倍数，避免 FP16 fallback。

#### RTX A5000

- 附录中还报告了在 **RTX A5000 工作站 GPU** 上的部署吞吐。

## 5. 实验数量与充分性

### 5.1 实验覆盖范围

论文实验较系统，主要包括以下几类：

#### 1. DecisionxR1 单模块剪枝实验

- 固定 MPCxR1 不剪枝；
- 对 DecisionxR1 进行不同剪枝率实验；
- 剪枝率包括：
  - 10%；
  - 20%；
  - 30%；
  - 45%。

- 对比：
  - PTP；
  - PTP+R；
  - BPF-RL；
  - BPF-SFT/RL；
  - dense 3B；
  - dense 1.5B。

#### 2. DecisionxR1 与 MPCxR1 联合剪枝实验

- 同时剪枝两个模块；
- 探索不同 Decision / MPC 剪枝率组合；
- 例如：
  - 10/10；
  - 30/10；
  - 30/30；
  - 45/10 等。

#### 3. 嵌入式部署实验

- 在 Jetson AGX Orin 上测量：
  - 内存占用；
  - decode throughput；
  - 端到端级联延迟。
- 在 RTX A5000 上补充测量：
  - prefill throughput；
  - decode throughput。

#### 4. 附录实验

- 从 1.5B checkpoint 出发进行剪枝；
- 非结构化剪枝的 DecisionxR1 standalone accuracy；
- ROS 仿真中的定性轨迹可视化。

---

### 5.2 实验是否充分

总体而言，实验较充分，尤其是在以下方面：

- 对不同剪枝时机进行了系统比较；
- 不仅报告模型大小，也报告任务性能和部署吞吐；
- 不只做单模块实验，还做了级联闭环系统实验；
- 在真实目标边缘设备 Jetson AGX Orin 上进行部署评估；
- 注意到生成 token 数变化对端到端延迟的影响。

但也存在不足：

- 任务场景主要集中在 RobotxR1 自动驾驶赛车环境，外部泛化性有限；
- 没有在其他机器人任务、其他自动驾驶 benchmark 或真实道路数据上验证；
- 训练算力和训练时间未公开，影响可复现性评估；
- 剪枝调度策略数量有限，未探索自适应剪枝间隔或更细粒度策略；
- MPCxR1 的训练数据来自原始 pipeline，而不是剪枝后的 DecisionxR1 输出，可能带来分布偏移。

---

### 5.3 公平性与客观性

- 论文强调保持 RobotxR1 原有：
  - 控制器；
  - prompts；
  - rewards；
  - 评估协议不变。
- 所有主要剪枝实验均从 Qwen2.5-3B checkpoint 开始，比较较公平。
- dense 1.5B 和 dense 3B 也在同一部署栈中重新测量，避免直接引用旧实现带来的不公平。
- 但仍有潜在公平性问题：
  - BPF-SFT/RL 的剪枝时机和比例可能经过经验选择；
  - PTP+R 的 recovery 训练是否与 BPF 拥有完全相同的总训练预算，虽然论文称所有 schedule 使用相同 SFT/RL 步数，但 recovery 细节仍可进一步说明；
  - 不同模型生成 token 数不同，会影响延迟，单纯比较吞吐并不完全公平。

## 6. 主要结论与发现

### 6.1 BPF 的性能-压缩权衡最好

- 在 DecisionxR1 剪枝实验中，BPF-RL 和 BPF-SFT/RL 相比 PTP、PTP+R 显著更优。
- 在 45% 剪枝率下：
  - BPF-RL 相比 PTP+R 提升约 9.7%，相比 PTP 提升约 15.2%；
  - BPF-SFT/RL 相比 PTP+R 提升约 11.4%，相比 PTP 提升约 17%。

---

### 6.2 BPF-SFT/RL 优于直接选择小模型

- 使用 3B 模型并通过 BPF-SFT/RL 剪枝，比直接使用 1.5B dense 模型有更好的性能-规模权衡。
- 论文给出的关键结果：
  - BPF-SFT/RL 在 45% 剪枝时减少 0.91B 参数；
  - 相对于 dense 3B 损失 7.61 个百分点控制适应性；
  - 即每损失 1 个百分点可减少 0.12B 参数。
- 直接从 3B 换到 1.5B：
  - 减少 1.55B 参数；
  - 但损失 21.7 个百分点；
  - 即每损失 1 个百分点只减少 0.071B 参数。
- 因此，BPF-SFT/RL 的规模-性能权衡比直接选小模型好约：

\[
1.69\times
\]

---

### 6.3 MPCxR1 比 DecisionxR1 更敏感

- 联合剪枝实验发现：
  - DecisionxR1 可以承受较大剪枝率，如 30% 或 45%；
  - MPCxR1 对剪枝更敏感。
- 例如：
  - 30/10 配置能接近 3B baseline；
  - 30/30 配置性能显著下降至约 mid-40% 区间。
- 这说明：
  - MPCxR1 直接生成 MPC 参数，对控制效果影响更直接；
  - 因此不能简单对所有模块使用相同剪枝率。

---

### 6.4 轻度联合剪枝甚至可能超过原始模型

- 在某些配置下，如 DecisionxR1 / MPCxR1 均剪枝 10%，性能可超过未剪枝 3B baseline。
- 这表明：
  - 剪枝可能移除对闭环任务无益甚至有害的冗余结构；
  - 适度结构压缩可能具有正则化效果。

---

### 6.5 端到端延迟不能只看模型参数量

- 论文一个重要发现是：
  - 剪枝后模型虽然更小、单模型吞吐更高；
  - 但生成 token 数可能变长；
  - 在级联 pipeline 中，这会抵消甚至反转加速收益。

- 例如：
  - 剪枝 DecisionxR1 可能改变传给 MPCxR1 的文本；
  - 剪枝 MPCxR1 可能导致生成更啰嗦的参数更新；
  - 从而使整体 latency 增加。

- 最佳延迟配置之一是：
  - DecisionxR1 剪枝 20%；
  - MPCxR1 不剪枝；
  - 端到端延迟降低 19.5%；
  - 任务性能仅下降 0.8%。

---

## 7. 优点与亮点

### 7.1 问题定位准确

- 论文没有只关注 LLM 压缩的一般指标，而是将问题放在具身控制和自动驾驶闭环系统中。
- 它指出机器人系统中的压缩目标不只是：
  - 参数量；
  - FLOPs；
  - 显存；
  - token/s；
- 还包括：
  - 闭环控制性能；
  - 生成长度；
  - 下游模块行为；
  - 端到端 latency。

---

### 7.2 剪枝时机研究有价值

- 论文的核心贡献不是提出新的剪枝重要性指标，而是研究：
  - **剪枝应该发生在 SFT / RL pipeline 的哪个阶段**。
- 这对具身 LLM 很重要，因为最终性能由闭环反馈塑造。
- BPF 将剪枝与 RL 过程结合，是一个面向机器人控制场景的实用策略。

---

### 7.3 实验设计较完整

- 包含：
  - 单模块剪枝；
  - 双模块联合剪枝；
  - 与小 dense 模型比较；
  - 与后训练剪枝比较；
  - 与 RL recovery 比较；
  - 嵌入式部署测试；
  - 端到端 latency 测试；
  - 非结构化剪枝补充；
  - 定性轨迹展示。

---

### 7.4 部署评估真实可信

- 论文没有只在服务器 GPU 上测试，而是在目标机器人平台 Jetson AGX Orin 上部署。
- 同时使用实际推理框架 llama.cpp、GGUF 和 Q5_K_M 量化，贴近真实部署流程。

---

### 7.5 发现了重要系统性现象

- 论文强调：
  - 小模型不一定端到端更快；
  - 剪枝不一定降低级联系统 latency；
  - 生成 token 长度会成为机器人 LLM 部署中的关键变量。
- 这一点对未来 embodied LLM 系统设计具有启发性。

---

## 8. 不足与局限

### 8.1 场景覆盖有限

- 实验主要集中在 RobotxR1 和 1:10 赛车自动驾驶环境。
- 尚未验证于：
  - 其他自动驾驶 benchmark；
  - 城市场景；
  - 多智能体交通；
  - 真实道路数据；
  - 机械臂、移动机器人、导航等其他具身任务。

---

### 8.2 模块未联合训练和剪枝

- DecisionxR1 和 MPCxR1 是分别训练和剪枝的。
- MPCxR1 训练时使用的是原始 pipeline 输出，而不是剪枝后的 DecisionxR1 输出。
- 因此，剪枝后的上游输出可能导致下游分布偏移。
- 这可能解释某些配置中 MPCxR1 输出变长、pipeline latency 增加的问题。

---

### 8.3 剪枝调度策略较固定

- 本文使用固定设置：
  - SFT 剪枝点固定；
  - RL warm-up 固定；
  - RL 中 3 次剪枝；
  - 每次间隔 100 步；
  - BPF-SFT/RL 中 SFT / RL 各承担一半剪枝率。
- 尚未探索：
  - 自适应剪枝时间；
  - 自适应剪枝比例；
  - 更频繁但更小幅的剪枝；
  - 基于奖励变化或梯度稳定性的动态剪枝策略。

---

### 8.4 剪枝与量化未联合优化

- 论文采用先剪枝、再 Q5_K_M 量化的流程。
- 剪枝过程不是 quantization-aware。
- 实际部署时还需要对 MLP 维度 zero-padding 以适配 llama.cpp，说明剪枝结构与推理 runtime 之间仍存在不匹配。
- 如果联合优化剪枝、量化和硬件对齐，可能获得更好效率。

---

### 8.5 RL 目标没有约束输出长度

- 论文发现剪枝可能导致生成更长文本，增加端到端延迟。
- 但 RL reward 中没有显式惩罚 token 长度。
- 这使得模型可能在任务性能上表现不错，却在实时部署中变慢。
- 未来应考虑加入：
  - 长度惩罚；
  - 结构化输出约束；
  - 只输出变化参数；
  - latency-aware reward。

---

### 8.6 缺少训练算力报告

- 论文未说明训练使用的 GPU 型号、数量、训练时长或能耗。
- 对于评估方法的实际成本和可复现性，这是一个缺口。
- 尤其 BPF 在 RL 过程中多次重新计算剪枝分数，可能带来额外训练开销，但论文没有量化该部分成本。

---

### 8.7 对更大模型和多模型族的泛化未知

- 实验主要基于 Qwen2.5-3B / 1.5B。
- 尚不清楚该方法是否同样适用于：
  - 7B、14B 甚至更大模型；
  - LLaMA、Mistral、Gemma 等其他模型族；
  - 多模态 LLM 或 VLA 模型。

---

# 总体评价

- 本文提出的 BPF 是一个面向具身 LLM 控制器的实用剪枝策略，核心价值在于将结构压缩放入 RL 闭环适应过程中，而不是训练后一次性处理。
- 实验证明，BPF-SFT/RL 在 RobotxR1 自动驾驶流水线中能取得较好的任务性能、模型大小和推理吞吐权衡，并优于直接使用较小 dense 模型。
- 论文尤其有价值的一点是从系统层面指出：机器人 LLM 压缩必须端到端评估，因为生成 token 数和模块级联效应会显著影响最终延迟。
- 其主要不足在于实验场景较单一、训练算力未披露、剪枝策略仍较固定，且尚未将剪枝、量化、输出长度控制和多模块联合训练统一优化。
