## 摘要
视觉-语言-动作（Vision-Language-Action, VLA）模型在机器人操作中展现出强大潜力，但在边缘硬件上的实时部署仍具挑战性。在本工作中，我们发现 VLM 的视觉与上下文 token 是部署延迟的主要来源：对于以 GEMM 为主的投影算子，在模型维度固定时，计算量随输入 token 数量线性增长。基于这一观察，我们提出 RhinoVLA，这是一种面向部署的 VLA 模型，并与 Huixi R1 边缘 SoC 协同设计。RhinoVLA 采用 token 高效的 Qwen3-VL 骨干网络和连续 Action Expert，在保留预训练多模态能力的同时，降低 VLM 侧的 token 与计算负担。为支持跨机器人学习，RhinoVLA 进一步引入了统一接口，结合 View Registry、72D 物理状态-动作槽空间以及 robot-instance LoRA，使异构机器人观测与动作模式能够在共享策略下对齐。在部署侧，RhinoVLA 通过硬件感知编译、混合精度执行和并行视觉编码进行优化。实验表明，RhinoVLA 在相近参数规模下取得了与 π0.5 相当的下游性能，同时在 Huixi R1 上实现了 11.69 Hz 的端到端推理速度，达到了 10 Hz 实时闭环控制目标。项目将在 https://github.com/HuixiAI/RhinoVLA 开源。

## Abstract
Vision-Language-Action (VLA) models have shown strong potential for robotic manipulation, but real-time deployment on edge hardware remains challenging. In this work, we identify VLM visual and context tokens as a major source of deployment latency: for GEMM-dominated projection operators, computation grows linearly with the number of input tokens when model dimensions are fixed. Motivated by this observation, we propose RhinoVLA, a deployment-oriented VLA model co-designed with the Huixi R1 edge SoC. RhinoVLA adopts a token-efficient Qwen3-VL backbone and a continuous Action Expert, reducing the VLM-side token and computation burden while preserving pretrained multimodal capability. To support cross-robot learning, RhinoVLA further introduces a unified interface that combines View Registry, 72D physical state-action slot space, and robotinstance LoRA, allowing heterogeneous robot observations and action schemas to be aligned under a shared policy. On the deployment side, RhinoVLA is optimized through hardware-aware compilation, mixed-precision execution, and parallel visual encoding. Experiments show that RhinoVLA achieves downstream performance comparable to π0.5 at a similar parameter scale, while reaching 11.69 Hz end-to-end inference on Huixi R1, meeting the 10 Hz real-time closedloop control target. The project will be open-sourced at https://github.com/HuixiAI/RhinoVLA.

## 2. 方法论：核心思想、关键技术细节与算法流程

### 2.1 总体架构：RhinoVLA

RhinoVLA 采用类似 π0.5 的两模块 VLA 架构：

- **VLM Backbone**：
  - 使用 Qwen3-VL-2B 作为视觉语言骨干；
  - 负责多视角图像、语言指令、上下文信息的编码。
- **Action Expert**：
  - 负责根据 VLM 输出的视觉语言上下文生成连续动作；
  - 使用 flow matching 方式预测动作 chunk；
  - 输出统一 72D 物理动作槽空间中的动作速度场。

该设计目标是在保留强多模态理解能力的同时，降低 VLM 侧 token 和计算负担。

---

### 2.2 Token 高效的 Qwen3-VL 骨干

论文认为 VLM 视觉 token 是部署延迟的重要来源，因此选择更 token-efficient 的 Qwen3-VL：

- 在常见图像设置下：
  - **Qwen3-VL**：每张图像约产生 **64 个 merged visual tokens**；
  - **PaliGemma-224**：每张图像使用 **256 个 image tokens**。
- 因此，Qwen3-VL 在视觉 token 数上约减少 **4 倍**。
- 对多相机机器人输入而言，这种 token 减少会直接降低：
  - VLM MLP projection 的 GEMM 计算量；
  - 内存访问压力；
  - 端到端推理延迟。

---

### 2.3 Action Expert 与 flow matching 动作生成

Action Expert 与 π0.5 规模相近：

- RhinoVLA Action Expert：约 **0.40B 参数**；
- π0.5 Action Expert：约 **0.43B 参数**。

Action Expert 的输入包括：

- Qwen3-VL 最后 18 层 KV cache；
- 当前 72D 机器人状态；
- 状态 mask；
- 动作 mask；
- noisy action chunk；
- flow matching 时间步；
- robot-instance index。

其目标是在统一 72D 动作槽空间上预测流速度。

论文中的 flow matching 过程可以概括为：

- 给定干净动作 chunk：

  \[
  z \in \mathbb{R}^{H \times 72}
  \]

- 采样高斯噪声：

  \[
  a \sim \mathcal{N}(0, I)
  \]

- 采样插值时间：

  \[
  t \in [0, 1]
  \]

- 构造中间状态：

  \[
  x_t = (1 - t)a + tz
  \]

- Action Expert 预测速度：

  \[
  \hat{v}_\theta = f_\theta(x_t, t, s, m_s, m_a, c_{vlm}, r)
  \]

其中：

- \(s\)：72D 机器人状态；
- \(m_s\)：状态 mask；
- \(m_a\)：动作 mask；
- \(c_{vlm}\)：Qwen3-VL 的视觉语言 KV cache；
- \(r\)：robot-instance LoRA 路由索引。

训练目标为 masked flow-matching loss，只在有效动作槽上计算监督：

\[
L_{FM} =
\frac{
\sum_{h,d} m_a(d)w(h,d)
\|\hat{v}_\theta(h,d) - (z(h,d)-a(h,d))\|_2^2
}{
\sum_{h,d} m_a(d)w(h,d) + \epsilon
}
\]

这样可以避免不存在的机器人关节或控制维度被错误监督为 0。

### 2.4 跨机器人统一接口

RhinoVLA 面向异构机器人数据训练，提出了三个关键机制。

#### 2.4.1 View Registry：统一相机视角接口

不同机器人数据集的相机数量、顺序、命名和模态不同。RhinoVLA 使用 View Registry 显式标记每张图像的角色和模态，例如：

- `[head | rgb]`
- `[left_wrist | rgb]`
- `[right_wrist | rgb]`
- `[head | depth]`

训练 prompt 模板类似：

```text
Task: {task}
Views:
[{role_1} | {modality_1}] + image_1
[{role_2} | {modality_2}] + image_2
...
```

作用是：

- 消除“第几张图像”与“哪个视角”之间的歧义；
- 支持不同机器人拥有不同数量和类型的相机；
- 让 VLM 能显式知道图像的视角语义。

---

#### 2.4.2 统一 72D 物理状态-动作槽空间

不同机器人动作维度和物理意义不同，不能直接按数组下标拼接训练。RhinoVLA 定义了一个统一的 72D 物理槽空间：

- D0–D6：Arm 0 关节；
- D7–D13：Arm 1 关节；
- D14–D15：双夹爪；
- D16–D31：Hand 0 灵巧手 DoF；
- D32–D47：Hand 1 灵巧手 DoF；
- D48–D50：头部 / 颈部 RPY；
- D51–D52：躯干 pitch / lift；
- D53–D54：折叠腿机构；
- D55–D57：腰部 RPY；
- D58–D60：底盘速度；
- D61–D71：保留槽位。

每个机器人只激活自己存在的物理槽，其余槽通过 binary mask 屏蔽。

该设计使得：

- 同一维度在所有机器人中具有固定物理意义；
- 缺失维度不会被错误监督；
- 单臂、双臂、移动底盘、夹爪、灵巧手等机器人可以共享一个动作接口。

---

#### 2.4.3 Robot-instance LoRA：机器人实例残差适配

即便使用统一视角和统一动作槽，不同机器人仍存在：

- 标定差异；
- 关节限位差异；
- 夹爪机械差异；
- 控制器差异；
- 相机安装位置差异；
- 动作尺度差异。

RhinoVLA 在 Action Expert 的 FFN 中插入 robot-instance LoRA：

- 共享 Action Expert 学习跨机器人通用策略；
- 每个机器人实例通过 instance_id 路由到自己的 LoRA；
- LoRA 学习该机器人特有的残差行为；
- 注意力模块和最终 72D 输出头保持共享。

这样有三个优点：

- 不破坏统一 72D 动作接口；
- 部署时可将 LoRA 合并进主模型；
- 新增机器人只需增加小型 LoRA 和归一化统计量。

### 2.5 部署优化：Huixi R1 上的系统协同设计

RhinoVLA 在 Huixi R1 边缘 SoC 上进行硬件优化，主要包括：

#### 编译优化

- 针对 R1 的片上 SPM 设计 FlashAttention-style tiling；
- 将 Q/K/V tile、softmax 统计量和部分输出尽量保留在片上；
- 对 Transformer block 做算子融合：
  - normalization；
  - linear projection；
  - bias；
  - activation；
  - residual；
- 通过更细粒度的 runtime task scheduling 提高芯片利用率。

#### 混合精度部署

采用 **INT8 权重 + FP16 激活**：

- 不采用 W8A8，因为会明显损害任务成功率和动作预测准确性；
- 使用 W8A16 减少权重存储和带宽；
- 针对 R1 定制 W8A16 GEMM kernel；
- 将权重加载、反量化和矩阵乘法融合到一个执行 pipeline 中。

论文报告：

- π0.5 VLM 中 latency-dominant 的 `up_proj`：
  - W16A16：191 μs；
  - 自定义 W8A16：113 μs；
  - 加速约 **1.69×**；
  - compute utilization 达 **50.6%**。

#### 并行视觉编码

RhinoVLA 使用三路图像输入：

- 头部相机；
- 左腕相机；
- 右腕相机。

将原先顺序处理三张图改为 batch 并行处理：

- 三视角视觉编码延迟从 **34.52 ms** 降至 **24.31 ms**。

### 3.6 推理效率实验

## 6. 主要结论与发现

- VLA 部署瓶颈不仅来自参数量，更来自 VLM 的视觉和上下文 token 数。
- Transformer MLP projection 是 VLM 延迟的主要来源，其计算量随 token 数线性增长。
- 使用 token-efficient 的 Qwen3-VL 可以显著降低 VLM 侧计算负担。
- 通过 View Registry、72D 物理槽空间和 robot-instance LoRA，可以将异构机器人数据对齐到统一策略接口。
- Robot-instance LoRA 能捕捉不同机器人之间的 residual embodiment variation，并改善动作预测。
- 在 LIBERO benchmark 上，RhinoVLA 达到 **90.0% 平均成功率**，低于 π0.5 的 **96.9%**，但超过多数 VLA 和 direct policy baseline。
- 在真实机器人上，RhinoVLA 在多个平台和任务中表现出一定迁移能力。
- 在 Huixi R1 上，RhinoVLA 通过模型与系统协同优化达到 **11.69 Hz**，满足 10 Hz 实时闭环控制目标。

## 7. 优点与亮点

- **问题定位清晰**：论文没有泛泛讨论模型压缩，而是通过 roofline 和算子级分析指出 token 数是核心延迟来源。
- **算法-硬件协同设计完整**：
  - 模型侧减少视觉 token；
  - 接口侧统一跨机器人数据；
  - 部署侧做编译、混合精度和并行编码优化。
- **Qwen3-VL 选型合理**：
  - 保留较强多模态能力；
  - 相比 PaliGemma 显著减少视觉 token。
- **统一 72D 动作槽空间具有工程实用性**：
  - 避免不同机器人动作向量语义混乱；
  - 适合多机器人数据混合训练；
  - 通过 mask 避免无效维度监督。
- **View Registry 简洁有效**：
  - 用显式视角标签解决相机顺序和命名不一致问题。
- **Robot-instance LoRA 设计较优雅**：
  - 不引入机器人专属输出头；
  - 保持统一部署图；
  - 支持低成本新增机器人。
- **真实部署结果有说服力**：
  - 报告了端到端 11.69 Hz；
  - 给出详细延迟分解；
  - 面向真实边缘芯片而非桌面 GPU。

---

## 8. 不足与局限

- **训练资源不透明**：
  - 未明确训练 GPU、数量、时长和计算成本；
  - 影响可复现性和资源评估。
- **真实机器人实验规模有限**：
  - 虽覆盖多个机器人，但每个任务的 trial 数和统计置信区间不充分；
  - 部分任务没有完整 baseline 对比。
- **与 π0.5 的公平性存在不确定性**：
  - π0.5 和 RhinoVLA 的训练数据、后训练流程、部署硬件可能不同；
  - 有些真实机器人任务 π0.5 未评估。
- **部署结果依赖 Huixi R1 生态**：
  - R1 是特定自研芯片；
  - 编译器、kernel 和硬件特性对结果影响大；
  - 在其他边缘设备上的泛化部署效果尚不明确。
- **LIBERO 上仍落后 π0.5**：
  - RhinoVLA 平均 90.0%，π0.5 为 96.9%；
  - 说明在纯任务性能上仍有差距。
- **72D 槽空间仍是人工设计接口**：
  - 新机器人若无法映射到现有槽位，需要扩展 schema；
  - 对复杂腿足机器人、全身控制、柔性执行器支持有限。
- **Instance LoRA 提升幅度较小**：
  - 在 action prediction 上提升一致但数值不大；
  - 还需要更大规模实验验证其长期泛化价值。
- **开源尚未完成**：
  - 论文称项目将开源，但实际可复现性取决于后续代码、权重、数据转换器和 R1 部署工具链公开程度。
