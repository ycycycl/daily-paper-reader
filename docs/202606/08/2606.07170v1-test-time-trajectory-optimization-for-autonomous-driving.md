## 摘要
用于自动驾驶的端到端规划器通常会生成一组候选轨迹，对每条轨迹进行评分，并返回得分最高的候选轨迹。然而，评分器仅在生成候选方案后才被应用，无法影响轨迹集合本身：无论评分器质量如何，较弱的候选集合都会限制规划性能。相反，我们将评分器视为一个学习得到的轨迹级奖励函数，并搜索使其最大化的轨迹。我们的方法 TOAD 在测试时运行交叉熵方法，并以规划器提出的候选轨迹进行热启动。该方法无需重新训练，可即插即用于现有规划器。在六种基础规划器上，TOAD 在 NAVSIM-v1（94.7 PDMS）、NAVSIM-v2（56.3 EPDMS）以及闭环 HUGSIM 基准上均提升了结果。代码将通过项目页面公开发布：https://valeoai.github.io/TOAD/。

## Abstract
End-to-end planners for autonomous driving typically generate a set of candidate trajectories, score each one, and return the highest-scoring candidate. However, the scorer is applied only after the proposals are generated and cannot influence the set of trajectories: a weak set of candidates limits planning performance regardless of the scorer's quality. We instead treat the scorer as a learned trajectory-level reward function and search for trajectories that maximize it. Our method, TOAD, runs the Cross-Entropy Method at test time, warm-started from the planner's proposals. It requires no retraining and is plug-and-play for existing planners. Across six base planners, TOAD improves results on NAVSIM-v1 (94.7 PDMS), NAVSIM-v2 (56.3 EPDMS), and the closed-loop HUGSIM benchmark. The code will be made publicly available via the project page: https://valeoai.github.io/TOAD/.

## 2. 方法论：核心思想、关键技术细节与算法流程

## 2.1 核心思想

- 论文提出 TOAD：**Test-Time Trajectory Optimization for Autonomous Driving**。
- TOAD 的核心做法是：
  - 冻结已有端到端规划器；
  - 冻结一个可泛化的轨迹 scorer；
  - 在测试时将 scorer 作为奖励函数；
  - 使用 Cross-Entropy Method，简称 CEM，在轨迹空间附近搜索更优轨迹。
- TOAD 不修改模型参数，不需要额外训练，因此是一个**测试时优化插件**。

---

## 2.2 轨迹 scorer 作为奖励函数

- 论文将未来轨迹表示为：

  \[
  \tau = (p_1, \ldots, p_{n_p})
  \]

  其中每个 pose 为：

  \[
  p_t = (x_t, y_t, \theta_t)
  \]

- scorer 接收观测信息 \(o\)、导航目标 \(g\) 和完整轨迹 \(\tau\)，输出一个标量分数：

  \[
  S(\tau; o, g) \in \mathbb{R}
  \]

- 传统方法只用它对固定候选轨迹排序；TOAD 则将其视为奖励函数，搜索：

  \[
  \tau^\star = \arg\max_\tau S(\tau; o, g)
  \]

- 不过直接优化轨迹可能导致轨迹不平滑或不可行，因此 TOAD 不直接在 pose 空间中采样，而是在控制空间中优化。

---

## 2.3 在控制空间中优化：自行车模型约束

- TOAD 使用可逆的 kinematic bicycle model，将轨迹和控制序列相互映射。
- 控制序列定义为：

  \[
  u = (u_t)_{t=1}^{n_p}, \quad u_t = (a_t, \omega_t)
  \]

  其中：
  - \(a_t\)：纵向加速度；
  - \(\omega_t\)：yaw rate，偏航角速度。

- 从控制序列出发，可通过 bicycle model rollout 得到轨迹：

  \[
  \tau = BM(u, v_0)
  \]

- 也可从轨迹反推出控制序列：

  \[
  u = BM^{-1}(\tau, v_0)
  \]

- 这种设计有两个优点：
  - 采样得到的轨迹更平滑、更符合车辆动力学约束；
  - 舒适性指标可直接在加速度、jerk、yaw rate 等控制量上计算。

---

## 2.4 基于基础规划器的 trust region

- TOAD 不从任意位置搜索，而是以基础规划器输出的轨迹为中心进行局部优化。
- 基础规划器输出：

  \[
  BASE\ PLANNER(o, g) = (\mathcal{T}, \tau_{base})
  \]

  其中：
  - \(\mathcal{T}\)：候选轨迹集合；
  - \(\tau_{base}\)：基础规划器选择的轨迹。

- TOAD 将 \(\tau_{base}\) 转换为控制序列 \(u_{base}\)，作为搜索锚点。
- 这样做的原因是：
  - scorer 在训练时通常只见过某些分布内的轨迹；
  - 如果测试时搜索过远，容易进入 scorer 不可靠区域；
  - 可能出现 reward hacking，即 scorer 给出高分但真实驾驶质量差。

---

## 2.5 CEM 优化目标

- TOAD 的 CEM 目标函数为：

  \[
  J(u) = S(BM(u, v_0); o, g) - \lambda_a C_{anchor}(u) - \lambda_c C_{comf}(u)
  \]

- 三个部分分别是：
  - **scorer reward**：鼓励轨迹获得更高 learned score；
  - **anchor regularization**：
  
    \[
    C_{anchor}(u) = \|u - u_{base}\|^2
    \]

    用于限制搜索不要偏离基础规划器太远；
  - **comfort regularization**：惩罚违反舒适性约束的行为，例如纵向/横向加速度、jerk、yaw rate 和 yaw acceleration 等。

---

## 2.6 CEM 算法流程

TOAD 的测试时优化流程如下：

1. 基础规划器输入观测 \(o\) 和目标 \(g\)，输出候选轨迹和基础轨迹 \(\tau_{base}\)。
2. 使用逆自行车模型将轨迹转换为控制序列。
3. 以 \(u_{base}\) 作为 CEM 初始均值：

   \[
   \mu_0 = u_{base}
   \]

4. 初始标准差来自基础规划器候选轨迹的分布：

   \[
   \sigma_0 = \max(\beta \cdot std(U), \epsilon)
   \]

5. 每轮 CEM：
   - 从高斯分布中采样 \(M\) 个控制序列；
   - 用 bicycle model rollout 成轨迹；
   - 使用 scorer + regularizer 计算目标函数 \(J(u)\)；
   - 选择得分最高的 \(E\) 个 elite 样本；
   - 用 elite 样本重新拟合高斯分布的均值和方差。
6. 迭代 \(K\) 次后，使用最终均值 \(\mu_K\) rollout 得到优化轨迹。
7. 最终在优化轨迹和原始基础轨迹之间选择得分更高者。

- 作者特别指出，最终选择时不再加入 anchor penalty，因为此时优化轨迹不应因偏离 anchor 而被额外惩罚。
- 同时与原始 \(\tau_{base}\) 比较，可保证返回轨迹在优化目标下不劣于基础轨迹。

---

## 2.7 scorer 的选择

- 一个关键发现是：**并非所有训练好的 scorer 都适合作为测试时优化的 reward**。
- 固定轨迹词表方法中的 scorer 往往只在自己的候选轨迹分布上准确，在分布外轨迹上不可靠。
- TOAD 最终主要使用 DrivoR 的 disentangled scorer，因为它被训练为评估自由解码轨迹，而不是只对固定轨迹词表排序，因此具有更好的 off-proposal 泛化能力。

## 6. 论文的主要结论与发现

- **结论 1：候选轨迹集合是当前 score-and-select 规划器的重要瓶颈。**
  - 仅仅训练更强 scorer 不够；
  - 如果候选集中没有好轨迹，最终输出仍会受限。

- **结论 2：scorer 可以作为测试时 reward 来优化轨迹。**
  - TOAD 将冻结 scorer 用作奖励函数；
  - 通过 CEM 在控制空间中搜索更优轨迹；
  - 不需要重新训练即可提升已有 planner。

- **结论 3：TOAD 在多个 planner 上均带来提升。**
  - NAVSIM-v1 上 DrivoR + TOAD 达到 94.7 PDMS；
  - NAVSIM-v2 上 DrivoR + TOAD 达到 56.3 EPDMS；
  - NAVSIM-v2 上该结果接近使用特权信息的 PDM-Closed 56.6；
  - TOAD 在六个基础 planner 上均提升 NAVSIM-v1 和 NAVSIM-v2 表现。

- **结论 4：弱 planner 的提升更大。**
  - 在 NAVSIM-v2 上：
    - iPad 从 34.7 提升到 49.8，提升 43.6%；
    - Hydra-MDP 从 40.9 提升到 49.7，提升 21.6%；
    - DrivoR 从 54.6 提升到 56.3，提升 3.1%。
  - 说明 TOAD 能缩小弱 planner 和强 scorer 可发现轨迹之间的差距。

- **结论 5：不是所有 scorer 都适合优化。**
  - 固定轨迹词表 scorer 在重排序时可能表现尚可，但作为 reward 搜索时可能崩溃；
  - 这说明常规 ranking evaluation 无法揭示 scorer 的 off-proposal 泛化能力；
  - DrivoR 的 disentangled scorer 更适合作为 TOAD 的 reward。

- **结论 6：TOAD 在闭环仿真中也能提升安全性和舒适性。**
  - 在 HUGSIM 上，DrivoR + TOAD 提升多个数据集的 NC、DAC、TTC 和 comfort；
  - 但有时 route completion 会下降，说明 TOAD 倾向更保守驾驶。

## 7. 优点与亮点

- **即插即用**
  - 不需要重新训练；
  - 可接入多种已有端到端规划器。

- **思想简洁有效**
  - 将 scorer 从“排序器”重新解释为“奖励函数”；
  - 利用经典 CEM 方法进行测试时优化。

- **充分利用已有模型能力**
  - 不修改 planner；
  - 不引入额外世界模型或占用图预测模块；
  - 只利用已有 scorer 的轨迹评价能力。

- **控制空间优化设计合理**
  - 通过 bicycle model 保证轨迹平滑和动力学可行；
  - 舒适性正则可以直接基于控制量计算。

- **trust region 机制降低 reward hacking 风险**
  - 搜索围绕基础 planner 的输出展开；
  - 避免过度偏离 scorer 的可靠区域。

- **实验覆盖较广**
  - 六个 planner；
  - NAVSIM-v1、NAVSIM-v2 和 HUGSIM；
  - 主实验、机制分析、消融实验均比较完整。

- **揭示了 scorer 评估中的隐藏问题**
  - 传统 ranking benchmark 可能无法反映 scorer 在分布外轨迹上的质量；
  - 论文指出未来 scorer 应具备可优化性和泛化性，而不仅是排序准确。

- **推理开销低**
  - 在共享 backbone 的情况下，TOAD 搜索本身只增加约 1.9 ms；
  - 对实际系统具有一定吸引力。

## 8. 不足与局限

- **依赖高质量且可泛化的 scorer**
  - TOAD 的成功高度依赖 DrivoR scorer；
  - 使用 GTRS 或 SparseDriveV2 scorer 作为 reward 时，性能反而下降；
  - 说明该方法并非对任意 scorer 都可靠。

- **存在 reward hacking 风险**
  - CEM 会主动寻找 scorer 高分区域；
  - 如果 scorer 在分布外不准确，优化可能产生真实质量差但 scorer 高分的轨迹。
  - anchor regularization 和 trust region 能缓解，但不能完全消除该问题。

- **搜索范围受限于基础 planner**
  - TOAD 以基础 planner 输出为 warm start 和 trust region；
  - 如果基础 planner 初始轨迹非常差，搜索空间可能仍不足以找到全局更优解。
  - 作者也承认，理想情况下希望从 scratch 搜索，但当前 scorer 尚不足以支持。

- **闭环实验覆盖不够全面**
  - HUGSIM 实验主要是在 DrivoR 上进行；
  - 没有展示六个基础 planner 在闭环仿真中全部接入 TOAD 的结果；
  - 因此 closed-loop 泛化结论仍有限。

- **真实道路部署尚未验证**
  - 实验基于 benchmark 和仿真；
  - 未在真实车辆或大规模在线交互环境中验证安全性、稳定性和实时性。

- **外接 scorer 会增加感知计算成本**
  - 对非 DrivoR planner，需要额外运行 DrivoR scorer backbone，约增加 100 ms；
  - 在严格实时系统中可能是负担。
  - 只有当 planner 与 scorer 共享 backbone 时，TOAD 才接近“免费”。

- **超参数虽有消融，但仍可能场景相关**
  - CEM 迭代次数、候选数量、elite 数量、comfort 权重、anchor 权重等可能影响结果；
  - 论文虽显示一定鲁棒性，但实际部署中仍可能需要调参。

- **更保守驾驶可能牺牲进度**
  - 在 NAVSIM-v2 和 HUGSIM 中，TOAD 常提升安全指标，但有时降低 ego progress 或 route completion；
  - 这表明其优化目标可能偏向安全和舒适，而对效率的平衡仍需进一步研究。
