## 摘要
视觉-语言-动作（Vision-Language-Action, VLA）策略通常以 Python/PyTorch 软件栈形式发布，并假定使用工作站级 GPU，这与机器人实际运行所依赖的硬件并不匹配。我们提出 vla.cpp，这是一个构建于 llama.cpp 之上的可移植 C++ 推理运行时。据我们所知，它是首个原生支持流匹配与扩散式 VLA 推理模式的 ggml 类引擎；在该模式中，缓存的视觉-语言前缀由一个交叉注意力动作专家消费，并通过若干求解器步骤进行积分。单一运行时通过统一的请求/响应协议支持横跨五类骨干网络和四类动作头家族的七种架构，并将每个模型打包为自包含的 bundle。在 LIBERO-Object 上，该引擎与最先进检查点的表现差距在 200 个 episode 中不超过 1 个 episode，并能以 1.3 GiB 内存运行 BitVLA 且达到 100% 成功率。同一 bundle 无需修改即可跨三档硬件运行，从消费级 GPU 到 8 GB 嵌入式模块皆可部署。跨硬件 roofline 分析表明，batch-1 VLA 推理受计算限制，因此提升利用率而非带宽是部署中的关键杠杆；基于该分析导出的 IMMA ladder GEMM 将 BitVLA 的单步延迟降低了 4.5 倍。随后，我们在 ALOHA 机械臂上设计了一项机载压力测试，用于隔离延迟约束：在其训练所用硬件上，学习得到的 VLA 必须在该约束下针对移动目标进行重新规划。代码、演示视频以及可复现实验基准框架可在 https://fai-modelopt-tech.github.io/vla-cpp.github.io/ 获取。

## Abstract
Vision-Language-Action (VLA) policies are typically shipped as Python/PyTorch stacks that assume a workstation-class GPU, a mismatch for the hardware on which robots actually run. We present vla.cpp, a portable C++ inference runtime built on llama.cpp. To our knowledge, it is the first ggml-class engine to natively serve the flow-matching and diffusion VLA inference pattern, in which a cached vision-language prefix is consumed by a cross-attending action expert integrated over several solver steps. A single runtime serves seven architectures spanning five backbone and four action-head families behind one request/response protocol, with each model packaged as a self-contained bundle. On LIBERO-Object, the engine matches a state-of-the-art checkpoint to within one episode out of 200, and runs BitVLA at 100% success in 1.3 GiB of memory. The same bundle runs unchanged across three hardware tiers, from a consumer GPU down to an 8 GB embedded module. A cross-hardware roofline analysis shows that batch-1 VLA inference is compute-bound, so utilization rather than bandwidth is the deployment lever; an IMMA ladder GEMM derived from this analysis cuts BitVLA per-step latency by 4.5x. We then frame an on-robot stress test on an ALOHA arm that isolates the latency constraint under which a learned VLA must replan against a moving target on the hardware it was trained for. Code, demo videos, and the reproducible benchmark scaffold are available at https://fai-modelopt-tech.github.io/vla-cpp.github.io/.

当前主流 VLA 模型通常以 **Python / PyTorch 软件栈**发布，并默认运行在工作站级 GPU 上。然而真实机器人常用的机载计算平台通常是资源受限的嵌入式设备，例如 **NVIDIA Jetson Orin 系列**，其内存、算力、软件环境和散热功耗都远弱于桌面 GPU。

这种错配带来几个实际问题：

- **内存不足**：大型 VLA 模型动辄数 GB 权重，Jetson Orin Nano 这类 8 GB 统一内存设备还需同时运行系统、相机、控制栈和仿真/机器人通信。

- **部署栈复杂**：Python/PyTorch 在边缘设备上依赖重、启动慢、运行时开销大，难以形成轻量、可移植、可复现的机器人部署方案。
- **现有 LLM 推理引擎不适配 VLA**：如 llama.cpp、vLLM、MLC-LLM 等主要面向 token-by-token 文本生成，不支持 VLA 中常见的流匹配 / 扩散式动作头、多步求解、跨注意力缓存等推理模式。

## 2. 方法论：核心思想、关键技术细节与算法流程

## 2.1 核心思想

vla.cpp 的核心思想是：

> 将 VLA 推理抽象为统一的两阶段流程，并在 llama.cpp / ggml 运行时中原生支持这一流程。

这两个阶段是：

1. **Prefix 阶段**  
   使用视觉编码器和语言模型，将图像、语言指令、机器人状态编码为多模态前缀序列。

2. **Action 阶段**  
   动作头通过 cross-attention 消费缓存的前缀表示，并通过流匹配或扩散求解器多步生成连续动作 chunk。

与传统 LLM 解码不同，VLA 推理不是逐 token 生成，而是生成连续动作向量，并且同一个视觉语言前缀会在多个 solver step 中被反复访问。

---

## 2.2 系统架构

vla.cpp 采用 **客户端-服务器式架构**：

### C++ Server

- 加载 GGUF / bundle 模型文件；
- 接收来自客户端的 observation；
- 执行 VLA 推理；
- 对动作进行反归一化；
- 返回 action chunk。

### Python Client

- 负责机器人或仿真环境交互；
- 管理闭环控制逻辑；
- 处理 action chunk 的执行、重规划、超时保护等。

### 通信协议

- 使用 **ZeroMQ** 进行通信；
- 使用 **Protocol Buffers** 定义请求和响应格式；
- 请求包含图像、语言指令、本体状态等；
- 响应包含反归一化后的动作 chunk。

这种设计使得运行时本身尽量保持无状态，而控制历史、动作融合、安全保持等机器人逻辑留在客户端实现。

## 2.4 关键技术细节

### 2.4.1 暴露语言模型最终隐藏状态

普通 LLM 推理运行时通常只输出 logits 或 pooled embedding。  
但 VLA 动作头需要 cross-attend 到完整的多模态前缀序列，因此 vla.cpp 需要暴露语言模型最终层的完整 hidden states：

```text
[tokens × hidden_dim]
```

这些 hidden states 会作为动作头 cross-attention 的 key / value 来源。

---

### 2.4.2 使用双向 attention mask 编码前缀

传统文本解码器使用因果 mask，只允许 token 看到之前的 token。  
但 VLA 的视觉-语言-状态前缀本质上是一个完整观测，应该允许图像、语言和状态之间互相访问。

因此 vla.cpp 在 prefix 阶段使用 **bidirectional attention mask**。

如果错误地使用 causal mask，论文指出会悄悄破坏缓存的 key/value，从而导致动作预测错误。

---

### 2.4.3 显式 cross-attention KV cache 生命周期

VLA 的动作头通常需要多步积分，例如流匹配或扩散模型需要执行多个 solver step。

在每个 step 中，动作头都会 cross-attend 到同一个视觉语言前缀。因此 vla.cpp 为此实现了独立的 cross-attention KV cache：

1. 每个 observation 到来后，先计算视觉语言前缀；
2. 从前缀中生成 cross-attention 的 key/value；
3. 在多个 solver step 中复用该 cache；
4. action chunk 生成完成后释放 cache。

这不同于 LLM 自回归解码中的 self-attention KV cache。

---

### 2.4.4 支持流匹配和扩散动作头

vla.cpp 支持七种 VLA 架构，覆盖五类 backbone 和四类 action head：

- SmolVLA；
- π0；
- BitVLA；
- Evo-1；
- GR00T-N1.5；
- GR00T-N1.6；
- GR00T-N1.7。

动作头类型包括：

- flow-matching cross-attention expert；
- joint-attention flow expert；
- cross-attention DiT；
- diffusion-transformer action head；
- BitVLA 的单步 MLP regression head。

## 2.5 算法流程

论文给出的标准 VLA 推理流程可以概括为：

1. 输入 observation：
   - 图像 \(I\)；
   - 语言指令 \(\ell\)；
   - 本体状态 \(s\)。

2. 使用 VLM 编码多模态前缀：

   ```text
   h = VLM_PREFIX(I, ℓ, s)
   ```

3. 从前缀 \(h\) 计算并缓存 cross-attention keys / values。

4. 初始化动作变量：
   - 流匹配中通常从高斯噪声初始化；
   - 扩散模型也可能使用噪声或均匀采样。

5. 对 solver step \(t = 1, ..., T\)：

   ```text
   a_t = ACTION_HEAD(a_{t-1}, h, t)
   ```

   动作头在每一步 cross-attend 到缓存前缀。

6. 得到最终动作 chunk \(a_T\)。

7. 使用模型自带统计量反归一化动作。

8. 返回机器人可执行的 action chunk。

## 2.6 低比特与 kernel 优化

论文特别针对 **BitVLA** 做了低比特推理优化。

BitVLA 使用 ternary / 1.58-bit 权重，论文将其权重打包为 2-bit layout，使模型占用大幅下降。

但作者指出：

> 低比特权重本身只减少内存占用，不自动带来速度提升。

原因是如果仍然使用低效 kernel，计算单元利用率不足，延迟不会明显下降。

因此论文实现了基于 **IMMA tensor core** 的 W2A8 ternary GEMM，将原本使用 CUDA core dp4a 的路径迁移到整数 tensor core 上，从而显著提升吞吐。

- 多数不支持统一 portable bundle；
- 不支持动态 cross-attention KV cache；

## 6. 主要结论与发现

## 6.1 vla.cpp 能保持行为一致性

在 LIBERO-Object 上，vla.cpp 对多个 VLA 架构都能复现 PyTorch checkpoint 的行为。

代表性结果：

- BitVLA 达到 **200/200，100% 成功率**；
- 与参考 checkpoint 的差距在 200 个 episode 中约为 1 个 episode 量级；
- 多个模型的成功率与参考实现在统计上接近。

这说明 vla.cpp 不只是“跑得快”，而是能较忠实地执行原模型。

---

## 6.2 边缘部署的主要瓶颈是内存容量

跨硬件测试显示，在 8 GB Jetson Orin Nano 上：

- SmolVLA、BitVLA、Evo-1 等较小模型可加载；
- GR00T-N1.6 / N1.7 无法加载；
- π0 和 GR00T-N1.5 只有在仿真器卸载到其他机器时才可运行。

论文结论是：

> 对小型机器人边缘模块而言，模型是否装得下往往比推理是否足够快更关键。

---

## 6.3 Batch-1 VLA 推理主要受计算单元利用率限制

Roofline 分析显示：

- 视觉语言 prefix 阶段处理数百 token，权重复用高，主要是 compute-bound；
- action expert token 数少，更多是 memory-bound；
- 但在实际部署配置下，prefix 阶段通常主导整体 forward 时间。

因此，优化方向不是单纯追求带宽，而是提高计算单元利用率。

---

## 6.4 低比特压缩主要解决容量，不自动解决速度

BitVLA 的 ternary 权重打包显著降低模型体积：

- on-disk 模型从约 5.6 GiB 降到 1.34 GiB；
- load-time high-water mark 从 11.1 GiB 降到 1.15 GiB。

但如果 kernel 不变，延迟并不会改善。

只有将 W2A8 GEMM 从 dp4a 路径迁移到 IMMA tensor core 后，才获得显著加速：

- RTX 3060：172.8 ms → 37.85 ms；
- AGX Orin：406.6 ms → 101.11 ms；
- 约 4× 到 4.6× 加速；
- 输出 bit-identical，成功率不变。

---

## 6.5 Reduced precision 可能导致机器人动作灾难性偏移

论文发现一个重要数值问题：

- 某些视觉 encoder 中 position embedding 的索引计算在 FP32 和 FP16 下会出现不同舍入；
- 如果模型主体是 half precision，但 position index 用 single precision 计算，就可能选错 embedding row；
- 这会导致视觉语言前缀偏移；
- 流匹配 solver 会在多步积分中放大该偏差；
- 最终 action chunk 在机器人动作空间中出现大幅位移。

在 SmolVLA 上，错误配置导致：

- position embedding relative error 从 0.003 增至 0.32；
- action chunk 最大差异约 1.97；
- 任务失败。

作者据此强调：

> 部署系统必须做 numerical parity gating，并且要 fail-loud，而不是静默产生错误动作。

---

## 6.6 真实机器人中，降低推理延迟能显著提高闭环成功率

在 ALOHA 实验中：

- vla.cpp 每个 chunk 推理约 470 ms；
- PyTorch baseline 约 620 ms；
- vla.cpp 每次重规划使用的 observation 更新鲜约 150 ms。

结果：

- vla.cpp：35/40，87.5% 成功率；
- PyTorch：16/40，40.0% 成功率。

这说明在真实机器人长时程任务中，推理延迟不仅是性能指标，还直接影响任务成功率。

---

## 7. 优点与亮点

## 7.1 系统贡献明确

论文提出的是一个实际可用的部署系统，而非单一模型优化。它提供：

- 统一 C++ runtime；
- 统一请求/响应协议；
- 统一 bundle 格式；
- 多模型支持；
- 多硬件支持。

这对 VLA 工程化部署有直接价值。

---

## 7.2 首次系统支持 flow / diffusion VLA 推理模式

相比传统 LLM runtime，vla.cpp 原生支持：

- cached multimodal prefix；
- cross-attention KV cache；
- 多步 denoising / flow matching solver；
- continuous action chunk 输出。

这正是现代 VLA 与普通 LLM 推理的关键差异。

---

## 7.3 支持架构范围广

一个 runtime 支持七种架构，覆盖：

- 小模型；
- 大模型；
- ternary 模型；
- flow matching；
- diffusion transformer；
- 单步 regression head。

这比仅适配单一 VLA 的部署工作更具普适性。

---

## 7.4 实验维度较完整

论文不仅报告速度，还报告：

- 行为保真度；
- 成功率；
- 内存占用；
- 跨硬件可运行性；
- roofline 分析；
- reduced precision 消融；
- 真实机器人闭环效果。

这种组合较好地覆盖了真实部署所关心的问题。

---

## 7.5 对“低比特等于更快”的澄清有价值

论文清楚地区分了：

- **模型压缩 / footprint**；
- **推理延迟 / utilization**。

它指出低比特权重首先解决内存容量，是否加速取决于 kernel 是否能充分利用硬件。

这一点对边缘模型部署很有启发。

---

## 7.6 真实机器人实验增强说服力

ALOHA 实验展示了：

- vla.cpp 不只是仿真中更快；
- 延迟降低能实际改善闭环任务成功率；
- action chunk 的 observation staleness 是真实部署中的关键问题。

---

## 8. 不足与局限

## 8.1 训练侧不涉及

本文主要关注推理运行时，不讨论：

- 如何训练更小的 VLA；
- 如何联合优化模型结构和部署；
- 训练成本、数据成本和泛化能力。

因此它解决的是“已有模型如何部署”的问题，而不是“如何训练更优模型”的问题。

---

## 8.2 低比特支持范围有限

论文中的低比特原生优化主要集中在 BitVLA 的 ternary 配置。

目前尚未展示：

- 对更多 4-bit / 8-bit VLA 的系统支持；
- 对不同 action head 的广泛量化；
- 对视觉 tower 和 diffusion head 的统一低比特方案。

---

## 8.3 大模型在 8 GB 设备上仍然受限

虽然 vla.cpp 显著改善部署能力，但并不能让所有模型都装进低端设备。

例如：

- GR00T-N1.6；
- GR00T-N1.7；

仍然无法在 8 GB Orin Nano 上加载。

论文也承认缺少动态 weight staging 机制。

---

## 8.4 TensorRT 等高度优化厂商栈没有端到端实测对比

论文通过能力矩阵说明 TensorRT-LLM、TensorRT Edge-LLM、ONNX Runtime 等不完全适配该任务。

但从性能角度看，缺少与专门手工优化 TensorRT engine 的端到端对比。

这使得“最快”这一点不能被完全证明；论文更强的主张是“统一、便携、跨架构”。

---

## 8.5 LIBERO 可能存在 benchmark 饱和问题

作者承认 LIBERO 任务可能高估泛化能力，部分模型在一些套件上接近满分。

因此 LIBERO 上的一两个 episode 差距虽然能说明行为保真度，但不能充分证明真实世界泛化能力。

---

## 8.6 真实机器人实验规模仍有限

ALOHA 实验虽然有说服力，但覆盖范围有限：

- 只有一个机器人平台；
- 只有两个任务；
- 只有 GR00T-N1.6 一个模型；
- 总 trial 数为 80 次。

还需要更多机器人、更多任务、更多动态环境来验证普适性。

---

## 8.7 Reduced precision 问题提示部署复杂性

论文发现的精度脆弱性虽然是重要贡献，但也说明：

- 多模态模型部署非常容易出现静默数值错误；
- 不同模型的视觉 tower、position encoding、indexing 逻辑差异很大；
- runtime 需要大量 parity test 才能保证安全。

这增加了维护和扩展新架构的工程成本。

---

## 总体评价

vla.cpp 是一篇偏系统与部署的 VLA 论文，核心价值在于将现代流匹配 / 扩散式 VLA 的特殊推理模式抽象并落地到轻量 C++ runtime 中。它证明了多架构 VLA 可以用统一 bundle、统一协议、统一运行时部署到从消费级 GPU 到 Jetson Orin Nano 的不同硬件上。

论文最重要的发现包括：

- VLA 推理的主导阶段通常是 compute-bound 的视觉语言 prefix；
- 边缘部署中内存容量和计算单元利用率是两个关键杠杆；
- 低比特压缩解决 footprint，但速度需要 kernel 级优化；
- reduced precision 在多模态机器人策略中可能导致静默但严重的动作错误；
- 实际机器人闭环任务中，推理延迟会通过 observation staleness 直接影响成功率。

整体来看，这篇论文对 VLA 的工程化、边缘部署和机器人实用化具有较高参考价值。
