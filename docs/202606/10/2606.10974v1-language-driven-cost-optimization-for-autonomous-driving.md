## 摘要
自动驾驶车辆的驾驶行为通常由其运动规划器的成本函数所支配，该函数编码了速度跟踪、平顺性、车道保持和碰撞避免等目标。然而，调节塑造该成本函数的参数是一项具有挑战性的任务，需要技术专业知识，这限制了车辆适应不断演变的交通场景或终端用户偏好的能力。本文提出了一种用于自动驾驶自适应成本设计的语言驱动框架。大语言模型（LLM）解释结构化场景描述和自然语言用户查询，以生成应用于风险感知模型预测路径积分（MPPI）控制器的参数。该系统包含一个人在环路验证阶段，在部署前以非技术语言描述所提出的行为变化并进行确认。用户还可以在部署前或部署后提供反馈，从而实现对车辆运动行为的迭代优化。该框架在真实驾驶场景中的多种查询下进行评估，以考察其有效性。仿真结果表明，该方法能够以直观方式成功诱导符合预期需求的行为变化，从而弥合智能车辆控制系统与终端用户之间的差距。

## Abstract
The driving behavior of autonomous vehicles is typically governed by the cost function of their motion planner, which encodes objectives such as speed tracking, smoothness, lane keeping, and collision avoidance. However, tuning the parameters that shape this cost function is a challenging task that requires technical expertise, limiting the vehicle's ability to adapt to evolving traffic scenarios or end-user preferences. This work presents a language-driven framework for adaptive cost design in autonomous driving. A Large Language Model (LLM) interprets structured scenario descriptions and natural language user queries to generate the parameters applied to a risk-aware Model Predictive Path Integral (MPPI) controller. The system incorporates a human-in-the-loop validation stage in which the proposed behavioral changes are described in non-technical language and confirmed prior to deployment. Users may additionally provide feedback either before or after deployment, enabling iterative refinement of the vehicle's motion behavior. The framework is evaluated across multiple queries in realistic driving scenarios to assess its effectiveness. Simulation results demonstrate that the method successfully induces behavioral changes that align with the intended requirements in an intuitive manner, thereby bridging the gap between intelligent vehicle control systems and end users.

1. 获取当前交通场景描述和用户查询；

## 2.2 车辆动力学与问题形式化

论文将自车建模为离散时间非线性随机系统：

\[
x_{t+1} = F(x_t, v_t)
\]

其中控制输入满足：

\[
v_t \sim \mathcal{N}(u_t, \Sigma)
\]

含义是：

- \(x_t\)：车辆在时刻 \(t\) 的状态；
- \(u_t\)：名义控制输入；
- \(v_t\)：带随机扰动的实际采样控制；
- \(\Sigma\)：控制噪声协方差；
- \(F\)：车辆动力学模型。

目标是在给定场景特征 \(s_t\) 和用户查询集合 \(q_t\) 的情况下，构造对应的代价函数 \(J_{s_t, q_t}\)，并求解最优控制序列。

## 2.3 MPPI 控制器

本文采用 **Model Predictive Path Integral，MPPI** 作为底层运动规划器。

MPPI 的基本过程是：

1. 在当前名义控制序列周围采样 \(K\) 条候选控制序列；
2. 将每条控制序列通过车辆动力学模型前向传播，得到候选轨迹；
3. 对每条轨迹计算总代价；
4. 使用指数加权机制赋予低代价轨迹更高权重；
5. 对采样控制序列加权平均，得到更新后的控制序列；
6. 只执行第一个控制动作；
7. 下一时刻滚动优化。

其权重计算形式为：

\[
w_k = \frac{1}{\eta} \exp \left(-\frac{1}{\beta}(S_k - \rho)\right)
\]

其中：

- \(S_k\)：第 \(k\) 条轨迹的总代价；
- \(\beta\)：温度参数；
- \(\rho = \min_k S_k\)：用于数值稳定；
- \(\eta\)：归一化常数。

最终控制序列为：

\[
U^* = \sum_{k=1}^{K} w_k V_k
\]

## 2.4 代价函数设计

论文设计的 MPPI 代价函数为：

\[
S_k =
\omega_t C_{\text{tracking}}
+ \omega_s C_{\text{speed}}
+ \omega_{st} C_{\text{steering}}
+ C_{\text{control}}
+ C_{\text{risk}}
\]

主要组成包括：

- **路径跟踪代价 \(C_{\text{tracking}}\)**
  - 惩罚车辆偏离参考路径；
  - 用于车道保持或车道变换。

- **速度代价 \(C_{\text{speed}}\)**
  - 惩罚车辆偏离参考速度；
  - 权重大时车辆更积极地跟踪目标速度。

- **转向角代价 \(C_{\text{steering}}\)**
  - 惩罚过大的转向角；
  - 权重大时车辆更倾向于直线、平缓行驶。

- **控制输入代价 \(C_{\text{control}}\)**
  - 惩罚加速度和转向速率；
  - 用于提升舒适性和平顺性。

- **风险代价 \(C_{\text{risk}}\)**
  - 根据预测的周围车辆轨迹计算碰撞概率；
  - 鼓励车辆选择低风险轨迹。

## 2.5 风险建模

论文使用风险感知代价来约束车辆与周围交通参与者的交互。

对于周围车辆 \(a\)，通过预测模型得到其位置概率密度，并计算自车与该车发生碰撞的边界区域。单个智能体的碰撞概率为：

\[
\hat{P}(y_t^a) =
\int_{R_t^{\circ,a}}
\tilde{p}(y_t^a | \cdot) dy_t^a
\]

多个智能体的联合碰撞概率假设条件独立，计算为：

\[
\hat{P}_{joint}(Y_t)
=
1 -
\prod_{a \in A}
(1 - \hat{P}(y_t^a))
\]

风险代价定义为：

\[
C_{\text{risk}}
=
\omega_{\text{soft}} \hat{P}_{joint}(Y_t)
+
\omega_{\text{hard}} \mathbf{1}_{\sigma}(\hat{P}_{joint}(Y_t))
\]

其中：

- 第一项是软风险惩罚；
- 第二项是硬风险阈值惩罚；
- 当碰撞概率超过阈值时，施加强惩罚；
- \(\omega_{\text{hard}}\) 设置为很大且不由 LLM 调整；
- LLM 主要调整 \(\omega_{\text{soft}}\)。

## 2.6 LLM 参数生成机制

LLM 不直接输出连续控制动作，而是输出代价权重。

可调参数包括：

\[
\omega =
\{
\omega_t,
\omega_s,
\omega_{st},
\omega_a,
\omega_{str},
\omega_{\text{soft}}
\}
\]

对应的语义包括：

- 速度跟踪；
- 加速度惩罚；
- 转向速率惩罚；
- 转向角惩罚；
- 横向偏移惩罚；
- 碰撞风险惩罚。

为提高稳定性，论文没有让 LLM 直接输出连续权重，而是让其输出整数权重：

\[
\lambda_i \in \{1, \dots, 10\}
\]

然后通过归一化映射为连续权重：

\[
\omega = \hat{\omega} \frac{\lambda}{\bar{\lambda}}
\]

其中：

- \(\lambda\)：LLM 输出的整数权重；
- \(\bar{\lambda}\)：整数权重均值；
- \(\hat{\omega}\)：专家预设的名义权重尺度。

这样做的目的包括：

- 避免 LLM 直接生成浮点数时的不稳定；
- 利用 LLM 对整数等级的较好理解能力；
- 保持代价函数整体尺度稳定；
- 让 LLM 更关注不同代价项之间的相对重要性。

## 2.7 Prompt 设计

论文设计了结构化提示词，要求 LLM 按四步输出：

1. **意图解释**
   - 用文字重述用户的驾驶目标；
   - 说明应如何调整车辆行为。

2. **权重参数**
   - 输出所有代价权重；
   - 包括未改变的权重；
   - 使用 0–10 的整数形式。

3. **浮点参数**
   - 输出参考速度等浮点参数。

4. **行为变化描述**
   - 用非技术语言向用户解释车辆会如何改变；
   - 不提及代价函数、权重或控制参数。

此外，提示词中还包含：

- 每个代价项的自然语言解释；
- 行为规则示例，例如：
  - 若需要强加速，则提高速度跟踪权重、降低输入惩罚；
  - 若车辆应更慢并易于制动，则降低速度跟踪；
  - 若需要快速转向，则降低转向相关惩罚；
  - 若相邻车道有车辆，则提高横向偏移惩罚。

## 2.8 人在环验证与在线反馈

论文强调 LLM 可能误解用户意图，因此加入两个反馈环节：

### 部署前验证

- LLM 生成参数后，先给用户展示非技术性行为解释；
- 用户确认后才部署；
- 如果用户认为不符合预期，可以追加反馈；
- LLM 基于历史对话重新生成参数；
- 重复直到用户确认。

### 运行中在线细化

- 车辆运行过程中，用户可以继续输入新指令；
- 系统保留完整对话历史；
- 用户可以引用之前的调整，例如：
  - “保持平顺转向，但更快一点。”
- LLM 将新指令与历史上下文结合，更新代价参数。

### 实验一：基于用户查询的行为自适应

实验目标：

- 验证 LLM 是否能根据不同自然语言需求调整代价权重；
- 观察车辆在速度、加速度、转向、并道时间等指标上的变化；

### 实验二：验证循环与用户反馈

场景仍类似于并道场景。

实验过程：

- 用户先提出一个指令；
- LLM 给出行为解释；
- 用户拒绝或补充反馈；
- LLM 重新调整参数；
- 最终部署修正后的行为。

例如：

- 用户：“我有点晕。”
- LLM：“车辆将更平稳地驾驶。”

- LLM 重新生成正常驾驶策略。

该实验验证的是：

- 人在环机制是否能纠正 LLM 初始理解；
- 对话反馈是否能生成更符合用户意图的行为；
- 部署前验证是否能降低错误行为被执行的风险。

### 实验三：场景描述驱动的行为切换

场景：

- 匝道汇入场景；
- 自车遇到一辆非合作车辆；
- 自车需要先让行，再加速汇入主路。

设计了两个高层场景描述：

- **s1：安全优先**
  - 车辆必须优先保证安全；
  - 随时准备停车；
  - 不应进入没有明确安全间隙的冲突区域；
  - 保持运动不是优先目标。

- **s2：效率优先**
  - 车辆已经安全汇入车道；
  - 前方道路清晰；
  - 应快速加速匹配交通流；
  - 舒适性和平顺性为次要目标；
  - 当前无直接碰撞风险。

实验比较三种配置：

1. 全程使用 s1 生成的权重；
2. 全程使用 s2 生成的权重；
3. 根据场景阶段动态组合 s1 和 s2。

结果显示：

- s1 更保守，能避免风险，但汇入后加速恢复较慢；
- s2 更激进，速度恢复快，但在交互阶段可能发生碰撞；
- 动态组合 s1 和 s2 能在交互阶段保守、在安全后积极加速，表现更合理。

## 3.4 Benchmark 与对比方法

严格来说，论文没有进行传统意义上的 benchmark 对比。

- 没有与以下方法进行定量对比：
  - 固定权重 MPPI；
  - 人工专家调参；
  - 学习型 MPC 参数调节器；
  - 端到端 VLA 模型；
  - RL 或 imitation learning 驾驶策略；
  - 其他 LLM 驱动规划方法。

论文的对比主要是**同一场景下不同用户查询或场景描述导致的不同行为结果**，属于内部对照实验，而不是与外部 baseline 的系统比较。

# 6. 主要结论与发现

论文的主要结论包括：

- LLM 可以将自然语言用户需求转化为 MPPI 控制器的代价函数参数；
- 通过调整速度、平顺性、车道保持和风险相关权重，车辆行为能呈现出明显差异；
- 舒适性查询会提高加速度、转向速率、转向角等平顺性相关惩罚，使驾驶更平稳；
- 紧急或赶时间查询会提高速度参考或速度跟踪权重，并降低部分平顺性惩罚，使车辆更积极；
- 安全或保守查询会提高风险权重，使车辆更谨慎；
- 人在环验证机制可以在部署前发现并修正 LLM 对用户意图的误解；
- 场景描述不仅能表达用户偏好，也能表达交通上下文，从而触发不同驾驶模式；
- 将 LLM 与传统风险感知 MPPI 结合，可以在保持控制器实时性的同时，提升用户交互性和行为可解释性。

# 7. 优点与亮点

## 7.1 架构设计合理

- LLM 不直接控制车辆，而是调节代价函数参数；
- 底层仍由 MPPI 负责实时轨迹优化；
- 这种设计避免了让 LLM 直接输出控制命令带来的安全和实时性风险。

## 7.2 自然语言到控制参数的桥接清晰

- 用户可以用自然语言表达需求；
- LLM 将需求映射为可执行的控制器权重；
- 降低了非专家参与车辆行为调节的门槛。

## 7.3 风险感知 MPPI 保留安全结构

- 代价函数中显式加入碰撞概率风险项；
- 不是纯粹依赖 LLM 判断安全；
- 通过 MPPI 采样优化在动态交通交互中规划轨迹。

## 7.4 人在环验证提升可解释性与信任

- LLM 输出参数前后都配有自然语言解释；
- 用户只需理解车辆行为变化，而不需要理解控制权重；
- 部署前确认机制有助于避免误解被直接执行。

## 7.5 支持在线迭代反馈

- 系统保留对话历史；
- 用户可以连续细化驾驶风格；
- 支持类似“保持刚才的平顺性，但快一点”的上下文相关指令。

## 7.6 权重离散化设计实用

- 让 LLM 输出 1–10 的整数权重，而非浮点数；
- 降低数值输出错误风险；
- 保持代价尺度稳定；
- 更适合 LLM 的语言和序数推理能力。

# 8. 不足与局限

## 8.1 实验规模有限

- 主要实验集中在并道和匝道汇入场景；
- 未覆盖大量城市驾驶场景，例如：
  - 无保护左转；
  - 复杂交叉口；
  - 行人和骑行者交互；
  - 施工区；
  - 高速拥堵；
  - 多车博弈场景。

## 8.2 缺少系统 baseline 对比

论文没有与以下方法做定量比较：

- 固定代价权重 MPPI；
- 专家手动调参；
- 基于学习的 MPC 参数调节；
- 强化学习驾驶策略；
- 端到端语言驾驶模型；
- 其他 LLM 规划或决策框架。

因此无法明确量化本文方法相对于现有方法的性能收益。

## 8.3 对 LLM 可靠性的分析不足

- 使用 GPT-4o-mini 且 temperature 为 0，但仍可能出现：
  - 意图误解；
  - 输出格式错误；
  - 参数不合理；
  - 解释与实际参数不一致；
  - 对冲突指令处理不稳定。
- 论文虽然通过人在环机制部分缓解风险，但没有系统评估 LLM 失败模式。

## 8.4 人在环机制可能影响实时应用

- 部署前需要用户确认；
- 在紧急驾驶场景下，用户可能无法及时反馈；
- 对完全自动化系统来说，人类确认可能增加交互负担；
- 论文未量化 LLM 调用和人工确认带来的延迟。

## 8.5 场景描述依赖人工或外部模块

- 本文假设系统能够获得结构化或自然语言场景描述；
- 但实际自动驾驶系统需要从感知、预测和地图中自动生成这些描述；
- 论文未实现完整的感知到语言场景描述管线。

## 8.6 安全保证仍有限

- MPPI 中有风险代价，但这不是形式化安全证明；
- LLM 调低风险权重可能导致危险行为，如 s2 场景中出现碰撞；
- 论文提到未来可加入 safety filter，但当前方法尚未包含强安全屏障或可达性验证。

## 8.7 权重映射仍依赖专家设计

- 名义权重 \(\hat{\omega}\) 由专家手工调节；
- LLM 只是调整相对权重；
- 如果初始代价项设计或尺度不合理，LLM 输出也可能无法产生预期行为。

## 8.8 评估指标偏低层，缺少用户体验评价

- 论文主要使用速度、加速度、转向、距离、并道时间等运动指标；
- 没有真实用户研究；
- 没有评估用户是否真的认为车辆行为符合自然语言需求；
- 没有舒适性、信任度、可解释性的人因实验。

# 总体评价

本文提出了一种有价值的自动驾驶行为个性化框架：利用 LLM 解析自然语言需求和场景描述，再将其转化为风险感知 MPPI 的代价函数参数。它避免了让 LLM 直接控制车辆的高风险，同时保留了优化控制器的实时性和结构化安全设计。论文的主要贡献在于将自然语言交互、人在环验证和代价函数自适应结合起来，为非专家用户调整自动驾驶行为提供了清晰可行的接口。

不过，当前工作更像是一个概念验证系统。实验规模、baseline 对比、可靠性分析和安全保证仍然有限。若要走向实际自动驾驶应用，还需要更大规模闭环 benchmark、严格安全过滤、自动场景理解模块，以及对 LLM 输出稳定性和用户体验的系统评估。
