## 摘要
基于推理的端到端（E2E）自动驾驶近年来成为一种有前景的方法，可通过在预测轨迹的同时生成人类可读的推理内容来提高驾驶决策的可解释性。这类方法通常生成多条轨迹以刻画多样化的未来行为，并可分为两类：（1）多推理，即为每条轨迹生成一个推理序列；（2）单推理，即所有轨迹共享同一个推理。前者以冗余计算为代价提供更丰富的多样性，而后者效率更高，但通常被认为会牺牲多样性。Alpamayo 1 作为一个代表性系统，采用多推理方法并取得了具有竞争力的轨迹预测性能。然而，该设计的效率在很大程度上仍未得到探索，因此是一个具有充分研究动机的对象。本文从两个方面对 Alpamayo 1 进行系统分析与改进。首先，我们将 Alpamayo 1 重新设计为单推理系统，在保持轨迹多样性的同时降低推理时延。通过大量实验，我们发现以单推理替代多推理并不会显著降低轨迹多样性。其次，我们通过消除不必要的复制操作和低效内核执行所导致的块间开销，加速基于扩散的动作生成。通过闭环和开环实验，我们验证了这两项优化，结果表明在保持轨迹多样性和预测质量的同时，推理时延降低了 69.23%。这些结果凸显了联合分析系统架构与运行时执行对于提升基于推理的端到端自动驾驶系统效率的重要性。

## Abstract
Reasoning-based end-to-end (E2E) autonomous driving has recently emerged as a promising approach to improving the interpretability of driving decisions as it can generate human-readable reasoning together with predicted trajectories. Such approaches commonly generate multiple trajectories to capture diverse future behaviors, and they fall into two categories: (1) multi-reasoning, where one reasoning sequence is generated per trajectory, and (2) single-reasoning, where a single reasoning is shared across all trajectories. The former offers richer diversity at the cost of redundant computation, while the latter is more efficient but is often assumed to sacrifice diversity. Alpamayo 1, a representative system, adopts the multi-reasoning approach and achieves competitive trajectory prediction performance. However, the efficiency of this design remains largely unexplored, making it a well-motivated subject for investigation. In this paper, we systematically analyze and improve Alpamayo 1 in two ways. First, we reduce inference latency while preserving trajectory diversity by redesigning Alpamayo 1 into a single-reasoning system. Through extensive experiments, we find that replacing multi-reasoning with single-reasoning does not meaningfully degrade trajectory diversity. Second, we accelerate diffusion-based action generation by eliminating inter-block overhead arising from unnecessary copy operations and inefficient kernel execution. Through closed-loop and open-loop experiments, we validate both optimizations, demonstrating a 69.23% reduction in inference latency while maintaining trajectory diversity and prediction quality. These results highlight the importance of jointly analyzing system architecture and runtime execution to improve the efficiency of reasoning-based E2E AD systems.

## 2. 方法论：核心思想、关键技术细节与流程

### 2.1 Alpamayo 1 原始推理流程分析

论文首先对 Alpamayo 1 的官方开源推理代码进行系统拆解。其推理流程包括四个阶段：

1. **预处理 Preprocessing**
   - 输入包括：
     - 4 个摄像头最近 4 帧图像，即共 16 张图像；
     - 系统提示和用户提示；
     - 过去 1.6 秒的自车轨迹，共 16 个姿态点。
   - 图像由 image tokenizer 切分为视觉 token。
   - 文本由 text tokenizer 编码。
   - 历史轨迹由 trajectory tokenizer 离散化并映射到与文本共享的 token 空间。

2. **推理 Reasoning**
   - 使用视觉编码器、文本/轨迹编码器和语言解码器生成 CoT token。
   - 视觉编码器基于 Qwen3-VL，包含 27 个 Transformer block。
   - 语言解码器包含 36 个 Transformer block。
   - 解码过程分为：
     - **prefill 阶段**：并行处理输入上下文并构建 KV cache；
     - **decode 阶段**：自回归生成 CoT 推理 token。
   - 语言解码器生成的 KV cache 会直接作为后续动作生成模块的上下文条件。

3. **动作生成 Action Generation**
   - 使用扩散式动作生成。
   - 从随机初始化的动作序列开始，经过 10 次迭代逐步去噪和修正。
   - 每次迭代中：
     - action encoder 将当前动作序列映射为动作 embedding；
     - action decoder 使用 36 个 Transformer block 生成动作更新；
     - 输出的增量动作乘以 0.1 后加到当前动作序列上。
   - 动作序列包括未来 64 个时间步，每步 0.1 秒，总预测范围为 6.4 秒。
   - 动作由加速度和曲率表示，可转换为二维轨迹。

4. **后处理 Postprocessing**
   - 将 CoT token 反解码为自然语言。
   - 将动作序列通过单车运动学模型转换为未来轨迹。

### 2.2 方法一：从多推理架构改为单推理架构

#### 原始多推理架构

- Alpamayo 1 默认在生成多条轨迹时采用多推理方式。
- 对于生成 \(N\) 条轨迹：
  - 输入 token 被复制为 batch size \(N\)；
  - 推理模块生成 \(N\) 段不同 CoT；
  - 每段 CoT 产生对应的 KV cache；
  - 动作生成模块基于不同 KV cache 生成 \(N\) 条轨迹。
- 轨迹多样性来自两个随机源：
  - CoT token 采样的随机性；
  - 动作序列随机初始化的随机性。

#### 改造后的单推理架构

- 论文将 Alpamayo 1 改造为单推理共享多轨迹架构。
- 对于生成 \(N\) 条轨迹：
  - 只运行一次推理模块；
  - 生成一段 CoT 和一个 KV cache；
  - 将该 KV cache 沿 batch 维度复制为 \(N\) 份；
  - 动作生成模块使用 \(N\) 个随机初始化动作序列，生成 \(N\) 条轨迹。
- 轨迹多样性主要来自动作生成阶段的随机初始化。

#### 核心假设

- 如果 CoT 多样性对最终动作生成的影响有限，那么多推理带来的额外推理计算是冗余的。
- 扩散式动作生成本身的随机初始化已经足以提供多轨迹多样性。

### 2.3 方法二：优化扩散式动作生成中的 KV cache 管理

#### 原始动态 KV cache 的问题

- Alpamayo 1 原本使用动态 KV cache 管理。
- 每个 action decoder block 会：
  - 读取 reasoning 阶段产生的 KV cache；
  - 生成新的 action KV；
  - 将旧 KV 与新 KV 通过类似 `torch.cat` 的方式拼接到新分配的 buffer 中。
- 这种方式适合聊天机器人等输出长度不可预测的 LLM 场景。
- 但在 Alpamayo 的动作生成中，扩散迭代次数和 KV 增长模式是固定的，因此动态分配和复制造成了不必要开销。

#### 静态 KV cache 管理

- 论文提出预先分配最大 KV cache buffer，并在扩散过程中进行原地写入。
- 关键依据：
  - reasoning 阶段结束后，VLM 产生的 KV cache 大小已确定；
  - diffusion 每次迭代新增 KV 的大小固定；
  - 因此可以离线 profiling 得到最大 KV cache 尺寸。
- 文中给出的例子：
  - 单条轨迹、20 个 CoT token 时，语言模型产生的 KV cache 约为 **454.3 MB**；
  - 动作生成阶段新增约 **9.44 MB**；
  - 最大 KV cache 约为 **463.74 MB**。
- 优化后：
  - 预先分配最大内存；
  - 通过 KV pointer 指向每个 block 对应的写入位置；
  - 避免反复复制和重新分配。

### 2.4 方法三：使用 CUDA Graph 降低 GPU kernel launch 开销

- 在静态 KV cache 管理之后：
  - KV cache 地址固定；
  - 每次 diffusion iteration 的张量形状固定；
  - GPU kernel 执行序列固定。
- 因此可以使用 **CUDA Graph capture**：
  - 先记录一整段 GPU kernel launch 序列；
  - 后续 diffusion iteration 直接 replay graph；
  - 减少 CPU 发起 kernel launch 的开销；
  - 降低 kernel 之间的 GPU idle gap。
- 论文实现中：
  - 第一次迭代用于正常执行；
  - 第二次迭代进行 graph capture；
  - 从第三次迭代开始使用 CUDA Graph replay。

## 6. 主要结论与发现

- **多推理是 Alpamayo 1 的主要延迟瓶颈之一**：
  - 多推理架构中，reasoning 计算随轨迹数量近似线性增长；
  - 生成更多轨迹时，重复推理造成显著冗余。

- **单推理并未显著损害轨迹预测性能**：
  - open-loop：
    - multi-reasoning minADE6@6.4s 为 **0.725**；
    - single-reasoning 为 **0.743**；
    - 差距较小。
  - closed-loop：
    - multi-reasoning DTF 为 **131.58 m**；
    - single-reasoning DTF 为 **137.12 m**；
    - single-reasoning 甚至略优。

- **动作生成在单推理架构中会成为新的主要瓶颈**：
  - 因为 reasoning 不再随轨迹数量增长，action generation 的占比明显上升。
  - 例如 6 条轨迹时，action generation 在 single-reasoning 中占比可达约 **42.06%**。

- **静态 KV cache 和 CUDA Graph 能有效降低动作生成延迟**：
  - 单轨迹 action generation 中：
    - static KV cache 带来约 **21.66%** 延迟降低；
    - 再加入 CUDA Graph 后总降低约 **24.65%**。
  - 轨迹数量越多，优化收益越明显；
  - 6 条轨迹时，action generation latency 从 **2.09 s** 降至 **1.19 s**，降低约 **43.06%**。

- **整体优化效果显著**：
  - 在 6 条轨迹时，端到端延迟从 **13.33 s** 降至 **4.10 s**；
  - 总体降低 **69.23%**。

## 7. 优点与亮点

- **问题定位清晰**：
  - 论文不是泛泛讨论模型加速，而是针对 Alpamayo 1 的真实推理代码进行细粒度剖析。
  - 明确找到了两个主要瓶颈：
    - 多推理冗余；
    - 扩散动作生成中的 KV cache 与 kernel launch 开销。

- **优化不依赖重新训练**：
  - 不改变模型参数；
  - 不需要额外训练数据；
  - 改动集中在推理路径和运行时管理上，工程可迁移性较强。

- **挑战了常见假设**：
  - 论文通过实验说明，多推理并不一定显著提升轨迹多样性；
  - 在 Alpamayo 1 中，动作生成阶段的随机初始化已经能提供足够多样性。

- **架构优化与系统优化结合紧密**：
  - 单推理是架构层面的优化；
  - static KV cache 和 CUDA Graph 是运行时系统优化；
  - 两者结合后带来显著端到端收益。

- **实验包含开环和闭环评估**：
  - 不只看延迟；
  - 还验证了轨迹预测精度和闭环驾驶稳定性。

- **工程细节较具体**：
  - 说明了 KV cache 大小变化；
  - 解释了为什么静态 KV cache 可行；
  - 说明了 CUDA Graph capture 的使用条件和执行方式。

## 8. 不足与局限

- **实验平台单一**
  - 延迟 profiling 只在 NVIDIA DGX Spark / GB10 Grace Blackwell 上进行。
  - 没有在车载级 GPU、低功耗边缘设备或多种硬件平台上测试。
  - 因此实际部署收益可能因硬件而异。

- **闭环实验信息不够充分**
  - 虽然使用了 AlpaSim，但文中没有详细说明闭环测试场景数量、类别分布和统计显著性。
  - DTF 是作者自定义指标，与 Alpamayo 原论文的闭环指标可能不完全一致。

- **轨迹多样性缺少系统定量评估**
  - 论文主要通过可视化和最终预测性能说明多样性未下降。
  - 没有使用如 pairwise trajectory distance、多模态覆盖率、轨迹熵等专门多样性指标。

- **只验证 Alpamayo 1**
  - 尽管作者认为优化也可用于 Alpamayo 1.5，但没有提供 Alpamayo 1.5 的完整实验结果。
  - 对其他 VLA 自动驾驶模型的泛化性仍未验证。

- **单推理可能存在场景依赖风险**
  - 在某些高度模糊、多意图、多交互主体场景下，不同 reasoning 可能确实有助于覆盖不同语义解释。
  - 本文结果说明在当前测试集上单推理足够，但不能完全证明多推理在所有场景中都无价值。

- **端到端延迟仍较高**
  - 即使优化后，6 条轨迹的延迟仍约为 **4.10 秒**。
  - 对严格实时自动驾驶控制来说仍然偏高。
  - 论文也指出此类 VLA 模型更可能作为安全监管、辅助决策或高层推理模块，而非直接高频控制器。

- **未讨论优化对内存占用的整体影响**
  - 静态 KV cache 通过预分配内存减少动态开销，但可能提高峰值内存占用。
  - 文中给出了单轨迹 KV cache 大小示例，但没有系统分析多轨迹情况下的显存压力。
