## 摘要
我们提出 ReflectDrive-2，这是一种用于自动驾驶的掩码离散扩散规划器，配备独立的动作专家，将规划表示为离散轨迹 token，并通过并行掩码解码生成轨迹。这一离散 token 空间支持原位轨迹修订：AutoEdit 使用同一模型重写选定 token，而无需辅助精炼网络。为训练这一能力，我们采用两阶段流程。首先，我们沿纵向进展和横向航向方向构造专家轨迹的结构感知扰动，并监督模型恢复原始专家轨迹。随后，我们使用强化学习（RL）对完整的决策—草拟—反思 rollout 进行微调，将终端驾驶奖励分配给最终编辑后的轨迹，并通过完整 rollout 转移传播策略梯度信用。完整 rollout 强化学习对于耦合草拟与编辑至关重要：仅在监督训练下，推理时 AutoEdit 对 PDMS 的提升最多为 0.3，而强化学习将其增益提高到 1.9。我们还为决策—草拟—反思流程协同设计了一套高效的反思式解码栈，结合共享前缀 KV 复用、交替步解码以及融合的端上去掩码。在 NAVSIM 上，ReflectDrive-2 在仅使用相机输入时达到 91.0 PDMS，在 best-of-6 oracle 设置下达到 94.8 PDMS，同时在 NVIDIA Thor 上的平均延迟为 31.8 毫秒。

## Abstract
We introduce ReflectDrive-2, a masked discrete diffusion planner with separate action expert for autonomous driving that represents plans as discrete trajectory tokens and generates them through parallel masked decoding. This discrete token space enables in-place trajectory revision: AutoEdit rewrites selected tokens using the same model, without requiring an auxiliary refinement network. To train this capability, we use a two-stage procedure. First, we construct structure-aware perturbations of expert trajectories along longitudinal progress and lateral heading directions and supervise the model to recover the original expert trajectory. We then fine-tune the full decision--draft--reflect rollout with reinforcement learning (RL), assigning terminal driving reward to the final post-edit trajectory and propagating policy-gradient credit through full-rollout transitions. Full-rollout RL proves crucial for coupling drafting and editing: under supervised training alone, inference-time AutoEdit improves PDMS by at most $0.3$, whereas RL increases its gain to $1.9$. We also co-design an efficient reflective decoding stack for the decision--draft--reflect pipeline, combining shared-prefix KV reuse, Alternating Step Decode, and fused on-device unmasking. On NAVSIM, ReflectDrive-2 achieves $91.0$ PDMS with camera-only input and $94.8$ PDMS in a best-of-6 oracle setting, while running at $31.8$ ms average latency on NVIDIA Thor.

## 2. 方法论：核心思想、关键技术与流程

## 2.1 总体框架：Decision–Draft–Reflect

ReflectDrive-2 的输入包括：

- 多视角相机视觉 token：
  - 左前、前方、右前三路相机；
  - 使用两个时间帧。
- 导航 / 路线指令 token：
  - 如保持车道、左转、直行等。
- 自车状态 token：
  - 速度、加速度、横摆角速度等。

输出为未来 4 秒的轨迹：

- 共 8 个 waypoint；
- 每个 waypoint 用一个纵向坐标 token 和一个横向坐标 token 表示；
- 因此轨迹长度为：
  - `L = 16` 个离散轨迹 token。

整体推理流程为：

1. **Decision：目标点预测**
   - 模型预测离散 BEV 坐标上的目标点 posterior。
   - 每个目标点代表一个行为级假设，例如：
     - 保持车道；
     - 让行；
     - 超车；
     - 换道；
     - 不同转弯线路。
   - 推理时通过 top-k 采样和 NMS 得到多个空间上不同的目标点。

2. **Draft：掩码离散扩散生成轨迹**
   - 选定目标点后，将终点 token 固定；
   - 其余轨迹 token 初始化为 `[MASK]`；
   - 模型通过少量并行去掩码步骤生成完整轨迹。

3. **Reflect：AutoEdit 原位轨迹修正**
   - AutoEdit 使用同一个条件 token 模型；
   - 不引入额外 refinement network；
   - 对低置信度或需要修改的轨迹 token 进行直接 token-to-token 重写；
   - 目标 token 通常保持固定，作为行为锚点。

## 2.2 掩码离散扩散轨迹建模

论文将未来轨迹表示为离散 BEV 坐标 token 序列：

\[
x_0 = [x_1, y_1, \ldots, x_8, y_8]
\]

训练时采用掩码扩散：

- 给定真实轨迹 token `x0`；
- 以一定概率将部分 token 替换为 `[MASK]`；
- 模型根据视觉、语言、状态上下文预测原始 token。

与部分离散扩散语言模型只在 masked position 上计算损失不同，论文采用 **all-position 监督**：

- 即对所有轨迹位置都计算交叉熵；
- 作者认为这样训练更稳定，生成轨迹更连贯。

## 2.3 AutoEdit：结构感知自编辑机制

AutoEdit 的目标是修正草稿轨迹中常见的结构化规划错误。

### 结构感知扰动

论文构造了两类扰动来训练 AutoEdit：

1. **纵向进展扰动**
   - 沿轨迹弧长方向缩放进展速度；
   - `β < 1` 表示过于保守、前进不足；
   - `β > 1` 表示过冲或减速不足。

2. **横向航向扰动**
   - 在自车坐标系中旋转整条轨迹；
   - 模拟车道偏移、转弯切线错误、横向漂移等问题。

模型训练目标是：

- 输入扰动后的轨迹 token；
- 输出原始专家轨迹 token；
- 使 AutoEdit 学会从结构化错误中恢复。

### 推理时 AutoEdit

AutoEdit 推理时并不是重新 mask token，而是：

- 输入当前具体轨迹 token；
- 预测每个位置的替换 token；
- 根据 commit mask 选择性提交部分替换；
- 形成更新后的轨迹。

因此它是 **直接 token-to-token 编辑**，而不是重新去噪。

## 2.4 约束感知监督：Drivable-Area Field Loss

仅靠 token 预测损失无法显式约束可行驶区域，因此论文加入了可行驶区域场损失。

核心思想：

- 根据模型对每个 waypoint 的坐标 token 分布，构造 BEV 上的空间概率分布；
- 使用一个 drivable-area cost field 对不可行驶或高风险区域赋予更高代价；
- 若模型把高概率分配到不可行驶区域，则受到更大惩罚。

该损失主要提升：

- Drivable Area Compliance，简称 DAC；
- 轨迹对道路边界和可行驶区域的遵守程度。

最终监督训练目标包括三部分：

- 掩码离散扩散损失 `LDLM`；
- 结构感知扰动恢复损失 `LSAP`；
- 可行驶区域场损失 `Lfield`。

## 2.5 强化学习：完整 Draft-and-Edit Rollout 对齐

论文最关键的方法贡献是：  
**用强化学习优化完整的“草拟 + 编辑”流程，而不是只优化单独的生成器或单独的编辑器。**

具体流程：

- 对每个场景采样多个目标点；
- 每个目标点生成若干轨迹草稿；
- 每条轨迹经过 AutoEdit；
- 最终编辑后的轨迹获得一个闭环驾驶评分作为 terminal reward；
- 使用 group-relative advantage 计算相对优势；
- 策略梯度同时作用于：
  - drafting 阶段的 token 更新；
  - AutoEdit 阶段的 token 重写。

关键意义：

- 监督训练下，草稿生成器只优化 token-level imitation；
- AutoEdit 只学习恢复人工扰动；
- 两者之间没有真正的闭环奖励耦合。
- RL 后：
  - drafter 学会生成“可被编辑器改善”的草稿；
  - editor 学会做出真正提高驾驶评分的修改；
  - 自编辑不再是后处理，而是策略 rollout 的一部分。

## 2.6 高效推理解码设计

论文还针对部署效率设计了反思式解码栈：

- **共享前缀 KV cache**
  - 目标点预测、轨迹草拟、AutoEdit 共享相同视觉 / 指令 / 状态前缀；
  - 避免重复计算上下文 KV。

- **可变 action cache 回退与重写**
  - action token 是可变的；
  - 每次更新轨迹 token 后，只回退并重算 action block，而不是重算所有上下文。

- **Action-expert FFN**
  - 对 action token 使用较小的 FFN；
  - 将隐藏维度从 4096 降到 1024；
  - 降低轨迹 token 解码成本。

- **融合 CUDA 去掩码 / token 更新**
  - 将置信度排序、token 选择和状态更新合并到端上 kernel；
  - 减少 CPU-GPU 同步。

- **Alternating Step Decode, ASD**
  - 全步骤帧运行完整 decision–draft–reflect；
  - 轻量帧将上一帧轨迹变换到当前坐标系，再用短 AutoEdit 修正；
  - 使 AutoEdit 同时作为帧内修正器和帧间时序 refinement。

  - 对核心论点“RL 使 AutoEdit 真正有效”有直接 ablation 支撑。
  - 区分了标准单轨迹结果和 best-of-6 oracle 结果，避免把 oracle 结果当作公平 benchmark。
  - 同时报告了精度和延迟，体现了一定部署导向。

- **不足之处**：
  - 只在 NAVSIM 一个公开 benchmark 上做主要评估，外部泛化证据有限。
  - navtest 只有 136 个场景，测试规模相对较小。
  - 训练数据、预训练权重、RL reward 实现和部分工程细节不够透明。
  - 与部分 LiDAR 方法对比时，传感器输入不同，不能完全等价比较。
  - best-of-6 oracle 虽然能证明候选轨迹质量，但不是实际可部署决策性能。
  - 没有在真实道路闭环或高保真交互仿真中验证。

总体来看，论文实验对核心机制的验证较充分，但对真实泛化能力和训练可复现性的支撑仍有限。

## 6. 主要结论与发现

- ReflectDrive-2 在 NAVSIM camera-only 单轨迹设置下达到：
  - **91.0 PDMS**。
- 在 best-of-6 oracle 设置下达到：
  - **94.8 PDMS**；
  - 与 NAVSIM human reference 的 94.8 PDMS 持平。
- 与其他 camera-only VLA 方法相比：
  - AutoVLA：89.1 PDMS；
  - DriveVLA-W0：90.2 PDMS；
  - ReCogDrive：90.8 PDMS；
  - ReflectDrive-2：91.0 PDMS。
- 最大优势主要体现在：
  - Ego Progress 达到 89.4，是表中最高；
  - 同时保持较高 DAC 和舒适性。
- AutoEdit 的关键发现：
  - 仅监督训练时，推理时 AutoEdit 最多只带来 +0.3 PDMS；
  - 经过完整 draft-and-edit rollout RL 后，AutoEdit 增益提升到 +1.9 PDMS。
- 可行驶区域场损失显著提升 DAC：
  - DLM：84.8 PDMS；
  - DLM + DACF：87.2 PDMS；
  - 增益为 +2.4 PDMS。
- 部署效率方面：
  - 经过多项优化后，NVIDIA Thor 平均延迟为 31.8 ms / frame；
  - 表明该方法具有一定实时部署潜力。

## 7. 优点与亮点

## 7.1 方法设计亮点

- **离散 token 轨迹表示适合编辑**
  - 轨迹被表示为有限 token 序列；
  - 可以自然选择某些 token 进行重写；
  - 比连续轨迹修正更容易实现结构化编辑。

- **AutoEdit 不需要额外 refinement network**
  - 草拟和编辑共享同一模型；
  - 减少系统复杂度；
  - 保持 token 空间一致。

- **结构感知扰动贴合驾驶错误模式**
  - 纵向 progress 和横向 heading 是自动驾驶规划中常见失误方向；
  - 训练扰动不是随机噪声，而是有驾驶语义的错误模拟。

- **完整 rollout RL 是核心创新**
  - 把 drafting 和 editing 作为一个统一策略过程；
  - 终端 reward 同时优化两个阶段；
  - 解决了监督训练下编辑器与生成器脱耦的问题。

- **目标点 posterior 提供行为多模态**
  - 目标点不仅是终点预测，也代表行为假设；
  - 支持换道、让行、不同转弯线路等多种可能行为。

## 7.2 实验与工程亮点

- **核心 ablation 清晰**
  - 特别是 Table 3 直接证明 RL 对 AutoEdit 的放大作用。

- **同时关注性能与延迟**
  - 论文不仅报告 PDMS，还详细报告部署优化链。

- **ASD 设计有实际部署意义**
  - 利用相邻帧时序连续性；
  - 降低重复完整规划的开销。

- **标准结果与 oracle 结果区分清楚**
  - 没有把 best-of-6 oracle 作为主要公平对比结论；
  - 这一点相对客观。

## 8. 不足与局限

## 8.1 实验覆盖有限

- 主要公开实验集中在 NAVSIM；
- 缺少在更多数据集或 benchmark 上的验证，例如：
  - nuPlan 原生闭环设置；
  - Waymo；
  - Argoverse；
  - CARLA；
  - 真实车端闭环测试。
- NAVSIM navtest 场景数量为 136，规模有限，可能不足以覆盖长尾复杂交互场景。

## 8.2 训练和复现透明度不足

- 模型使用 proprietary pretrained weights；
- 未报告详细训练算力：
  - GPU 型号；
  - GPU 数量；
  - 训练时间；
  - batch size；
  - RL 采样成本。
- 这会影响学术复现和公平比较。

## 8.3 与部分基线比较存在输入差异

- ReflectDrive-2 是 camera-only；
- 一些 end-to-end baseline 使用 Camera + LiDAR；
- 虽然 ReflectDrive-2 在 PDMS 上超过部分 LiDAR 方法，但这种比较并非严格同等输入条件。
- 更公平的重点应是 camera-only VLA 方法之间的比较。

## 8.4 Best-of-6 oracle 不代表真实部署能力

- Best-of-6 达到 94.8 PDMS，但依赖 oracle 选择最高分轨迹；
- 实际部署中无法直接知道哪条候选轨迹 PDMS 最高；
- 因此该结果更多说明多模态候选质量，而非真实在线决策性能。

## 8.5 动作表示存在精度上限

- 轨迹使用固定分辨率 BEV 坐标 token；
- 空间精度受 token bin size 限制；
- 对需要高精度控制的场景可能不够细腻。
- 作者也指出未来可以考虑：
  - 更细坐标词表；
  - residual offsets；
  - 离散-连续混合 action head。

## 8.6 AutoEdit 扰动类型仍较有限

- 当前 AutoEdit 主要训练两类扰动：
  - 纵向速度 / 进展；
  - 横向航向偏移。
- 但真实驾驶中的失败还包括：
  - 让行时机错误；
  - cut-in 反应不足；
  - gap selection 错误；
  - 多智能体博弈失败。
- 这些复杂交互错误尚未被充分建模。

## 8.7 RL reward 仍是代理目标

- RL 使用闭环规划评分作为 terminal reward；
- 虽然高效，但仍是现实驾驶目标的近似；
- 在真实交通中，安全、舒适、礼让、法规和人类驾驶偏好更复杂。
- 需要更高保真交互仿真或真实数据闭环验证。

# 总体评价

ReflectDrive-2 的核心贡献在于把自动驾驶轨迹规划从“一次性生成”改造成“可编辑、可反思、可通过强化学习对齐”的离散 token 决策过程。论文证明，仅靠监督训练得到的编辑器效果有限，而将草拟与编辑纳入同一个 RL rollout 后，AutoEdit 才能显著提升最终驾驶评分。方法在 NAVSIM 上取得了很强的 camera-only 结果，并通过一系列工程优化展示了实时部署潜力。不过，论文在训练算力透明度、跨数据集泛化、真实交互验证和 oracle 设置解释方面仍存在局限。
