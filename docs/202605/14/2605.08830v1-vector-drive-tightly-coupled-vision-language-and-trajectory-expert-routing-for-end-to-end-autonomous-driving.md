## 摘要
端到端自动驾驶要求模型理解交通场景、推断驾驶意图，并生成可执行的运动规划。近期的视觉-语言-动作（VLA）模型继承了大规模视觉-语言预训练中的语义先验，但仍面临耦合权衡：完全共享的骨干网络能够保留多模态交互，却可能使语言推理与轨迹预测相互纠缠；而解耦的推理-动作流水线可减少任务冲突，但会削弱语义与运动之间的耦合。我们提出 VECTOR-DRIVE，一种基于 Qwen2.5-VL-3B 构建的紧耦合 VLA 框架。VECTOR-DRIVE 通过共享自注意力保持所有 token 的耦合，并根据 token 语义对前馈计算进行路由。视觉和语言 token 由视觉-语言专家处理，以保留语义先验；而目标点、自车状态和含噪动作 token 则被路由至轨迹专家，以进行面向运动的计算。在动作 token 通路上，流匹配规划器将含噪动作 token 细化为未来路点和速度曲线。该设计在单一多模态 Transformer 内耦合语义推理与运动规划，同时分离特定任务的 FFN 计算。在 Bench2Drive 上，VECTOR-DRIVE 取得了 88.91 的驾驶评分，并优于具有代表性的端到端和基于 VLA 的基线方法。定性结果和消融实验进一步验证了共享注意力、语义感知专家路由、渐进式训练以及基于流的动作解码的优势。

## Abstract
End-to-end autonomous driving requires models to understand traffic scenes, infer driving intent, and generate executable motion plans. Recent vision-language-action (VLA) models inherit semantic priors from large-scale vision-language pretraining, yet still face a coupling trade-off: fully shared backbones preserve multimodal interaction but may entangle language reasoning and trajectory prediction, whereas decou pled reasoning-action pipelines reduce task conflict but weaken semantic-motion coupling. We propose VECTOR-DRIVE, a tightly coupled VLA framework built on Qwen2.5-VL-3B. VECTOR-DRIVE keeps all tokens coupled through shared self attention and routes feed-forward computation according to token semantics. Vision and language tokens are processed by a Vision-Language Expert to preserve semantic priors, while target-point, ego-state, and noisy action tokens are routed to a Trajectory Expert for motion-specific computation. On the action-token pathway, a flow-matching planner refines noisy action tokens into future waypoints and speed profiles. This design couples semantic reasoning and motion planning within a single multimodal Transformer while separating task-specific FFN computation. On Bench2Drive, VECTOR-DRIVE achieves 88.91 Driving Score and outperforms representative end-to end and VLA-based baselines. Qualitative results and ablations further validate the benefits of shared attention, semantic-aware expert routing, progressive training, and flow-based action de coding.

- **核心问题**  
  现有 VLA 自动驾驶模型在“语义理解”和“轨迹规划”之间存在明显的结构性权衡：
  - **完全共享式 VLM/VLA 架构**：所有任务共用同一个 Transformer 和 FFN 参数，能够保持视觉、语言、动作之间的强交互，但语言推理通常偏离散生成，轨迹预测则是连续几何回归，两者优化目标和归纳偏置不同，容易产生任务干扰。
  - **解耦式推理-动作架构**：先由 VLM 进行高层语义理解，再由独立动作模块生成轨迹，能够缓解任务冲突，但语义信息往往只通过文本摘要、高层决策或粗粒度 latent conditioning 传递给规划模块，削弱了细粒度语义到运动规划的耦合。

- **论文整体含义**  
  本文提出的 **VECTOR-Drive** 试图在两者之间取得平衡：  
  **在注意力层面保持所有 token 的共享交互，在 FFN 层面根据 token 语义进行专家路由。**  
  这意味着模型既能保留视觉-语言-动作之间的细粒度上下文耦合，又能让语言理解和轨迹生成分别拥有更适合自身任务的前馈计算路径，从而提升端到端自动驾驶的闭环性能。

## 2. 方法论：核心思想、关键技术细节与流程

### 2.1 总体架构

- VECTOR-Drive 基于 **Qwen2.5-VL-3B** 构建，是一个统一的多模态 VLA Transformer。
- 输入包括：
  - 前视 RGB 图像 `I`
  - 全局目标提示 `G`
  - 目标点 `P`
  - 高层导航命令 `U`
  - 自车状态 `S = [v, a, θ, θ˙]`
  - 含噪动作 token `ZA`
- 输出包括：
  - 语言层面的链式思考与驾驶指令 `O`
  - 未来空间路径 `Yp`：20 个路点，每米一个点
  - 未来速度曲线 `Yv`：10 个时间步速度

整体映射为：

- 输入：`I, G, P, U, S`
- 输出：`O, Yp, Yv`

即模型同时完成场景理解、语言推理和可执行轨迹规划。

---

### 2.2 统一 Token 化

论文将所有模态映射到统一 token 空间：

- 图像编码为视觉 token：`ZI`
- 全局目标和高层命令编码为语言 token：`ZG, ZU`
- 目标点和自车状态通过轻量 MLP 投影为：`ZP, ZS`
- 含噪动作状态 `Xτ` 和时间嵌入 `e(τ)` 经过动作预处理器得到：`ZA(τ)`

随后将这些 token 交错组织为统一序列：

```text
Z = Interleave(ZG, ZI, ZP, ZU, ZS, ZA)
```

这种设计使模型能够在一个 Transformer 序列中同时处理语义、导航、自车状态和动作生成。

---

### 2.3 共享自注意力 + 语义感知 FFN 专家路由

VECTOR-Drive 的核心设计是：

> **所有 token 共享 self-attention，但不同语义类型的 token 进入不同 FFN 专家。**

具体流程如下：

1. **共享自注意力**
   - 所有 token 先经过同一个 self-attention 层。
   - 视觉、语言、导航、自车状态和动作 token 可以在注意力中交换上下文。
   - 动作 token 因此能够直接感知语义信息、目标点和驾驶条件。

2. **根据 token 类型进行静态路由**
   - 视觉和语言相关 token：
     - `ZG, ZI, ZU`
     - 路由至 **Vision-Language Expert**
   - 轨迹相关 token：
     - `ZP, ZS, ZA`
     - 路由至 **Trajectory Expert**

3. **专家 FFN 计算**
   - 每个专家均采用 gated FFN 结构，包括 gate、up、down 投影。
   - Vision-Language Expert 负责保持和适配语言/视觉语义先验。
   - Trajectory Expert 负责运动规划相关的连续几何计算。

4. **合并回原始 token 顺序**
   - 两类专家输出再按原始交错顺序合并，进入下一层 Transformer。

这种设计不是学习型 top-k MoE，而是**基于 token 类型的静态路由**，因此不需要额外 router 参数，也不需要负载均衡损失。

---

### 2.4 注意力掩码设计

为避免动作信息泄漏到语言推理，论文设计了因果多模态注意力掩码：

- 动作 token 放在序列末尾。
- 动作 token 可以关注前面的视觉、语言、导航和自车状态 token。
- 语言侧 token 不能看到未来动作 token。
- 含噪动作 token 内部允许双向注意力，以增强轨迹一致性。

这保证了：

- 轨迹生成能够利用语义信息；
- 语言生成不会被未来动作状态污染。

---

### 2.5 语言生成分支

语言分支基于最终 Vision-Language hidden states 进行自回归生成：

- 生成内容包括：
  - Chain-of-Thought 场景推理
  - 简洁驾驶指令

该分支通过 LM Head 完成，目标是增强模型对交通场景、意图和行为约束的语义理解。

---

### 2.6 流匹配动作规划器

论文在动作 token 通路上引入 **Flow Matching Planner**，用于将高斯噪声逐步变换为未来轨迹和速度曲线。

#### 空间路径生成

- 真实未来路径为 `Yp`
- 高斯噪声为 `ε`
- 插值时间 `τ ∈ [0, 1]`
- 含噪路径状态：

```text
Xτ = (1 - τ)ε + τYp
```

- 目标向量场：

```text
v* = Yp - ε
```

模型学习预测从噪声到真实轨迹的连续向量场。

推理时：

1. 从高斯噪声 `X0 ~ N(0, I)` 初始化路径。
2. 通过条件 ODE：

```text
dXτ / dτ = v̂ω(Xτ, τ | condition)
```

3. 用 Euler ODE solver 迭代求解。
4. 最终得到 20 个未来路点。

#### 速度曲线生成

- 速度分支采用相同的 flow-matching 思路。
- 预测 10 步未来速度。
- 路径和速度使用统一的多模态条件，但分别有独立的 Flow Head。

---

### 2.7 多模态条件设计

Flow Head 的条件来自两类信息：

- **隐式语义条件**
  - 从最终 Vision-Language hidden states 投影得到。
  - 表示场景语义、交通规则、交互意图等高层信息。

- **显式驾驶条件**
  - 自车状态
  - 导航表示
  - 目标点

最终将动作 hidden states、隐式语义条件和显式驾驶条件拼接后输入 Flow Head，预测轨迹向量场。

### 2.8 三阶段训练策略

论文采用渐进式三阶段训练：

1. **Stage I：Vision-Language Learning**
   - 冻结轨迹专家和运动头。

   - 目标：学习场景理解、语义推理和驾驶指令生成。

3. **Stage III：Joint Full-Parameter Fine-Tuning**
   - 解冻所有模块。
   - 联合优化语言损失和驾驶损失。
   - 目标：对齐语义推理和轨迹规划。

  - 完全解耦 VLM 性能明显较低，说明粗粒度语义条件不足以支撑高质量闭环规划。

  - 三阶段训练优于直接 joint training，说明先适配语义、再学习驾驶技能、最后联合对齐更稳定。
  - Flow Matching 解码器优于回归解码和 DDPM 解码，说明连续向量场建模更适合生成可执行轨迹。

- 论文核心发现是：  
  **在注意力层保持语义-运动强耦合，同时在 FFN 层分离任务计算，是提升 VLA 自动驾驶闭环性能的有效结构设计。**

---

## 7. 优点与亮点

- **结构设计清晰且有针对性**
  - 针对 VLA 自动驾驶中“全共享”和“全解耦”的矛盾，提出共享 attention 与专家 FFN 路由的中间方案。

- **语义与运动耦合更细粒度**
  - 所有 token 在每层 self-attention 中交互，使动作 token 能持续吸收视觉、语言、导航和状态信息。

- **任务专用计算缓解干扰**
  - Vision-Language Expert 与 Trajectory Expert 分离不同 FFN 计算，有助于减少语言生成和轨迹规划之间的优化冲突。

- **静态语义路由简单稳定**
  - 不依赖复杂 learned router。
  - 避免负载均衡损失。
  - 计算路径可解释性更强。

- **Flow Matching Planner 适合连续轨迹生成**
  - 从噪声逐步生成轨迹和速度曲线，较回归头更能建模复杂未来运动分布。
  - 相比 DDPM，ODE 求解流程更直接，10 步 Euler 推理较简洁。

- **实验覆盖了关键组件**
  - 对路由、耦合、训练策略、解码器都有消融验证。
  - 闭环评估比单纯 open-loop 更有说服力。

- **结果表现突出**
  - 在 Bench2Drive 上超过多个代表性端到端和 VLA 方法，尤其在合流、超车、制动和交通标志遵守等交互能力上提升明显。

---

## 8. 不足与局限

- **泛化验证不足**
  - 实验主要集中在 CARLA/Bench2Drive，缺少 nuScenes、Waymo、真实车端数据或其他仿真环境验证。
  - 难以判断模型是否能泛化到真实道路长尾场景。

- **数据来源可能带来偏差**
  - 训练数据由 PDM-lite 规则专家生成。
  - 模型可能学习到规则专家的行为模式和局限，而不是真正具备开放环境下的鲁棒驾驶能力。

- **缺少推理效率分析**
  - 方法基于 Qwen2.5-VL-3B，并引入双专家 FFN 与流匹配迭代解码。
  - 论文没有报告推理帧率、延迟、显存占用和车端部署成本。
  - 对实时自动驾驶系统而言，这是关键缺失。

- **实验公平性仍需更多说明**
  - 不同基线可能使用不同输入模态、训练数据规模或模型参数量。
  - 论文未充分说明所有方法是否在严格相同设置下比较。

- **统计可靠性不足**
  - 没有报告多随机种子实验。
  - 没有置信区间或标准差。
  - 难以确认小幅提升是否稳定显著。

- **语言能力评估较弱**
  - 虽然模型生成 CoT 和驾驶指令，但语言质量只通过可视化展示。
  - 没有语义一致性、可解释性、幻觉率或安全相关语言错误的量化指标。

- **安全性验证有限**
  - Bench2Drive 虽然包含多种交互场景，但论文没有系统测试极端长尾风险，例如：
    - 传感器噪声
    - 遮挡
    - 异常交通参与者
    - 对抗性语言指令
    - 罕见交通规则冲突

- **Flow Matching 推理成本可能较高**
  - 轨迹生成需要 10 步 Euler ODE 迭代，每步都要处理当前 noisy action token。
  - 相比单步回归方法，实时性和计算开销需要进一步评估。
