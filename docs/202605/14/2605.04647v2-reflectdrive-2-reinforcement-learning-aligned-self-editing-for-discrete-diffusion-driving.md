## 摘要
我们提出 ReflectDrive-2，这是一种用于自动驾驶的掩码离散扩散规划器，配备独立的动作专家，将规划表示为离散轨迹 token，并通过并行掩码解码生成轨迹。这一离散 token 空间支持原位轨迹修订：AutoEdit 使用同一模型重写所选 token，而无需辅助精修网络。为训练这种能力，我们采用两阶段流程。首先，我们沿纵向进展和横向航向方向构造专家轨迹的结构感知扰动，并监督模型恢复原始专家轨迹。随后，我们使用强化学习（RL）对完整的决策--草拟--反思 rollout 进行微调，将终端驾驶奖励分配给最终编辑后的轨迹，并通过完整 rollout 转移传播策略梯度信用。完整 rollout 强化学习对于耦合草拟与编辑至关重要：仅在监督训练下，推理时 AutoEdit 对 PDMS 的提升最多为 $0.3$，而强化学习将其增益提高到 $1.9$。我们还为决策--草拟--反思流程协同设计了一套高效的反思式解码栈，结合共享前缀 KV 复用、交替步解码（Alternating Step Decode）以及融合的设备端去掩码。在 NAVSIM 上，ReflectDrive-2 在仅使用摄像头输入时达到 $91.0$ PDMS，在 best-of-6 oracle 设置下达到 $94.8$ PDMS，同时在 NVIDIA Thor 上的平均延迟为 $31.8$ 毫秒。

## Abstract
We introduce ReflectDrive-2, a masked discrete diffusion planner with separate action expert for autonomous driving that represents plans as discrete trajectory tokens and generates them through parallel masked decoding. This discrete token space enables in-place trajectory revision: AutoEdit rewrites selected tokens using the same model, without requiring an auxiliary refinement network. To train this capability, we use a two-stage procedure. First, we construct structure-aware perturbations of expert trajectories along longitudinal progress and lateral heading directions and supervise the model to recover the original expert trajectory. We then fine-tune the full decision--draft--reflect rollout with reinforcement learning (RL), assigning terminal driving reward to the final post-edit trajectory and propagating policy-gradient credit through full-rollout transitions. Full-rollout RL proves crucial for coupling drafting and editing: under supervised training alone, inference-time AutoEdit improves PDMS by at most $0.3$, whereas RL increases its gain to $1.9$. We also co-design an efficient reflective decoding stack for the decision--draft--reflect pipeline, combining shared-prefix KV reuse, Alternating Step Decode, and fused on-device unmasking. On NAVSIM, ReflectDrive-2 achieves $91.0$ PDMS with camera-only input and $94.8$ PDMS in a best-of-6 oracle setting, while running at $31.8$ ms average latency on NVIDIA Thor.

## 2. 方法论：核心思想、关键技术细节与流程

### 2.1 总体框架：ReflectDrive-2

ReflectDrive-2 是一个面向自动驾驶规划的 **掩码离散扩散 VLA 规划器**，输入包括：

- 多视角摄像头视觉 token；
- 导航 / 路线指令语言 token；
- 自车状态 token，如速度、加速度、横摆角速度等。

输出为未来 4 秒的自车轨迹，表示为离散 BEV 坐标 token。

整体流程为：

1. **Decision：目标点预测**
   - 模型预测一个目标点后验分布。
   - 每个目标点对应一个行为假设，例如保持车道、让行、变道、超车、不同转弯路径等。

2. **Draft：掩码离散扩散轨迹草拟**
   - 将未来轨迹离散化为 token。
   - 从全掩码序列开始，通过少量并行去掩码步骤生成轨迹。

3. **Reflect：AutoEdit 自编辑**
   - 对初始草稿中部分低置信度或可能错误的轨迹 token 进行重写。
   - 编辑过程使用同一个条件 token 模型，不需要额外 refinement 网络。

---

### 2.2 离散轨迹表示

- 论文将未来 4 秒轨迹表示为 8 个 waypoint。
- 每个 waypoint 包含一个纵向坐标 token 和一个横向坐标 token。
- 因此轨迹序列长度为：

\[
x_0 = [x_1, y_1, ..., x_8, y_8]
\]

共 16 个离散坐标 token。

- 最后一个 waypoint \((x_8, y_8)\) 被视为目标点 token，用来锚定行为假设。

---

### 2.3 掩码离散扩散生成

- 训练时，模型随机将轨迹 token 替换为 `[MASK]`。
- 模型需要根据视觉、语言、状态上下文恢复原始轨迹 token。
- 与部分离散扩散语言模型只监督 masked 位置不同，本文采用 **全位置交叉熵监督**：
  - 即无论 token 是否被 mask，都参与预测损失。
  - 作者认为这能带来更稳定的优化和更连贯的轨迹草稿。

推理时：

- 从全 mask 的轨迹 token 开始；
- 目标点 token 固定；
- 其余 token 通过少量并行去掩码步骤生成；
- 每一步选择置信度最高的一批 token 提交。

---

### 2.4 AutoEdit：同一 token 空间中的轨迹自编辑

AutoEdit 是本文的核心模块之一。

其特点是：

- 不引入额外 refinement 网络；
- 不把 token 重新变成 `[MASK]`；
- 而是直接输入当前具体轨迹 token，并预测替换 token；
- 只提交被选中的 token 替换结果。

AutoEdit 的更新形式可以概括为：

\[
x^{(k+1)} = m^{(k)} \odot \hat{x}^{(k+1)} + (1 - m^{(k)}) \odot x^{(k)}
\]

其中：

- \(x^{(k)}\)：第 \(k\) 轮编辑前轨迹；
- \(\hat{x}^{(k+1)}\)：模型预测的替换轨迹 token；
- \(m^{(k)}\)：提交掩码，决定哪些 token 被替换；
- 目标点 token 通常保持固定，作为行为锚点。

---

### 2.5 结构感知扰动监督

为了让 AutoEdit 学会修正真实驾驶中常见错误，论文构造两类结构化扰动：

#### 纵向进度扰动

通过沿轨迹弧长缩放进度：

\[
\tilde{z}_i = Interp(z_0, \beta d_i)
\]

- \(\beta < 1\)：模拟过于保守、行驶不足；
- \(\beta > 1\)：模拟过冲、刹车不足或速度过快。

#### 横向航向扰动

通过在自车坐标系下旋转轨迹：

\[
\tilde{z}_i = R(\alpha) z_i
\]

- 用于模拟横向偏移、转弯角度错误、车道偏离等。

训练目标是让模型从扰动轨迹 token 恢复专家轨迹 token：

\[
L_{SAP} = - \mathbb{E}_{x_0,T} \frac{1}{L} \sum_i \log q_\theta(x_i^0 | \tilde{x}_0, c)
\]

### 2.6 可行驶区域约束场损失

除了 token 级 imitation loss，论文还加入了可行驶区域约束损失。

核心思想：

- 将模型对每个 waypoint 的坐标 token 概率转化为 BEV 空间分布；
- 根据可行驶区域构造 cost field；
- 如果模型把较高概率分配给不可行驶或高风险区域，则施加惩罚。

该损失可以理解为：

\[
L_{field} = \sum_t \sum_{i,j} -\log(1 - p_{xy}^{(t)}[i,j]) C[i,j]
\]

其中：

- \(p_{xy}^{(t)}\)：第 \(t\) 个 waypoint 的空间概率分布；
- \(C[i,j]\)：BEV 栅格上的可行驶区域代价。

### 2.7 强化学习对齐：完整 draft-and-edit rollout

这是论文最重要的设计之一。

监督训练虽然能让模型具备编辑能力，但论文发现：

- 仅监督训练下，推理时 AutoEdit 对 PDMS 的提升最多只有 **+0.3**；
- 加入完整 rollout 强化学习后，AutoEdit 的提升达到 **+1.9 PDMS**。

强化学习流程：

1. 对每个场景采样多个目标点；
2. 每个目标点生成多个轨迹草稿；
3. 对草稿执行 AutoEdit；
4. 用最终编辑后的轨迹计算闭环驾驶奖励；
5. 将奖励通过策略梯度同时分配给：
   - 目标选择；
   - 草拟阶段 token 更新；
   - AutoEdit 阶段 token 重写。

奖励使用 group-relative advantage：

\[
A_g = R(\tau_g) - \frac{1}{G} \sum_j R(\tau_j)
\]

其中：

- \(R(\tau_g)\)：候选轨迹的闭环规划得分；
- \(G\)：同一场景下采样的候选轨迹数量。

关键点在于：

- 奖励只看最终 post-edit 轨迹；
- 但梯度同时作用于 drafting 和 editing 的 token 转换；
- 因此草拟器学会生成“可被编辑器改善”的草稿，编辑器学会执行真正提升驾驶得分的修改。

### 2.8 高效推理设计

论文还针对部署设计了反思式推理优化栈：

- **共享前缀 KV cache**
  - 视觉、导航、状态等上下文在目标预测、轨迹草拟、AutoEdit 中共享。
  - 避免重复计算前缀。

- **Action-expert FFN**
  - 针对短轨迹 token 块使用更小的动作专家 FFN。
  - 将隐藏维度从 4096 降到 1024。

- **融合 CUDA 去掩码 / token 更新**
  - 将置信度排序、token 选择、状态更新融合到设备端 kernel 中，减少 CPU-GPU 同步。

- **Alternating Step Decode, ASD**
  - 在连续帧中交替使用：
    - full-step：完整 decision-draft-reflect；
    - lite-step：将上一帧轨迹变换到当前自车坐标系，再用 AutoEdit 快速修正。
  - 使 AutoEdit 同时承担帧内轨迹反思和帧间时间连续性修正的作用。

8. **目标点数量与 NMS 阈值敏感性实验**
   - 分析 goal proposal 数量和 NMS threshold 对 best-of-N 表现的影响。

9. **定性可视化**
   - 目标点多样性；
   - AutoEdit 对轨迹的实际修正效果。

### 5.2 实验是否充分

总体来看，实验设计较丰富，覆盖了：

- 主结果；
- 组件消融；
- 推理效率；
- 多模态决策能力；
- 自编辑效果；
- 部署可行性。

尤其是 Table 3 对 AutoEdit 在监督训练和 RL 后的收益进行了直接对照，较好地支撑了论文的核心论点。

### 5.3 客观性与公平性问题

但仍存在一些需要谨慎解读的地方：

- **数据集单一**
  - 主要实验集中在 NAVSIM。
  - 没有在更多真实闭环仿真 benchmark 或实车数据上验证。

- **部分对比输入模态不同**
  - ReflectDrive-2 是 camera-only；
  - 一些端到端 baseline 使用 camera + LiDAR。
  - 虽然 camera-only 取得更高 PDMS 是亮点，但不同输入模态下的比较仍需谨慎。

- **Best-of-6 是 oracle 设置**
  - 作者已说明这不是标准 benchmark 结果。
  - 该结果更适合作为多模态候选质量分析，而不是实际部署性能。

- **预训练权重为 proprietary**
  - 模型从私有预训练权重初始化。
  - 这会影响复现性，也使得与公开方法的公平对比存在一定不确定性。

- **训练算力未公开**
  - 无法评估方法训练成本和可复现门槛。

## 6. 主要结论与发现

- ReflectDrive-2 在 NAVSIM 标准单轨迹 camera-only 设置下达到 **91.0 PDMS**。
- 在 best-of-6 oracle 设置下达到 **94.8 PDMS**，与 NAVSIM human reference 相当。
- 监督训练下，AutoEdit 的推理增益很小，最多约 **+0.3 PDMS**。
- 经过完整 draft-and-edit rollout 强化学习后，AutoEdit 的推理增益提升到 **+1.9 PDMS**。
- 这说明：
  - 编辑器本身被监督训练出来并不够；
  - 关键在于通过终端驾驶奖励让草拟器和编辑器协同适配。
- 目标点后验能够产生多样化行为假设，而不只是同一轨迹附近的噪声采样。
- 高效解码栈可以将平均推理延迟降低到约 **30.2 ms / frame**，具备一定部署潜力。
- AutoEdit 不仅可用于单帧内轨迹反思，还可用于连续帧间的 temporal refinement。

## 7. 优点与亮点

### 7.1 方法设计亮点

- **离散 token 空间适合原地编辑**
  - 相比连续轨迹表示，token 化轨迹天然支持局部替换和自编辑。

- **统一模型完成生成与编辑**
  - AutoEdit 不需要额外 refinement 网络。
  - 降低系统复杂度，也使 drafting 和 editing 更容易联合优化。

- **结构感知扰动贴近驾驶错误分布**
  - 纵向进度扰动和横向航向扰动对应自动驾驶规划中的常见失败模式。
  - 比随机扰动更有针对性。

- **强化学习完整耦合 draft 与 edit**
  - 论文没有只对最终轨迹做简单 reranking，而是将策略梯度分配到草拟和编辑阶段。
  - 这是核心创新点之一。

- **目标点后验显式建模多模态行为**
  - 目标点作为行为假设锚点，使系统能表达不同驾驶意图。

### 7.2 实验设计亮点

- 消融实验直接验证了：
  - DAC field loss 的作用；
  - AutoEdit supervised training 的作用；
  - full-rollout RL 的作用；
  - 推理时 AutoEdit 在 RL 前后的收益差异。

- 同时报告规划质量和推理延迟，兼顾算法效果与部署效率。

- Best-of-6 oracle 被明确标注为分析性结果，而非标准结果，这一点较为严谨。

### 7.3 工程部署亮点

- 共享前缀 KV cache、action-expert FFN、CUDA fused unmasking 和 ASD 构成完整优化链。
- 不是单一 trick，而是围绕 decision-draft-reflect 架构协同设计。
- ASD 将上一帧计划作为当前帧初始草稿，再用 AutoEdit 修正，较符合自动驾驶连续帧场景。

## 8. 不足与局限

### 8.1 实验覆盖不足

- 主要只在 NAVSIM 上验证。
- 缺少在更多 benchmark 上的泛化测试，例如：
  - nuPlan 原始闭环环境；
  - CARLA；
  - Waymo / Argoverse 类真实驾驶数据；
  - 更复杂多智能体交互场景。

### 8.2 真实闭环交互验证有限

- NAVSIM 是数据驱动的非反应式仿真 benchmark。
- 论文中的 RL reward 仍是轻量闭环规划分数代理。
- 该设置未必完全反映真实交通中其他智能体对自车行为的反应。

### 8.3 训练与复现透明度不足

- 使用 proprietary pretrained weights。
- 未报告训练 GPU 数量、训练时间、总计算量等。
- 外部研究者较难完全复现结果。

### 8.4 Best-of-6 结果存在 oracle 偏差

- Best-of-6 需要知道哪个候选轨迹 PDMS 最高。
- 实际部署中无法直接获得 oracle PDMS。
- 因此 94.8 PDMS 更像是候选集上限，而不是实际系统性能。

### 8.5 AutoEdit 扰动类型仍有限

目前 AutoEdit 的结构扰动主要覆盖：

- 纵向进度；
- 横向航向。

但实际自动驾驶错误还包括：

- 让行时机错误；
- cut-in 车辆响应错误；
- 间隙选择错误；
- 多车交互博弈失败；
- 交通规则语义理解错误。

论文也承认未来需要扩展到 interaction-level failures。

### 8.6 离散坐标表示存在精度上限

- 轨迹 waypoint 被离散化为 BEV coordinate token。
- 生成精度受限于坐标 bin size。
- 更高精度可能需要：
  - 更细 token vocabulary；
  - residual offset；
  - hybrid discrete-continuous action head。

### 8.7 安全性指标并非全面领先

- 虽然 PDMS 总分最高，但 ReflectDrive-2 在 NC 和 TTC 上并非所有方法中最好。
- 其主要提升来自 EP，即 ego progress。
- 因此需要进一步确认其 progress 提升是否会在更复杂交互环境中带来潜在风险。

---

## 总结

ReflectDrive-2 的核心贡献是将自动驾驶规划建模为一个可编辑的离散 token 生成过程，并通过完整 rollout 强化学习将轨迹草拟和自编辑统一对齐到最终驾驶奖励。实验表明，单纯训练一个编辑器并不能显著提升规划表现，而让草拟器和编辑器在同一奖励信号下协同优化，才是 AutoEdit 获得明显收益的关键。该方法在 NAVSIM 上取得了领先的 camera-only PDMS，并通过系统级推理优化展示了一定部署潜力。不过，论文仍受限于单一 benchmark、私有预训练权重、训练资源不透明、真实交互验证不足等问题，其实际道路部署价值仍需更广泛验证。
