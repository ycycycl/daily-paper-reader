## 摘要
端到端自动驾驶要求模型理解交通场景、推断驾驶意图，并生成可执行的运动规划。近期的视觉-语言-动作（VLA）模型继承了大规模视觉-语言预训练所获得的语义先验，但仍面临一种耦合权衡：完全共享的骨干网络能够保留多模态交互，却可能使语言推理与轨迹预测相互纠缠；而解耦的推理-动作流水线虽能减少任务冲突，却会削弱语义与运动之间的耦合。我们提出 VECTOR-DRIVE，这是一个基于 Qwen2.5-VL-3B 构建的紧耦合 VLA 框架。VECTOR-DRIVE 通过共享自注意力保持所有 token 的耦合，并根据 token 语义对前馈计算进行路由。视觉和语言 token 由视觉-语言专家处理，以保留语义先验；而目标点、自车状态和含噪动作 token 则被路由至轨迹专家，以进行面向运动的专门计算。在动作 token 通路上，基于流匹配的规划器将含噪动作 token 细化为未来航点和速度剖面。该设计在单一多模态 Transformer 中耦合语义推理与运动规划，同时分离任务特定的 FFN 计算。在 Bench2Drive 上，VECTOR-DRIVE 达到 88.91 的驾驶评分，并优于具有代表性的端到端和基于 VLA 的基线方法。定性结果与消融实验进一步验证了共享注意力、语义感知专家路由、渐进式训练以及基于流的动作解码所带来的优势。

## Abstract
End-to-end autonomous driving requires models to understand traffic scenes, infer driving intent, and generate executable motion plans. Recent vision-language-action (VLA) models inherit semantic priors from large-scale vision-language pretraining, yet still face a coupling trade-off: fully shared backbones preserve multimodal interaction but may entangle language reasoning and trajectory prediction, whereas decou pled reasoning-action pipelines reduce task conflict but weaken semantic-motion coupling. We propose VECTOR-DRIVE, a tightly coupled VLA framework built on Qwen2.5-VL-3B. VECTOR-DRIVE keeps all tokens coupled through shared self attention and routes feed-forward computation according to token semantics. Vision and language tokens are processed by a Vision-Language Expert to preserve semantic priors, while target-point, ego-state, and noisy action tokens are routed to a Trajectory Expert for motion-specific computation. On the action-token pathway, a flow-matching planner refines noisy action tokens into future waypoints and speed profiles. This design couples semantic reasoning and motion planning within a single multimodal Transformer while separating task-specific FFN computation. On Bench2Drive, VECTOR-DRIVE achieves 88.91 Driving Score and outperforms representative end-to end and VLA-based baselines. Qualitative results and ablations further validate the benefits of shared attention, semantic-aware expert routing, progressive training, and flow-based action de coding.

## 1. 核心问题与整体含义：研究动机与背景

### 1.1 研究背景

端到端自动驾驶希望直接从多模态输入中生成可执行的未来运动轨迹，通常包括：

- 摄像头图像；
- 导航目标；
- 高层驾驶指令；
- 自车状态；
- 未来轨迹或控制动作。

近年来，视觉-语言-动作模型（Vision-Language-Action, VLA）逐渐被引入自动驾驶领域。这类模型通常基于大规模视觉语言模型（VLM/MLLM），能够利用预训练模型中已有的语义理解能力，从而提升复杂交通场景、长尾交互场景中的认知与决策能力。

### 1.2 核心问题

论文指出，现有 VLA 自动驾驶模型主要存在一种“耦合权衡”：

#### 路线一：完全共享骨干网络

即使用一个共享的 VLM/LLM backbone 同时处理：

- 视觉-语言语义理解；
- 链式推理；
- 轨迹预测或控制动作生成。

优点是多模态交互充分，语义信息可以直接影响动作生成。

但缺点是：

- 语言推理通常是离散的自回归生成任务；
- 轨迹规划是连续的几何回归或生成任务；
- 两者优化目标、归纳偏置和梯度行为差异较大；
- 完全共享参数可能导致任务干扰，影响运动规划精度。

#### 路线二：推理与动作生成解耦

即 VLM 负责高层语义理解和推理，另一个单独的动作模块负责轨迹或控制生成。

优点是降低语言推理和连续控制之间的任务冲突。

但缺点是：

- 语义到动作之间通常只通过粗粒度接口连接；
- 例如文本摘要、高层决策或隐变量条件；
- 细粒度的语义约束难以充分传递到轨迹生成过程；
- 在交互式驾驶场景中，语义-运动耦合不足可能导致规划不够稳健。

### 1.3 论文整体意义

VECTOR-Drive 的核心目标是解决上述矛盾：

> 既保持视觉、语言、动作 token 之间的紧密交互，又避免语言推理和轨迹规划在 FFN 计算中相互干扰。

因此，论文提出一种“共享注意力 + 专家路由 FFN”的紧耦合 VLA 架构：

- 通过共享 self-attention 保持所有 token 的跨模态交互；
- 通过基于 token 类型的专家 FFN 路由分离语言语义计算和轨迹运动计算；
- 通过流匹配规划器将噪声动作 token 逐步细化为未来路径和速度剖面。

## 2. 方法论：核心思想、关键技术与流程

## 2.1 整体架构

VECTOR-Drive 构建在 Qwen2.5-VL-3B 之上，是一个端到端视觉-语言-动作框架。

输入包括：

- 前视 RGB 图像 `I`；
- 全局目标提示 `G`；
- 目标点 `P`；
- 高层驾驶指令 `U`；
- 自车状态 `S`；
- 噪声动作表示 `ZA`。

输出包括：

- 语言层面的链式推理和驾驶指令 `O`；
- 未来空间路径 `Yp`；
- 未来速度剖面 `Yv`。

论文将自车状态定义为：

```text
S = [v, a, θ, θ_dot]
```

其中：

- `v`：速度；
- `a`：加速度；
- `θ`：航向角；
- `θ_dot`：偏航角速度。

轨迹输出包括：

- 20 个未来路径点，每米一个 waypoint；
- 10 个未来速度预测值。

---

## 2.2 统一 token 化

论文将所有模态映射到统一隐藏空间：

- 全局目标提示 `G` → 文本 token `ZG`；
- 高层命令 `U` → 文本 token `ZU`；
- 图像 `I` → 视觉 token `ZI`；
- 目标点 `P` → MLP 投影为 `ZP`；
- 自车状态 `S` → MLP 投影为 `ZS`；
- 噪声动作状态 `Xτ` 和时间嵌入 `e(τ)` → 动作 token `ZA(τ)`。

这些 token 被交错排列为统一序列：

```text
Z = Interleave(ZG, ZI, ZP, ZU, ZS, ZA)
```

这种设计使视觉、语言、导航、自车状态与动作 token 能够在同一个 Transformer 序列中交互。

---

## 2.3 共享自注意力 + 语义感知专家路由

这是 VECTOR-Drive 的核心创新。

每一层 Transformer 分为两个阶段：

### 阶段一：共享 self-attention

所有 token 首先进入共享自注意力层。

作用是：

- 让视觉、语言、导航、状态和动作 token 相互感知；
- 保持语义信息与运动规划信息的紧密耦合；
- 使动作 token 能在每一层聚合语义和驾驶条件信息。

论文中的形式为：

```text
H_tilde = H + Attn(LN(H))
```

其中 attention 的 Q、K、V、O 投影是共享的。

---

### 阶段二：按 token 类型路由到不同 FFN 专家

共享 attention 后，模型根据 token 类型将 token 分为两组：

#### 视觉-语言专家 Vision-Language Expert

处理：

- 全局目标 token；
- 图像 token；
- 高层命令 token。

即：

```text
ZG, ZI, ZU
```

主要负责：

- 场景理解；
- 语言推理；
- 驾驶意图表达；
- 保留 VLM 语义先验。

#### 轨迹专家 Trajectory Expert

处理：

- 目标点 token；
- 自车状态 token；
- 噪声动作 token。

即：

```text
ZP, ZS, ZA
```

主要负责：

- 轨迹相关计算；
- 动作表示更新；
- 运动规划和速度预测。

---

## 2.4 静态语义路由而非学习式路由

论文采用的是静态 token-type-aware routing，而不是 MoE 中常见的 learned top-k routing。

也就是说：

- 每类 token 的专家归属由语义类型固定决定；
- 不额外引入 router 参数；
- 不需要负载均衡损失；
- 避免路由不稳定问题。

论文也做了 learned routing 的消融，发现 learned routing 与静态路由性能几乎相当，因此作者选择更简单、稳定的静态语义路由。

---

## 2.5 注意力掩码设计

为了防止动作 token 泄漏到语言生成中，论文使用因果多模态 attention mask：

- 噪声动作 token 内部允许双向注意力；
- 其他 token 使用因果注意力；
- 动作 token 放在序列末尾；
- 动作 token 可以关注前面的语义和驾驶条件；
- 语言 token 不能访问未来动作状态。

这样可以保证：

- 轨迹生成被语义条件充分指导；
- 语言生成不会受到未来动作信息泄漏的影响。

## 2.6 语言生成分支

最终的视觉-语言 hidden states 被送入 LM Head，进行自回归语言生成。

输出包括：

- Chain-of-Thought 场景推理；
- 简洁驾驶指令；
- 当前交通行为解释。

语言生成目标为标准交叉熵损失：

```text
L_lang = -Σ log p(o_k | o_<k, H_vl)
```

## 2.7 基于流匹配的动作规划器

论文使用 flow matching 生成未来路径和速度，而不是直接回归轨迹。

### 空间路径生成

设真实未来路径为 `Yp`，高斯噪声为 `ε`，插值时间为 `τ ∈ [0,1]`。

构造噪声路径状态：

```text
Xτ = (1 - τ) ε + τ Yp
```

目标向量场为：

```text
v* = Yp - ε
```

模型学习预测从噪声到真实轨迹的连续向量场。

训练时，Flow Head 输入：

- routed action hidden states；
- 隐式语义条件；
- 显式驾驶条件。

输出预测向量场。

推理时：

1. 从高斯噪声 `X0` 初始化；
2. 使用条件 ODE：

```text
dXτ / dτ = v_hat(Xτ, τ | C)
```

3. 通过 Euler solver 迭代；
4. 得到最终未来路径 `Yp_hat`。

论文中推理使用 10 步 Euler ODE solver。

---

### 速度剖面生成

速度分支与空间路径分支使用类似的 flow matching 机制。

区别是：

- 空间分支生成 20 个 waypoint；
- 速度分支生成 10 个未来速度值；
- 二者共享统一的多模态条件；
- 但使用不同的 Flow Head。

## 2.8 三阶段训练策略

论文提出 progressive training，以稳定语义学习、轨迹学习和联合优化。

### Stage I：视觉-语言学习

冻结：

- motion heads。

训练：

- Vision-Language Expert；

目标：

损失：


### Stage II：驾驶技能学习

训练：

总驾驶损失为：


其中论文设置：

解冻全部参数，联合优化：

- 行为阶段变化。

### 3. 架构消融实验

对比：

- Decoupled VLM；
- Weakly Coupled VLM；
- Learned Routing；
- Single Action-Oriented Expert；
- VECTOR-Drive。

验证：

- 共享注意力；
- 专家 FFN 路由；
- 静态语义路由；
- 视觉-语言专家保留的作用。

### 4. 三阶段训练消融

对比：

- Joint training only；
- w/o Stage I VL learning；
- w/o Stage II driving learning；
- w/o Stage III joint fine-tuning；
- Full three-stage training。

验证 progressive training 的必要性。

### 5. 动作解码头消融

对比：

- VLM + Regression；
- VLM + DDPM；
- VECTOR-Drive 的 flow matching planner。

验证 flow matching 动作解码的优势。

### 6. CoT 与驾驶指令可视化

展示模型生成的场景推理和驾驶指令，说明语义推理与行为较为一致。

## 5.2 实验充分性评价

整体来看，实验较为完整，尤其是：

- 使用闭环 benchmark，而非仅 open-loop 指标；
- 对比了多种代表性方法；
- 有主结果、定性分析和多组消融；
- 消融覆盖了架构、训练策略和解码器三个核心因素；
- 细分 Multi-Ability 指标能体现交互式驾驶能力。

但也存在不足：

- 只在 Bench2Drive / CARLA 仿真环境中验证；
- 缺少真实道路数据集或实车实验；
- 未报告统计方差或多随机种子结果；
- 未明确所有基线是否在完全相同训练数据和设置下比较；
- 训练数据为自采集 300 万帧，可能使与部分基线的比较存在数据规模差异；
- 没有详细讨论推理延迟和部署成本。

因此，论文实验对验证方法本身是比较充分的，但对泛化能力、真实世界适用性和公平性仍需要进一步证据。

# 6. 主要结论与发现

论文的主要结论可以概括为：

## 6.1 共享注意力 + 专家 FFN 路由是有效的

VECTOR-Drive 表明：

- 完全共享计算会导致语言和轨迹任务干扰；
- 完全解耦会削弱语义与动作之间的细粒度联系；
- 共享 self-attention 保持跨模态交互；
- 专家 FFN 分离任务特定计算；
- 两者结合能更好地平衡语义推理和运动规划。

## 6.2 静态语义路由足够有效

Learned Routing 的 DS 为 88.85，SR 为 71.82%。

VECTOR-Drive 静态路由为：

- DS：88.91；
- SR：71.82%。

两者几乎相同，说明：

- 主要性能收益来自 shared attention 和专家 FFN 分离；
- 复杂 learned router 并非必要；
- 静态语义路由更简单、稳定、无额外负载均衡损失。

## 6.3 三阶段训练优于直接联合训练

直接 joint training only：

- DS：85.67；
- SR：69.09%。

完整三阶段训练：

- DS：88.91；
- SR：71.82%。

说明：

- 先适配视觉语言能力；
- 再专门学习驾驶技能；
- 最后联合微调；

比从一开始直接联合训练更加稳定。

## 6.4 Flow Matching 比回归和 DDPM 更优

动作解码消融结果：

- Regression：86.01 DS；
- DDPM：87.67 DS；
- Flow Matching：88.91 DS。

说明连续向量场建模更适合将噪声动作 token 细化为可执行轨迹和速度剖面。

## 6.5 交互式驾驶能力提升明显

VECTOR-Drive 在：

- merging；
- overtaking；
- braking；
- traffic-sign compliance；

等场景中表现突出，说明其语义理解能较好转化为可执行运动规划。

# 7. 优点与亮点

## 7.1 架构设计清晰且有针对性

论文没有简单地选择“完全共享”或“完全解耦”，而是提出中间方案：

- attention 层共享；
- FFN 层分专家；
- token 层面静态路由。

这种设计很好地对应了 VLA 自动驾驶中的核心矛盾。

---

## 7.2 保持语义-动作紧耦合

动作 token 在每一层都能通过共享 attention 关注：

- 视觉场景；
- 语言指令；
- 导航目标；
- 自车状态。

因此，相比粗粒度 conditioning，VECTOR-Drive 的语义到轨迹传递更细致。

---

## 7.3 降低任务干扰

视觉-语言 token 和轨迹相关 token 分别进入不同 FFN 专家，有助于减少：

- 语言生成任务；
- 连续轨迹规划任务；

之间的参数竞争和梯度冲突。

---

## 7.4 Flow Matching 适合连续轨迹生成

相比单步回归，flow matching 能够建模从噪声到轨迹的连续生成过程；

相比 DDPM，flow matching 通常推理路径更直接，本文也通过消融证明其在 Bench2Drive 上效果更好。

---

## 7.5 实验覆盖核心组件

论文不仅报告主结果，还系统消融了：

- 耦合方式；
- 路由方式；
- 专家结构；
- 训练策略；
- 动作解码器。

这些实验较好地支撑了方法有效性。

---

## 7.6 闭环评测更有说服力

Bench2Drive 是闭环自动驾驶 benchmark，比单纯 open-loop 轨迹误差更能体现：

- 累积误差；
- 决策稳定性；
- 场景交互能力；
- 真实驾驶行为质量。

---

# 8. 不足与局限

## 8.1 仅在仿真环境中验证

实验主要基于 CARLA / Bench2Drive。

虽然 Bench2Drive 场景丰富，但仍然属于仿真环境，不能完全代表真实道路中的：

- 传感器噪声；
- 复杂交通参与者行为；
- 罕见长尾事件；
- 地图误差；
- 天气与光照真实分布；
- 人类驾驶不确定性。

---

## 8.2 缺少真实世界数据或实车验证

论文没有展示：

- nuScenes、Waymo、Argoverse 等真实数据集结果；
- 真实车辆闭环测试；
- sim-to-real 迁移能力。

因此，目前结论更适用于仿真闭环驾驶，不足以证明真实道路可部署性。

---

## 8.3 公平性存在潜在疑问

训练集为作者自采集的约 300 万帧 CARLA 数据，并使用 PDM-lite 专家生成。

论文没有详细说明：

- 所有基线是否使用相同训练数据；
- 数据规模是否一致；
- 数据采集策略是否对 VECTOR-Drive 更有利；
- 各基线是否被重新训练或引用原结果。

因此，虽然结果领先，但跨方法比较的绝对公平性仍需谨慎解读。

---

## 8.4 缺少统计显著性分析

论文未报告：

- 多随机种子结果；
- 方差或置信区间；
- 多次运行稳定性。

在闭环自动驾驶评测中，路线随机性和交通交互可能导致结果波动，因此缺少统计分析是一个限制。

---

## 8.5 工程部署信息不足

虽然论文给出训练 GPU 和 epoch 数，但没有报告：

- 推理延迟；
- FPS；
- 单帧规划耗时；
- 模型参数量变化；
- 与 Qwen2.5-VL-3B 原模型相比的额外开销；
- 10 步 ODE 求解是否能满足实时驾驶要求。

对于自动驾驶系统而言，实时性和稳定性非常关键。

---

## 8.6 前视单目输入可能限制感知能力

论文问题设定中主要使用前视 RGB 图像。

这可能限制模型对以下情况的理解：

- 侧后方车辆；
- 盲区交通参与者；
- 多车道交互；
- 复杂路口全局交通结构。

如果要部署到更复杂场景，可能需要扩展到多摄像头、多传感器或 BEV 表示。

---

## 8.7 CoT 语言输出未被定量评估

论文展示了 CoT 和 instruction 可视化，但未单独评估语言输出质量，例如：

- 推理是否忠实；
- 是否存在幻觉；
- 是否与真实交通规则一致；
- 语言解释与轨迹行为是否严格一致。

因此，语言推理部分更多作为辅助展示，而非严格验证过的安全解释模块。

---

# 总结性评价

VECTOR-Drive 是一篇针对 VLA 自动驾驶中“语义推理与轨迹规划如何耦合”的方法论文。其核心贡献在于提出共享 self-attention 与语义感知专家 FFN 路由相结合的架构，使模型既能保持跨模态信息交互，又能减少语言任务和运动任务之间的计算干扰。再结合 flow matching 动作规划器和三阶段训练策略，模型在 Bench2Drive 上取得了领先闭环表现。

总体而言，该工作方法设计合理、实验消融较完整，尤其对 VLA 自动驾驶架构设计有较强参考价值。但其验证仍主要局限于 CARLA 仿真闭环环境，缺少真实道路、实时性、统计稳定性和公平训练条件的进一步证明。
