## 摘要
基于推理的端到端（E2E）自动驾驶近来成为一种颇具前景的方法，可通过在预测轨迹的同时生成便于人类理解的推理过程来提升驾驶决策的可解释性。此类方法通常会生成多条轨迹以捕捉多样化的未来行为，并可分为两类：（1）多推理，即为每条轨迹生成一个推理序列；（2）单推理，即所有轨迹共享一个推理过程。前者以冗余计算为代价提供更丰富的多样性，而后者效率更高，但通常被认为会牺牲多样性。Alpamayo 1 作为一个代表性系统，采用多推理方法并取得了具有竞争力的轨迹预测性能。然而，这一设计的效率在很大程度上尚未得到探索，因此是一个具有充分研究动机的课题。本文从两个方面系统地分析并改进 Alpamayo 1。首先，我们将 Alpamayo 1 重新设计为单推理系统，在保持轨迹多样性的同时降低推理延迟。通过大量实验，我们发现用单推理替代多推理并不会显著降低轨迹多样性。其次，我们通过消除由不必要的复制操作和低效的核函数执行所导致的块间开销，加速基于扩散的动作生成。通过闭环和开环实验，我们验证了这两项优化，证明在保持轨迹多样性和预测质量的同时，推理延迟降低了 69.23%。这些结果强调了联合分析系统架构与运行时执行对于提升基于推理的端到端自动驾驶系统效率的重要性。

## Abstract
Reasoning-based end-to-end (E2E) autonomous driving has recently emerged as a promising approach to improving the interpretability of driving decisions as it can generate human-readable reasoning together with predicted trajectories. Such approaches commonly generate multiple trajectories to capture diverse future behaviors, and they fall into two categories: (1) multi-reasoning, where one reasoning sequence is generated per trajectory, and (2) single-reasoning, where a single reasoning is shared across all trajectories. The former offers richer diversity at the cost of redundant computation, while the latter is more efficient but is often assumed to sacrifice diversity. Alpamayo 1, a representative system, adopts the multi-reasoning approach and achieves competitive trajectory prediction performance. However, the efficiency of this design remains largely unexplored, making it a well-motivated subject for investigation. In this paper, we systematically analyze and improve Alpamayo 1 in two ways. First, we reduce inference latency while preserving trajectory diversity by redesigning Alpamayo 1 into a single-reasoning system. Through extensive experiments, we find that replacing multi-reasoning with single-reasoning does not meaningfully degrade trajectory diversity. Second, we accelerate diffusion-based action generation by eliminating inter-block overhead arising from unnecessary copy operations and inefficient kernel execution. Through closed-loop and open-loop experiments, we validate both optimizations, demonstrating a 69.23% reduction in inference latency while maintaining trajectory diversity and prediction quality. These results highlight the importance of jointly analyzing system architecture and runtime execution to improve the efficiency of reasoning-based E2E AD systems.

## 2. 方法论：核心思想、关键技术与流程

### 2.1 Alpamayo 1 原始推理流程分析

论文首先拆解了 Alpamayo 1 的运行时推理架构。其流程可分为四个模块：

1. **预处理 Preprocessing**
   - 输入：
     - 四路相机图像：front、front-zoom、left、right；
     - 文本 prompt；
     - 过去 1.6 秒的自车轨迹，共 16 个 pose，10 Hz 采样。
   - 处理方式：
     - 图像被切分为 patch token；
     - 文本经过 text tokenizer；
     - 历史轨迹经 trajectory tokenizer 转为离散 token。

2. **推理 Reasoning**
   - 包含：
     - vision encoder；
     - text / trajectory encoder；
     - language decoder；
     - token sampler。
   - 视觉编码器基于 Qwen3-VL，包含 27 个 transformer blocks。
   - 语言解码器包含 36 个 transformer blocks。
   - 推理阶段分为：
     - **prefill**：并行处理全部输入 token，构建初始 KV cache；
     - **decode**：自回归生成 CoT 推理 token。
   - 生成的 CoT token 和对应 KV cache 会作为后续动作生成的条件。

3. **动作生成 Action Generation**
   - 使用 action decoder 生成未来动作序列。
   - 动作序列形式为：

     \[
     a = \{(a_i, k_i)\}_{i=1}^{64}
     \]

     其中：
     - \(a_i\)：第 \(i\) 个未来时间步的加速度；
     - \(k_i\)：第 \(i\) 个未来时间步的曲率；
     - 每个时间步间隔 0.1 秒；
     - 因此预测未来 6.4 秒。

   - 动作生成采用扩散式迭代 refinement：
     - 初始为随机动作序列；
     - 经过 10 次迭代；
     - 每次迭代中 action encoder 将动作序列编码为 embedding；
     - action decoder 结合 action embedding 和推理阶段生成的 KV cache，输出动作更新量；
     - 更新量乘以 0.1 后加入当前动作序列。

4. **后处理 Postprocessing**
   - 将动作序列通过 unicycle dynamics 转换为未来轨迹：

     \[
     \tau = \{(x_i, y_i, \theta_i^{yaw})\}_{i=1}^{64}
     \]

   - 同时将 CoT token detokenize 为人类可读文本。

### 2.2 方法一：从多推理改为单推理

#### 原始多推理架构

- 对于生成 \(N\) 条轨迹，Alpamayo 1 原始设计会复制输入 token，形成 batch size 为 \(N\) 的输入。
- 每份输入都会独立经过：
  - vision encoder；
  - language decoder prefill；
  - language decoder decode；
  - CoT 生成；
  - KV cache 生成。
- 然后每条轨迹基于各自的 KV cache 进行动作生成。
- 多样性来自两个随机源：
  - 推理阶段 token sampling 的随机性；
  - 动作生成阶段随机初始化动作序列。

#### 论文提出的单推理架构

- 作者将 Alpamayo 改造为：
  - 只生成一次 CoT 推理；
  - 只构建一份 KV cache；
  - 然后将这份 KV cache 沿 batch 维复制；
  - 用同一个推理上下文生成多条随机初始化的动作序列。
- 单推理架构的多样性只来自：
  - 动作序列随机初始化；
  - 扩散式动作生成过程。
- 核心假设是：

  > 多条不同 CoT 推理未必能显著增加轨迹多样性；随机动作初始化本身可能已经足够产生多样轨迹。

- 实验结果表明，这个假设基本成立。

### 2.3 方法二：静态 KV cache 管理

#### 原始动态 KV cache 的问题

- Alpamayo 1 的 action generation 中，action decoder 有 36 个 transformer blocks。
- 每个 block 会引用 reasoning 阶段对应 block 的 KV cache，并生成新的 KV 条目。
- 原始实现采用动态 KV cache 管理：
  - 每个 block 都会将已有 KV cache 与新生成 KV tensor 拼接；
  - 通过类似 `torch.cat` 的方式分配新 buffer；
  - 旧 buffer 被丢弃；
  - 这个过程在多个 block 和多轮扩散迭代中重复发生。
- 这种方式适合 chatbot 等输出长度不确定的 LLM 场景，但在 Alpamayo 的动作生成中并不高效。

#### 静态 KV cache 的核心思想

- 论文观察到：
  - action generation 的迭代次数固定为 10；
  - action token 维度固定；
  - 每轮 diffusion 生成的 KV 增量大小固定；
  - 因此 action generation 期间所需 KV cache 最大容量可以预先估计。
- 例如在单轨迹、20 个 CoT token 情况下：
  - reasoning 传来的 KV cache 约为 454.3 MB；
  - action generation 过程中 KV cache 增加约 9.44 MB；
  - 最大约为 463.74 MB。
- 因此作者采用：
  - 预先分配最大 KV buffer；
  - 每个 transformer block 使用指针定位自己的 KV slot；
  - diffusion 迭代中通过 in-place 写入更新 KV；
  - 避免重复 copy、concat 和 reallocation。

### 2.4 方法三：CUDA Graph 减少 kernel launch 开销

- 静态 KV cache 使得：
  - 内存地址固定；
  - tensor shape 固定；
  - 每轮 diffusion 调用的 GPU kernels 序列固定。
- 因此可以使用 **CUDA Graph capture**：
  - 先记录一整段 kernel launch 序列；
  - 后续 diffusion iteration 直接 replay；
  - 避免 CPU 逐个发起 kernel launch 的开销。
- 这对于 action generation 中大量细粒度 kernel 尤其有效，因为：
  - 每个 kernel 执行时间可能很短；
  - CPU launch overhead 会导致 GPU idle gap；
  - CUDA Graph 可以减少这些空隙，提高 GPU 利用率。

## 6. 主要结论与发现

### 6.1 多推理带来显著冗余

- 原始 Alpamayo 1 的多推理架构中，随着轨迹数量增加：
  - vision encoding；
  - language prefill；
  - action generation；
  都会增加。
- 特别是 reasoning 中的 vision 和 prefill 延迟随轨迹数量近似线性上升。
- 这说明为每条轨迹重新生成 reasoning 存在明显重复计算。

---

### 6.2 单推理不会明显降低轨迹质量

论文关键实验结果如下：

| 架构 | Open-loop minADE6@6.4s | Closed-loop DTF |
|---|---:|---:|
| Multi-reasoning | 0.725 | 131.58 m |
| Single-reasoning | 0.743 | 137.12 m |

- open-loop 中，single-reasoning 的 minADE 略高，即误差略大，但差异很小；
- closed-loop 中，single-reasoning 的 DTF 反而略高；
- 说明单推理没有明显损害轨迹预测质量和驾驶稳定性。

---

### 6.3 随机动作初始化足以提供较多轨迹多样性

- 多推理的假设是：不同 reasoning 会带来更丰富轨迹。
- 但实验发现：
  - 多个 CoT 往往语义高度重叠；
  - 多推理生成的轨迹多样性并没有显著强于单推理；
  - action generation 中的随机初始化已经能产生足够多样的候选轨迹。

---

### 6.4 动作生成成为单推理架构下的新瓶颈

- 在 multi-reasoning 架构中，reasoning 占据大量延迟；
- 改为 single-reasoning 后，reasoning 延迟基本固定；
- 随着轨迹数增加，action generation 延迟占比快速上升。
- 例如：
  - 6 条轨迹时，single-reasoning 中 action generation 占比达到约 42.06%；
  - 11 条轨迹时，占比达到约 52.30%。

---

### 6.5 静态 KV cache 和 CUDA Graph 能显著降低动作生成延迟

- 单轨迹 action generation 中：
  - 静态 KV cache 带来 21.66% 延迟下降；
  - 静态 KV cache + CUDA Graph 总共带来 24.65% 延迟下降。
- 多轨迹时收益更大：
  - 1 条轨迹时 action generation 从 0.65 s 降到 0.47 s，下降 27.69%；
  - 6 条轨迹时从 2.09 s 降到 1.19 s，下降 43.06%。
- 原因是多轨迹 batch 生成会加重内存带宽压力，而静态 KV cache 减少了内存复制和分配。

---

### 6.6 整体延迟最高下降 69.23%

- 完整优化包括：
  - single-reasoning 架构；
  - 静态 KV cache；
  - CUDA Graph capture。
- 在生成 6 条轨迹时：
  - baseline Alpamayo 延迟为 13.33 s；
  - 优化后为 4.10 s；
  - 总体下降 69.23%。

---

## 7. 优点与亮点

### 7.1 研究问题实际且重要

- 论文聚焦真实 VLA 自动驾驶系统的推理延迟，而不是只讨论模型精度；
- 对自动驾驶部署具有实际意义；
- 特别适合实时系统、边缘计算和自动驾驶工程优化方向。

### 7.2 对 Alpamayo 运行时架构分析细致

- 作者基于官方开源代码分析了 3,462 行 Python 推理代码；
- 将系统拆成 preprocessing、reasoning-vision、reasoning-prefill、reasoning-decode、action-generation 等模块；
- 使延迟瓶颈定位较清晰。

### 7.3 架构优化思路简洁有效

- 从 multi-reasoning 到 single-reasoning 的改造不修改模型参数；
- 不需要重新训练；
- 只改变 inference pipeline；
- 工程成本相对较低，但收益很大。

### 7.4 挑战了“多推理必然带来更好多样性”的常见假设

- 论文通过实验说明：
  - 多 CoT 推理不一定带来显著不同的驾驶行为；
  - 轨迹多样性可能主要来自扩散式动作生成阶段；
  - 这对后续 VLA 自动驾驶架构设计有启发意义。

### 7.5 运行时优化具有通用参考价值

- 静态 KV cache 适用于输出形态固定、迭代次数固定的推理任务；
- CUDA Graph 适用于重复执行相同 kernel 序列的场景；
- 这些优化不仅对 Alpamayo 有用，也可能适用于其他 diffusion planner 或 VLA 推理系统。

### 7.6 评估维度较全面

- 同时覆盖：
  - 延迟；
  - 轨迹多样性；
  - open-loop accuracy；
  - closed-loop stability；
  - action generation 消融；
  - 端到端整体优化结果。

---

## 8. 不足与局限

### 8.1 轨迹多样性评估不够量化

- 论文主要通过可视化展示 multi-reasoning 和 single-reasoning 的轨迹相似性；
- 缺少明确的多样性指标，例如：
  - pairwise trajectory distance；
  - coverage；
  - entropy；
  - mode diversity；
  - collision-free diversity。
- 因此“多样性未显著下降”的结论虽然有说服力，但仍可进一步量化增强。

### 8.2 Closed-loop 实验细节不足

- 文中报告了 DTF，但对以下内容介绍有限：
  - closed-loop 场景数量；
  - 场景类型；
  - 难度分布；
  - 随机种子；
  - 失败案例分析。
- 因此闭环结论还需要更多实验支撑。

### 8.3 只验证了 Alpamayo 1

- 本文优化针对 Alpamayo 1 的运行时结构；
- 虽然作者称 Alpamayo 1.5 也可直接应用，但没有正式报告 Alpamayo 1.5 的优化实验；
- 也没有在 ORION、DiffVLA、IRL-VLA、ColaVLA 等其他 VLA 自动驾驶模型上验证。
- 因此泛化性仍需进一步确认。

### 8.4 对硬件平台依赖较强

- 实验主要在 NVIDIA DGX Spark / GB10 Grace Blackwell 平台上进行；
- CUDA Graph、内存带宽瓶颈、kernel launch overhead 的收益可能随硬件变化而变化；
- 缺少在车规计算平台或更低功耗边缘设备上的验证。

### 8.5 仍未达到严格实时自动驾驶要求

- 即使优化后，6 条轨迹端到端延迟仍约为 4.10 秒；
- 这对于高频闭环控制仍然太慢；
- 作者也承认 VLA 模型可能更多用于安全监管、辅助决策或与轻量模型并行，而非直接作为唯一实时控制器。

### 8.6 静态 KV cache 假设依赖固定推理形态

- 静态 KV cache 依赖：
  - 最大 CoT token 数可估计；
  - diffusion iteration 固定；
  - action sequence 长度固定；
  - tensor shape 稳定。
- 如果未来模型采用动态规划 horizon、自适应 diffusion step 或更长 reasoning token，静态分配策略需要重新设计。

### 8.7 未讨论优化对内存占用峰值的影响

- 静态 KV cache 会预分配最大 buffer；
- 这减少了动态分配和复制，但可能增加某些情况下的峰值保留内存；
- 论文主要关注延迟，没有系统报告内存占用变化。

---

## 总结

本文对 Alpamayo 1 这一推理式端到端自动驾驶 VLA 模型进行了系统级延迟分析，指出原始多推理架构和扩散式动作生成中的动态 KV cache / kernel launch 开销是主要性能瓶颈。作者将其改造为单推理共享架构，并通过静态 KV cache 和 CUDA Graph 优化动作生成，在不修改模型参数、不明显损害轨迹多样性和预测性能的情况下，将 6 轨迹生成的端到端推理延迟从 13.33 秒降低到 4.10 秒，降幅达 69.23%。论文的主要贡献在于证明了推理式自动驾驶模型的效率不仅取决于模型结构，也高度依赖推理流水线、内存管理和 GPU 执行机制的协同优化。
