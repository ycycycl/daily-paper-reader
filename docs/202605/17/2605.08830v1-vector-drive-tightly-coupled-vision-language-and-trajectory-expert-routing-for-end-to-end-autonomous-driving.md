## 摘要
端到端自动驾驶要求模型理解交通场景、推断驾驶意图，并生成可执行的运动规划。近年来的视觉-语言-动作（VLA）模型继承了大规模视觉-语言预训练中的语义先验，但仍面临一种耦合权衡：完全共享的骨干网络能够保留多模态交互，却可能使语言推理与轨迹预测相互纠缠；而解耦的推理-动作流水线可减少任务冲突，却削弱了语义与运动之间的耦合。我们提出 VECTOR-DRIVE，一种基于 Qwen2.5-VL-3B 构建的紧耦合 VLA 框架。VECTOR-DRIVE 通过共享自注意力保持所有 token 的耦合，并根据 token 语义对前馈计算进行路由。视觉和语言 token 由视觉-语言专家处理，以保留语义先验；而目标点、自车状态和噪声动作 token 则被路由至轨迹专家，以进行运动特定的计算。在动作 token 路径上，基于流匹配的规划器将噪声动作 token 细化为未来路径点和速度曲线。该设计在单一多模态 Transformer 内耦合语义推理与运动规划，同时分离任务特定的 FFN 计算。在 Bench2Drive 上，VECTOR-DRIVE 取得了 88.91 的 Driving Score，并优于具有代表性的端到端和基于 VLA 的基线方法。定性结果和消融实验进一步验证了共享注意力、语义感知专家路由、渐进式训练以及基于流的动作解码所带来的优势。

## Abstract
End-to-end autonomous driving requires models to understand traffic scenes, infer driving intent, and generate executable motion plans. Recent vision-language-action (VLA) models inherit semantic priors from large-scale vision-language pretraining, yet still face a coupling trade-off: fully shared backbones preserve multimodal interaction but may entangle language reasoning and trajectory prediction, whereas decou pled reasoning-action pipelines reduce task conflict but weaken semantic-motion coupling. We propose VECTOR-DRIVE, a tightly coupled VLA framework built on Qwen2.5-VL-3B. VECTOR-DRIVE keeps all tokens coupled through shared self attention and routes feed-forward computation according to token semantics. Vision and language tokens are processed by a Vision-Language Expert to preserve semantic priors, while target-point, ego-state, and noisy action tokens are routed to a Trajectory Expert for motion-specific computation. On the action-token pathway, a flow-matching planner refines noisy action tokens into future waypoints and speed profiles. This design couples semantic reasoning and motion planning within a single multimodal Transformer while separating task-specific FFN computation. On Bench2Drive, VECTOR-DRIVE achieves 88.91 Driving Score and outperforms representative end-to end and VLA-based baselines. Qualitative results and ablations further validate the benefits of shared attention, semantic-aware expert routing, progressive training, and flow-based action de coding.

## 2. 方法论：核心思想、关键技术细节与算法流程

### 2.1 总体框架

- VECTOR-Drive 基于 **Qwen2.5-VL-3B** 构建。
- 输入包括：
  - 前视 RGB 图像；
  - 全局目标 prompt；
  - 高层导航命令；
  - 目标点；
  - 自车状态；
  - 噪声动作 token。
- 输出包括：
  - 语言侧输出：Chain-of-Thought 推理和驾驶指令；
  - 运动侧输出：未来路径点和速度曲线。

形式上，模型学习条件映射：

\[
(O, Y_p, Y_v) = F(I, G, P, U, S)
\]

其中：
- \(I\)：前视图像；
- \(G\)：全局目标；
- \(P\)：目标点；
- \(U\)：高层命令；
- \(S\)：自车状态；
- \(O\)：语言推理与驾驶指令；
- \(Y_p\)：未来路径点；
- \(Y_v\)：未来速度曲线。

### 2.2 统一多模态 Token 化

模型将不同模态统一映射到 Transformer token 空间：

- 图像输入被编码为视觉 token：\(Z_I\)
- 全局目标和高层命令被编码为语言 / 导航 token：\(Z_G, Z_U\)
- 目标点和自车状态通过 MLP 投影为 token：\(Z_P, Z_S\)
- 噪声动作状态和时间嵌入被编码为动作 token：\(Z_A(\tau)\)

最终形成交错排列的多模态 token 序列：

\[
Z = Interleave(Z_G, Z_I, Z_P, Z_U, Z_S, Z_A)
\]

这种设计使语义信息、导航信息、自车状态和动作生成过程在同一 Transformer 序列中交互。

### 2.3 共享自注意力 + 语义感知专家路由

VECTOR-Drive 的核心结构是：

- **所有 token 共用自注意力层**；
- **不同类型 token 使用不同 FFN 专家**。

具体来说，每层 Transformer 先对所有 token 做共享 self-attention：

\[
\tilde{H}^{(l)} = H^{(l-1)} + Attn^{(l)}(LN(H^{(l-1)}))
\]

之后根据 token 类型进行路由：

- 视觉、语言、导航命令 token 被路由到 **Vision-Language Expert**；
- 目标点、自车状态、噪声动作 token 被路由到 **Trajectory Expert**。

两个专家均采用 gated FFN 结构：

\[
FFN_e(X) = [\sigma(XW_{e,gate}) \odot (XW_{e,up})] W_{e,down}
\]

其中：
- \(e=0\)：视觉语言专家；
- \(e=1\)：轨迹专家；
- \(\sigma\)：SiLU 激活函数；
- \(\odot\)：逐元素乘法。

专家处理后的 token 再合并回原始交错顺序。

### 2.4 注意力掩码设计

为了避免动作 token 信息泄漏到语言生成侧，论文设计了因果多模态 attention mask：

- 动作 token 可以关注前面的视觉、语言、导航和自车状态 token；
- 语言侧 token 不能访问未来动作状态；
- 噪声动作 token 内部允许双向注意力，以提升 waypoint 之间的一致性。

这使得运动规划可以利用语义上下文，但语言推理不会被未来动作信息污染。

### 2.5 语言生成分支

语言分支使用 LM Head，从最终视觉语言 hidden states 中自回归生成：

\[
p(O | H^{(L)}_{vl}) = \prod_{k=1}^{K} p(o_k | o_{<k}, H^{(L)}_{vl})
\]

语言输出包括：
- 场景理解；
- Chain-of-Thought 推理；
- 简洁驾驶指令。

虽然语言质量不是主要评测指标，但它作为语义先验帮助轨迹规划。

### 2.6 基于流匹配的动作规划器

论文使用 **flow matching** 生成未来路径和速度。

#### 路径生成

设真实路径为 \(Y_p\)，高斯噪声为 \(\epsilon\)，插值时间为 \(\tau \in [0,1]\)，则噪声路径状态为：

\[
X_\tau = (1-\tau)\epsilon + \tau Y_p
\]

目标向量场为：

\[
v^\star = Y_p - \epsilon
\]

模型学习预测向量场：

\[
\hat{v}_\omega(X_\tau, \tau | C_{fm}^{(L)})
\]

推理时从高斯噪声初始化：

\[
X_0 \sim \mathcal{N}(0, I)
\]

然后通过 ODE：

\[
\frac{dX_\tau}{d\tau} = \hat{v}_\omega(X_\tau, \tau | C_{fm}^{(L)})
\]

逐步积分得到最终轨迹。

论文中未来路径为：
- **20 米规划范围**；
- **每米一个 waypoint**；
- 共 **20 个路径点**。

#### 速度生成

速度曲线同样采用流匹配方式生成：

- 输出 10 个未来速度点；
- 与路径分支共享语义条件和显式驾驶条件；
- 使用独立的速度 Flow Head。

### 2.7 多模态条件构造

流匹配规划器的条件包括两类：

- **隐式语义条件**：由最终视觉语言 hidden states 投影得到；
- **显式驾驶条件**：由自车状态、导航信息和目标点投影得到。

最终条件为：

\[
C_{fm}^{(L)} = Concat(C_{imp}^{(L)}, C_{exp})
\]

### 2.8 三阶段训练策略

论文采用渐进式训练以稳定语言推理和轨迹生成：

#### Stage I：视觉语言学习

- 冻结轨迹专家和运动头；
- 训练视觉语言专家和 LM Head；
- 目标是学习场景理解、语义推理和驾驶指令生成。

损失：

\[
L_{lang} = -\sum_{k=1}^{K}\log p(o_k | o_{<k}, H^{(L)}_{vl})
\]

#### Stage II：驾驶技能学习

- 冻结视觉语言专家；
- 训练轨迹专家、flow heads 和条件投影模块；
- 目标是学习路径和速度生成。

路径流匹配损失：

\[
L_{path} = E[\|f^p_\omega(F^p_\tau) - (Y_p - \epsilon_p)\|_2^2]
\]

速度流匹配损失：

\[
L_{speed} = E[\|f^v_\eta(F^v_\tau) - (Y_v - \epsilon_v)\|_2^2]
\]

还加入二阶差分平滑正则：

\[
L_{smooth} = \frac{1}{18} \sum_{k=2}^{19} \|\hat{p}_{k+1} - 2\hat{p}_k + \hat{p}_{k-1}\|_2^2
\]

综合驾驶损失：

\[
L_{drive} = \lambda_{path}L_{path} + \lambda_{smooth}L_{smooth} + \lambda_{speed}L_{speed}
\]

#### Stage III：联合全参数微调

- 解冻所有专家和预测头；
- 端到端联合优化：

\[
L = L_{lang} + L_{drive}
\]

  - 使用闭环 Bench2Drive，而不是仅使用 open-loop 指标；
  - 与大量端到端和 VLA 方法对比；
  - 包含架构、训练策略和解码器三类消融；
  - 同时报告 DS、SR、Efficiency、Comfort 和 Multi-Ability，评价维度较全面。

- **不足之处**：
  - 只在 CARLA / Bench2Drive 上验证，缺少真实道路数据或其他仿真 benchmark；
  - 训练数据由 PDM-lite 规则专家生成，可能存在专家策略偏差；
  - 对比方法是否全部在完全一致训练数据、模型规模和推理设置下运行，论文没有充分展开；
  - 缺少统计显著性分析，例如多随机种子均值和方差；
  - 缺少推理速度、计算开销和部署可行性分析；
  - 语言生成质量主要做可视化，没有独立量化评估。

总体而言，实验对于证明方法在 Bench2Drive 上有效是较充分的，但对于泛化性、真实场景适用性和计算效率的验证仍有限。

---

## 6. 主要结论与发现

- VECTOR-Drive 在 Bench2Drive 上取得当前对比方法中的最佳或接近最佳闭环表现：
  - Driving Score：88.91；
  - Success Rate：71.82%；
  - Multi-Ability Mean：72.03%。

- 与完全共享或完全解耦的 VLA 架构相比，**共享 self-attention + 专家 FFN 路由** 是更有效的折中方案。

- 语义信息不应只作为粗粒度条件输入给动作模块，而应在 token 级别与运动生成过程交互。

- 视觉语言专家和轨迹专家的分离有助于减少语言推理与连续轨迹预测之间的任务干扰。

- 三阶段训练比直接端到端联合训练更稳定，其中视觉语言预适应对后续规划能力尤其重要。

- flow matching 相比直接回归和 DDPM，在该任务中能更好地生成平滑、可执行的未来轨迹与速度曲线。

## 7. 优点与亮点

### 方法设计亮点

- **紧耦合但不完全共享**  
  通过共享自注意力实现跨模态交互，通过专家 FFN 实现任务分离，结构设计较清晰。

- **静态语义感知路由简单有效**  
  不使用复杂 learned router，避免额外路由参数和负载均衡损失，同时性能接近 learned routing。

- **同时生成语言推理和运动轨迹**  
  模型不仅输出轨迹，还能生成 Chain-of-Thought 和驾驶指令，有助于增强可解释性。

- **flow matching 用于连续轨迹规划**  
  相比单步回归，流匹配更适合从噪声中逐步生成连续动作分布。

- **路径和速度联合建模**  
  不仅预测空间路径，还生成时间速度曲线，更接近实际控制需求。

### 实验设计亮点

- 使用闭环 Bench2Drive 评估，比纯 open-loop 轨迹误差更能反映真实驾驶表现。
- 对比方法覆盖传统端到端模型、规划导向模型和 VLA 模型。
- 消融实验较有针对性，分别验证：
  - 架构耦合；
  - 专家路由；
  - 训练阶段；
  - 解码器选择。
- Multi-Ability 指标展示了模型在合流、超车、刹车、让行和交通标志遵守等细粒度能力上的提升。

## 8. 不足与局限

### 8.1 数据与泛化局限

- 实验仅在 CARLA 的 Bench2Drive 中进行，缺少真实世界数据验证。
- 训练数据来自 PDM-lite 规则专家，模型可能学习到该专家的行为偏好和局限。
- 未展示跨城市、跨传感器、跨天气分布或 sim-to-real 泛化能力。

### 8.2 对比公平性不够透明

- 虽然对比方法很多，但论文没有详细说明：
  - 所有 baselines 是否使用相同训练数据；
  - 是否统一模型规模；
  - 是否统一传感器输入；
  - 是否统一推理预算。
- 因此，部分性能差异可能同时受到数据、模型规模和训练设置影响。

### 8.3 计算成本与部署问题

- 基于 Qwen2.5-VL-3B，模型规模较大。
- 推理时 flow matching 需要 10 次 Euler ODE step，每一步可能都要经过 Transformer 和 Flow Head，实时性压力较大。
- 论文没有报告：
  - FPS；
  - 单帧延迟；
  - 显存占用；
  - 与其他方法的推理开销比较。
- 因此，实际车载部署可行性仍不明确。

### 8.4 安全性与鲁棒性验证不足

- 没有系统评估极端长尾场景，例如：
  - 突然横穿行人；
  - 遮挡车辆；
  - 交通参与者异常行为；
  - 传感器失效；
  - 对抗性语言或视觉干扰。
- 缺少安全约束或可验证安全机制，模型生成轨迹是否始终满足动力学和交通规则约束仍需进一步验证。

### 8.5 语言推理评估有限

- 论文展示了 CoT 和驾驶指令可视化，但没有量化语言质量。
- CoT 是否真正参与因果决策，还是仅作为辅助监督信号，仍有进一步分析空间。
- 没有评估语言输出错误时对轨迹规划的影响。

### 8.6 架构复杂度较高

- 模型包含：
  - 多模态 Transformer；
  - 双专家 FFN；
  - 语言头；
  - 路径 Flow Head；
  - 速度 Flow Head；
  - 多阶段训练流程。
- 虽然性能提升明显，但工程复杂度、调参成本和复现实验成本较高。
