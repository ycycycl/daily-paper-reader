## 摘要
基于推理的端到端（E2E）自动驾驶近来成为一种有前景的方法，可通过在预测轨迹的同时生成可供人类阅读的推理过程来提升驾驶决策的可解释性。此类方法通常生成多条轨迹以捕捉多样化的未来行为，并可分为两类：（1）多推理，即每条轨迹生成一个推理序列；（2）单推理，即所有轨迹共享同一个推理。前者以冗余计算为代价提供更丰富的多样性，而后者效率更高，但通常被认为会牺牲多样性。Alpamayo 1 是一个具有代表性的系统，采用多推理方法并实现了具有竞争力的轨迹预测性能。然而，该设计的效率在很大程度上尚未得到探索，因此成为一个具有充分研究动机的课题。本文从两个方面对 Alpamayo 1 进行了系统分析与改进。首先，我们将 Alpamayo 1 重新设计为单推理系统，在保持轨迹多样性的同时降低推理延迟。通过大量实验，我们发现以单推理替代多推理并不会显著降低轨迹多样性。其次，我们通过消除不必要的复制操作和低效内核执行所带来的块间开销，加速基于扩散的动作生成。通过闭环和开环实验，我们验证了这两项优化，结果表明在保持轨迹多样性和预测质量的同时，推理延迟降低了 69.23%。这些结果凸显了联合分析系统架构与运行时执行对于提升基于推理的端到端自动驾驶系统效率的重要性。

## Abstract
Reasoning-based end-to-end (E2E) autonomous driving has recently emerged as a promising approach to improving the interpretability of driving decisions as it can generate human-readable reasoning together with predicted trajectories. Such approaches commonly generate multiple trajectories to capture diverse future behaviors, and they fall into two categories: (1) multi-reasoning, where one reasoning sequence is generated per trajectory, and (2) single-reasoning, where a single reasoning is shared across all trajectories. The former offers richer diversity at the cost of redundant computation, while the latter is more efficient but is often assumed to sacrifice diversity. Alpamayo 1, a representative system, adopts the multi-reasoning approach and achieves competitive trajectory prediction performance. However, the efficiency of this design remains largely unexplored, making it a well-motivated subject for investigation. In this paper, we systematically analyze and improve Alpamayo 1 in two ways. First, we reduce inference latency while preserving trajectory diversity by redesigning Alpamayo 1 into a single-reasoning system. Through extensive experiments, we find that replacing multi-reasoning with single-reasoning does not meaningfully degrade trajectory diversity. Second, we accelerate diffusion-based action generation by eliminating inter-block overhead arising from unnecessary copy operations and inefficient kernel execution. Through closed-loop and open-loop experiments, we validate both optimizations, demonstrating a 69.23% reduction in inference latency while maintaining trajectory diversity and prediction quality. These results highlight the importance of jointly analyzing system architecture and runtime execution to improve the efficiency of reasoning-based E2E AD systems.

## 2. 方法论：核心思想、关键技术细节与算法流程

### 2.1 Alpamayo 1 原始推理流程分析

论文首先对 Alpamayo 1 的官方开源推理代码进行系统拆解。其推理流程包括四个阶段：

1. **预处理（Preprocessing）**
   - 输入包括：
     - 多摄像头图像；
     - 文本 prompt；
     - 历史自车轨迹。
   - 图像被切分为视觉 token；
   - 文本由文本 tokenizer 编码；
   - 历史轨迹被离散化并转成轨迹 token。

2. **推理阶段（Reasoning）**
   - 使用视觉编码器处理图像；
   - 使用语言解码器生成 CoT 推理 token；
   - 语言解码器由 36 个 Transformer block 构成；
   - 推理阶段包括：
     - **prefill**：并行处理输入上下文，构建 KV cache；
     - **decode**：自回归生成 CoT token。

3. **动作生成阶段（Action Generation）**
   - 由动作编码器和动作解码器组成；
   - 动作解码器同样是 Transformer 结构；
   - 通过扩散式迭代生成动作序列；
   - 默认执行 10 次 denoising / refinement 迭代；
   - 输出为未来若干时刻的加速度和曲率。

4. **后处理（Postprocessing）**
   - 将 CoT token 解码成人类可读文本；
   - 将动作序列通过自行车 / 单轮车动力学模型转换为未来轨迹。

### 2.2 优化一：从多推理架构改为单推理架构

#### 原始多推理架构

- 原始 Alpamayo 生成 N 条轨迹时，会复制输入 N 次；
- 每个复制样本分别经过推理模块；
- 因此会生成：
  - N 段不同 CoT 推理；
  - N 份不同 KV cache；
  - N 条动作序列 / 轨迹。

其特点是：

- 优点：
  - 每条轨迹拥有独立推理；
  - 理论上可能带来更高轨迹多样性。
- 缺点：
  - 推理模块重复执行 N 次；
  - 视觉编码、prefill 和 decode 的计算成本随轨迹数量增长；
  - 推理延迟明显增加。

#### 本文提出的单推理架构

本文将 Alpamayo 改造成 **single-reasoning multi-trajectory** 架构：

1. 输入只经过一次视觉编码和语言推理；
2. 生成一段 CoT 推理；
3. 得到一份 KV cache；
4. 将该 KV cache 沿 batch 维度复制；
5. 动作生成模块对 N 个随机初始化的动作序列并行执行扩散生成；
6. 输出 N 条轨迹。

核心思想是：

> 轨迹多样性不一定必须来自多段语言推理，也可以主要来自动作生成阶段的随机初始化。

也就是说，即使所有轨迹共享同一段 CoT 推理，扩散动作生成中的随机初始动作序列仍可产生多样轨迹。

### 2.3 优化二：静态 KV cache 管理

#### 原始问题

在 Alpamayo 的动作生成阶段，每个 Transformer block 都需要使用推理阶段产生的 KV cache，并生成新的 KV 项。

原始实现使用动态 KV cache 管理方式：

- 每次迭代中，将已有 KV cache 与新 KV tensor 通过类似 `torch.cat` 的方式拼接；
- 需要反复分配新内存；
- 需要反复复制已有 KV；
- 导致大量内存 copy 和 reallocation 开销。

这种设计适合聊天机器人等输出长度不确定的 LLM 场景，但并不适合自动驾驶中的固定周期推理任务。

#### 本文优化方式

作者观察到：

- 自动驾驶推理任务通常是静态周期任务；
- 动作生成阶段的迭代次数固定；
- 每次扩散生成的 KV 增量大小固定；
- 因此最大 KV cache 大小可通过离线 profiling 预估。

基于此，本文提出 **静态 KV cache 管理**：

1. 在动作生成前预分配最大所需 KV buffer；
2. 每个 Transformer block 对应固定的 KV slot；
3. 每次迭代时通过指针定位写入位置；
4. 新 KV 直接原地写入预分配 buffer；
5. 避免动态拼接、重新分配和全量复制。

文中给出的具体数值为：

- 单轨迹、假设 20 个 CoT token 时：
  - 推理阶段产生的 KV cache 约为 **454.3 MB**；
  - 动作生成阶段额外增加约 **9.44 MB**；
  - 最大 KV cache 约为 **463.74 MB**。

### 2.4 优化三：CUDA Graph 减少 GPU kernel launch 开销

#### 原始问题

动作生成阶段由 10 次扩散迭代组成，每次迭代包含多个 Transformer block 和大量细粒度 GPU kernel。

原始执行方式中：

- CPU 逐个发起 GPU kernel；
- 每个 kernel launch 都有调度开销；
- 当 kernel 执行时间较短时，launch overhead 可能导致 GPU 空闲；
- 多次迭代下该开销累积明显。

#### 本文优化方式

在引入静态 KV cache 后：

- KV buffer 地址固定；
- tensor shape 固定；
- 每轮 diffusion 的计算图和 kernel 调用序列固定。

因此作者使用 **CUDA Graph Capture**：

1. 在前几轮执行中完成 warmup；
2. 捕获一整段 GPU kernel launch 序列；
3. 后续迭代直接 replay CUDA graph；
4. 避免逐 kernel 发起调用；
5. 减少 CPU-GPU 调度间隙，提高 GPU 利用率。

论文中实现细节是：

- 第二次 diffusion iteration 进行 graph capture；
- 第三次及之后使用 graph replay；
- 虽然 capture 本身有一次额外开销，但后续迭代可显著降低平均延迟。

- 端到端延迟收益。

这些实验基本能够支撑论文的主要结论。

但也存在一些不足：

- 轨迹多样性主要依赖可视化展示，缺少更系统的定量多样性指标；
- 开放环只报告 minADE6@6.4s，缺少 minFDE、collision rate、off-road rate 等更多自动驾驶常用指标；
- 闭环 DTF 是作者自定义指标，与 Alpamayo 原论文可能不完全一致；
- 实验主要基于 Alpamayo 1，虽然作者声称可迁移到 Alpamayo 1.5，但没有提供充分实测结果；
- 硬件平台较强，缺少在车规芯片或边缘 GPU 上的验证。

因此，实验对于证明“优化有效”是较充分的，但对于证明“可直接部署到真实自动驾驶系统”还不够充分。

---

## 6. 主要结论与发现

论文的主要发现包括：

- **Alpamayo 1 原始多推理架构存在明显冗余计算**：
  - 多条轨迹需要多次 reasoning；
  - 推理延迟随轨迹数量近似线性增长。

- **单推理不一定显著降低轨迹多样性**：
  - 实验显示，单推理共享 KV cache 后，动作生成阶段的随机初始化仍能产生多样轨迹；
  - multi-reasoning 产生的多段 CoT 往往语义相近，未必带来实质行为差异。

- **预测性能基本保持不变**：
  - 开放环：
    - multi-reasoning：minADE6@6.4s = 0.725；
    - single-reasoning：minADE6@6.4s = 0.743。
  - 二者差距较小。
  - 闭环：
    - multi-reasoning：DTF = 131.58 m；
    - single-reasoning：DTF = 137.12 m。
  - single-reasoning 甚至略好。

- **动作生成成为单推理架构下的重要瓶颈**：
  - 因为 reasoning 被压缩为一次后，Action-Gen 在总延迟中占比上升；
  - 轨迹数越多，动作生成占比越高。

- **静态 KV cache 和 CUDA graph 能显著降低动作生成延迟**：
  - 单轨迹动作生成中：
    - 静态 KV cache 降低约 21.66%；
    - 加上 CUDA graph 后总降低约 24.65%。
  - 多轨迹情况下优化收益更大：
    - 6 条轨迹时 Action-Gen 延迟从 2.09 s 降至 1.19 s；
    - 降幅约 43.06%。

- **端到端延迟最多降低 69.23%**：
  - 在 6 条轨迹设置下：
    - 原始 Alpamayo：13.33 s；
    - 优化后：4.10 s。

## 7. 优点与亮点

### 7.1 问题定位清晰

- 论文不是单纯提出一个新模型，而是对已有代表性系统 Alpamayo 1 做深入系统分析；
- 关注实际推理延迟，具有明显部署价值。

### 7.2 同时覆盖架构和系统层优化

论文的优化不是单一层面的，而是结合：

- 模型推理架构：
  - multi-reasoning 改为 single-reasoning；
- 运行时内存管理：
  - dynamic KV cache 改为 static KV cache；
- GPU 执行机制：
  - 使用 CUDA graph 降低 kernel launch overhead。

这种联合优化较有系统工程价值。

### 7.3 保留可解释性

相比一些 single-reasoning 方法将推理压缩为 latent planning token，本文保留了 Alpamayo 的 CoT reasoning，并让完整 reasoning KV cache 直接参与动作生成。

这使得：

- 模型仍能输出人类可读推理；
- 优化没有明显破坏解释链条；
- 对安全验证更友好。

### 7.4 消融实验清楚

论文分别评估了：

- 架构替换的效果；
- 静态 KV cache 的效果；
- CUDA graph 的增益；
- 不同轨迹数量下的 scaling。

这使得每个优化模块的贡献较容易理解。

### 7.5 结论有一定反直觉价值

论文挑战了一个常见假设：

> 多轨迹预测一定需要多段独立推理才能保持多样性。

实验显示，在 Alpamayo 中，动作扩散阶段的随机性已经足够提供较多轨迹多样性。

---

## 8. 不足与局限

### 8.1 轨迹多样性的量化不足

- 论文主要通过可视化展示 single-reasoning 与 multi-reasoning 的轨迹相似性；
- 缺少定量多样性指标，例如：
  - pairwise trajectory distance；
  - endpoint diversity；
  - behavioral mode coverage；
  - entropy-based diversity metric。

因此，“多样性未下降”的结论虽有实验支持，但仍可进一步加强。

### 8.2 闭环评估指标自定义，横向可比性有限

- DTF 是作者自行定义的指标；
- Alpamayo 原文中的闭环评估细节不完全一致；
- 因此与其他自动驾驶模型的闭环表现不易直接比较。

### 8.3 场景覆盖可能有限

- 开放环使用 394 个 20 秒场景；
- 对 latency 和系统优化来说足够有参考价值；
- 但对自动驾驶安全性而言，长尾场景、极端交互场景、恶劣天气、复杂路口等覆盖仍不明确。

### 8.4 硬件泛化性不足

- 实验在 NVIDIA DGX Spark / GB10 Grace Blackwell 平台上完成；
- 未验证在实际车载部署平台上的效果，例如：
  - NVIDIA Orin；
  - Thor；
  - 边缘 GPU；
  - 低功耗自动驾驶计算平台。
- CUDA graph 和静态 KV cache 在不同硬件、不同 batch size、不同运行时环境下的收益可能不同。

### 8.5 仍未达到严格实时自动驾驶要求

- 即使优化后，6 条轨迹端到端延迟仍约为 4.10 秒；
- 对高频闭环控制而言仍然较高；
- 论文也承认 VLA 模型更可能作为辅助、安全监督或高层规划模块，而非直接替代低延迟控制模型。

### 8.6 对推理内容一致性的分析不足

- single-reasoning 共享一段 CoT，但多条轨迹可能存在不同驾驶行为；
- 若多条轨迹行为差异较大，同一段 reasoning 是否能充分解释所有轨迹，仍值得讨论；
- 论文主要关注轨迹预测质量，没有深入分析 reasoning 与每条轨迹之间的语义一致性。

### 8.7 对 Alpamayo 1.5 仅声称可迁移，缺少实证

- 作者提到 Alpamayo 1.5 与 Alpamayo 1 共享推理架构，因此优化可直接应用；
- 但文中没有系统报告 Alpamayo 1.5 上的延迟和性能实验。

---

## 总结

本文对 Alpamayo 1 的推理延迟进行了系统级剖析，并提出了三项关键优化：将多推理改为单推理、使用静态 KV cache、通过 CUDA graph 降低 kernel launch 开销。实验表明，在基本保持轨迹多样性、开放环预测精度和闭环稳定性的情况下，端到端推理延迟最多降低 69.23%。论文的重要价值在于证明：对于推理式 E2E 自动驾驶系统，提升效率不仅需要模型设计优化，也需要深入理解推理时的内存管理和 GPU 执行机制。
