## 摘要
基于推理的端到端（E2E）自动驾驶近年来成为提升驾驶决策可解释性的一种有前景的方法，因为它能够在预测轨迹的同时生成可供人类阅读的推理过程。此类方法通常生成多条轨迹以刻画多样化的未来行为，并可分为两类：（1）多推理，即为每条轨迹生成一个推理序列；（2）单推理，即所有轨迹共享同一个推理过程。前者以冗余计算为代价提供更丰富的多样性，而后者效率更高，但通常被认为会牺牲多样性。Alpamayo 1 是一个具有代表性的系统，采用多推理方法并取得了具有竞争力的轨迹预测性能。然而，该设计的效率在很大程度上仍未得到探索，因此是一个具有充分研究动机的课题。本文从两个方面对 Alpamayo 1 进行了系统分析与改进。首先，我们将 Alpamayo 1 重新设计为单推理系统，在保持轨迹多样性的同时降低推理延迟。通过大量实验，我们发现用单推理替代多推理并不会显著降低轨迹多样性。其次，我们通过消除不必要的复制操作和低效内核执行所导致的块间开销，加速基于扩散的动作生成。通过闭环和开环实验，我们验证了这两项优化，证明在保持轨迹多样性和预测质量的同时，推理延迟降低了 69.23%。这些结果凸显了联合分析系统架构与运行时执行对于提升基于推理的端到端自动驾驶系统效率的重要性。

## Abstract
Reasoning-based end-to-end (E2E) autonomous driving has recently emerged as a promising approach to improving the interpretability of driving decisions as it can generate human-readable reasoning together with predicted trajectories. Such approaches commonly generate multiple trajectories to capture diverse future behaviors, and they fall into two categories: (1) multi-reasoning, where one reasoning sequence is generated per trajectory, and (2) single-reasoning, where a single reasoning is shared across all trajectories. The former offers richer diversity at the cost of redundant computation, while the latter is more efficient but is often assumed to sacrifice diversity. Alpamayo 1, a representative system, adopts the multi-reasoning approach and achieves competitive trajectory prediction performance. However, the efficiency of this design remains largely unexplored, making it a well-motivated subject for investigation. In this paper, we systematically analyze and improve Alpamayo 1 in two ways. First, we reduce inference latency while preserving trajectory diversity by redesigning Alpamayo 1 into a single-reasoning system. Through extensive experiments, we find that replacing multi-reasoning with single-reasoning does not meaningfully degrade trajectory diversity. Second, we accelerate diffusion-based action generation by eliminating inter-block overhead arising from unnecessary copy operations and inefficient kernel execution. Through closed-loop and open-loop experiments, we validate both optimizations, demonstrating a 69.23% reduction in inference latency while maintaining trajectory diversity and prediction quality. These results highlight the importance of jointly analyzing system architecture and runtime execution to improve the efficiency of reasoning-based E2E AD systems.

## 2. 方法论：核心思想、关键技术与流程

### 2.1 Alpamayo 1 的原始推理架构

Alpamayo 1 是一个视觉语言动作模型（VLA），核心包含三个 Transformer 组件：

- **Vision Encoder**
  - 基于 Qwen3-VL 的视觉编码器；
  - 将多摄像头图像切分为视觉 token；
  - 输出视觉 embedding。

- **Language Decoder**
  - 用于生成 CoT reasoning；
  - 自回归地产生文本推理 token；
  - 在生成过程中构建 KV cache；
  - 该 KV cache 作为后续动作生成的重要上下文。

- **Action Decoder**
  - 基于扩散式迭代生成动作序列；
  - 输出加速度和曲率；
  - 动作序列再转换为未来车辆轨迹。

完整推理流程包括四个阶段：

1. **预处理**
   - 图像 tokenization；
   - 文本 prompt tokenization；
   - 历史 ego 轨迹 tokenization。

2. **推理阶段**
   - 视觉编码；
   - 文本与轨迹 embedding；
   - Language decoder 生成 CoT token；
   - 累积 KV cache。

3. **动作生成阶段**
   - 从随机初始化的动作序列开始；
   - 经过 10 次扩散迭代逐步 refinement；
   - 每次迭代中 action decoder 使用 reasoning 阶段生成的 KV cache 作为条件。

4. **后处理**
   - 将 reasoning token 解码为文本；
   - 将动作序列转换为 6.4 秒未来轨迹。

### 2.2 优化一：从多推理架构改为单推理架构

#### 原始多推理架构

在原始 Alpamayo 中，如果要生成 `N` 条轨迹，会复制输入 token 形成 batch size 为 `N` 的输入：

- 每条轨迹生成一段独立 CoT；
- 每段 CoT 对应一份不同的 KV cache；
- 每条动作轨迹由对应 KV cache 条件化生成。

其多样性来源包括：

- CoT token sampling 的随机性；
- action sequence 随机初始化的随机性。

但问题是：

- Vision encoder、language prefill、language decode 都会随轨迹数量增加而重复计算；
- 推理延迟随轨迹数近似线性增加；
- 不适合需要多轨迹采样的实时或准实时系统。

#### 作者提出的单推理架构

作者将 Alpamayo 修改为：

- 只对输入执行一次 reasoning；
- 生成一个 CoT 序列；
- 得到一份 KV cache；
- 将该 KV cache 沿 batch 维度复制；
- 多个随机初始化的 action sequence 共享同一个 reasoning KV cache；
- action decoder 仍然批量生成多条不同轨迹。

也就是说，设计从：

```text
N 个输入副本 → N 个 reasoning → N 个 KV cache → N 条轨迹
```

改为：

```text
1 个输入 → 1 个 reasoning → 1 个 KV cache → 复制给 N 个动作生成分支 → N 条轨迹
```

#### 核心假设

论文挑战了一个常见假设：

> 多条不同 reasoning 是产生多样轨迹的必要条件。

作者认为，在 Alpamayo 中，轨迹多样性很大程度上已经来自扩散动作生成阶段的随机初始化，因此即使多条轨迹共享同一个 reasoning，也仍然能得到足够的轨迹多样性。

实验结果支持这一点：单推理并没有显著降低轨迹多样性和预测性能。

### 2.3 优化二：静态 KV cache 管理

#### 原始问题

在 action generation 阶段，Alpamayo 使用动态 KV cache 管理方式：

- 每个 transformer block 会把已有 KV cache 与新生成的 KV tensor 拼接；
- 使用类似 `torch.cat` 的方式重新分配和复制内存；
- 每次扩散迭代、每个 block 都可能发生不必要的拷贝和重新分配；
- 在 36 个 action transformer block 和 10 次 diffusion iteration 下，这种开销会被放大。

这种动态方式适合通用 LLM 聊天场景，因为输出长度不确定。但在自动驾驶中，动作生成过程更固定：

- 预测 horizon 固定；
- diffusion iteration 数固定；
- action token 数固定；
- KV cache 增长模式可提前分析。

#### 作者的静态 KV cache 方案

作者利用动作生成阶段可预测的内存使用模式：

- 预先 profiling 得到最大 KV cache 大小；
- 在 diffusion 开始前一次性分配足够大的 KV buffer；
- 后续每次迭代直接进行 in-place 写入；
- 用 KV pointer 追踪每个 transformer block 对应的写入位置；
- 避免重复内存分配和大规模拷贝。

文中提到，在单轨迹、20 个 CoT token 的假设下：

- reasoning 阶段传入的 KV cache 约为 **454.3 MB**；
- action generation 会额外增加约 **9.44 MB**；
- 最大 KV cache 大小约为 **463.74 MB**。

因此，作者可以提前分配静态缓存并复用。

### 2.4 优化三：CUDA Graph 捕获与重放

#### 原始问题

扩散式 action generation 包含大量小粒度 GPU kernel 调用：

- CPU 逐个 launch kernel；
- 每次 launch 都有调度开销；
- 当 kernel 执行时间较短时，launch overhead 会导致 GPU 空闲；
- 这种细粒度 launch 会降低 GPU 利用率。

#### CUDA Graph 优化

由于采用静态 KV cache 后：

- memory address 固定；
- tensor shape 固定；
- 每次 diffusion iteration 的计算图结构固定；
- kernel launch 序列也固定。

因此作者使用 CUDA Graph：

1. 在前几次迭代中进行 warmup；
2. 在第二次 diffusion iteration 捕获完整 kernel launch 序列；
3. 后续 iteration 直接 replay graph；
4. 减少逐 kernel launch 的 CPU 调度开销；
5. 提升 GPU 连续执行效率。

相对公平的地方：

- 不修改模型权重；
- 对比的是同一个 Alpamayo 模型的不同推理实现；
- open-loop 中同样生成 6 条轨迹；
- 延迟测量取 100 次平均；
- 既评估性能，也评估延迟。

需要注意的潜在问题：

- closed-loop 评估指标 DTF 是作者自定义的，而不是 Alpamayo 原论文中完全相同的官方指标；
- trajectory diversity 主要通过可视化和性能间接说明，缺少更系统的多样性量化指标；
- latency profiling 主要在单一硬件平台 DGX Spark 上完成，跨硬件泛化性尚不明确；
- 使用的 open-loop 场景数量为 394，对自动驾驶复杂长尾场景而言仍有限。

## 6. 主要结论与发现

- **多推理架构的主要瓶颈在于重复 reasoning 计算**  
  当生成多条轨迹时，原始 Alpamayo 会为每条轨迹复制输入并生成独立 reasoning，导致 vision encoding、language prefill 和 decode 等阶段的计算随轨迹数增加。

- **单推理并不会明显降低轨迹多样性**  
  实验发现，多条独立 CoT reasoning 往往并没有带来足够不同的驾驶行为；相反，扩散动作生成中的随机初始化已经能够提供较强轨迹多样性。

- **单推理预测性能与多推理接近**  
  Open-loop 中：
  - multi-reasoning：0.725；
  - single-reasoning：0.743。  
  差异很小。

  Closed-loop 中：
  - multi-reasoning：131.58 m；
  - single-reasoning：137.12 m。  
  单推理甚至略优。

- **action generation 在单推理架构下成为新的主要瓶颈**  
  由于 reasoning 部分被大幅压缩，随着轨迹数量增加，扩散动作生成在总延迟中的占比快速上升。

- **静态 KV cache 与 CUDA Graph 能有效降低动作生成延迟**
  单轨迹 action generation 中：
  - 静态 KV cache 带来约 21.66% 降低；
  - 加上 CUDA Graph 后总降低约 24.65%。

- **多轨迹场景下优化收益更大**
  当轨迹数为 6 时，action generation 延迟从 2.09 s 降至 1.19 s，降低约 43.06%。

- **整体延迟显著下降**
  在生成 6 条轨迹时：
  - baseline Alpamayo：13.33 s；
  - optimized Alpamayo：4.10 s；
  - 总体降低 **69.23%**。

## 7. 优点与亮点

- **问题选择有现实意义**
  - 推理式自动驾驶的可解释性很重要，但延迟是实际部署障碍；
  - 本文直接针对 VLA 自动驾驶系统的推理效率问题，工程价值较高。

- **分析粒度细**
  - 不只是报告总体延迟；
  - 将延迟拆解为 preprocessing、vision、prefill、decode、action generation；
  - 能清楚定位瓶颈来源。

- **方法简单但有效**
  - 单推理架构不需要重新训练；
  - 不改变模型参数；
  - 只调整推理 pipeline；
  - 工程落地成本较低。

- **挑战了多推理的常见假设**
  - 论文实验证明，多 reasoning 不一定带来显著更好的轨迹多样性；
  - 在 Alpamayo 中，动作扩散阶段的随机性已经足够重要。

- **运行时优化针对性强**
  - 静态 KV cache 利用了自动驾驶任务周期性、输入输出结构相对固定的特点；
  - CUDA Graph 利用了 diffusion iteration 结构重复的特点；
  - 两者结合合理。

- **评估覆盖 open-loop 和 closed-loop**
  - 不只看预测误差；
  - 也看闭环稳定性；
  - 更符合自动驾驶系统评估需求。

- **与现有模型兼容性较好**
  - 作者指出 Alpamayo 1.5 与 Alpamayo 1 共享类似推理架构，因此优化可直接迁移。

---

## 8. 不足与局限

- **没有重新训练或模型级优化**
  - 本文主要是推理系统优化；
  - 没有探索模型结构本身是否可以进一步压缩；
  - 也没有结合量化、剪枝、蒸馏等模型加速手段。

- **硬件平台较单一**
  - 主要在 NVIDIA DGX Spark / GB10 Grace Blackwell 上测试；
  - 没有在车规级芯片、边缘 GPU、低功耗平台上验证；
  - 实际车载部署延迟可能不同。

- **闭环指标为作者自定义**
  - DTF 指标虽然合理，但不是标准统一 benchmark；
  - 与其他自动驾驶论文的闭环结果不容易直接比较。

- **轨迹多样性评估不够量化**
  - 论文展示了可视化结果；
  - 但缺少系统性 diversity metric，例如轨迹间平均距离、多模态覆盖率、mode recall 等；
  - “不显著降低多样性”的结论仍有一定经验性。

- **场景覆盖有限**
  - Open-loop 使用 394 个场景；
  - 对自动驾驶长尾场景而言规模不大；
  - 对极端复杂场景、稀有交通事件、恶劣天气等覆盖不明确。

- **单推理可能存在语义一致性风险**
  - 如果同一个场景确实存在多种语义解释，例如“左转”与“直行”均合理，单一 CoT 可能限制高层意图多样性；
  - 本文结果说明 Alpamayo 当前设置下影响不大，但不代表所有 VLA 自动驾驶模型都适用。

- **静态 KV cache 依赖可预测执行模式**
  - 该优化适合固定 horizon、固定 diffusion step、固定 action token 数的自动驾驶任务；
  - 对输出长度高度动态的通用 LLM 应用不一定适用。

- **总体延迟仍然偏高**
  - 优化后 6 条轨迹仍约 4.10 秒；
  - 对严格实时控制而言仍不足；
  - 更适合作为高层规划、安全监督或低频 reasoning 模块，而非高频底层控制器。
