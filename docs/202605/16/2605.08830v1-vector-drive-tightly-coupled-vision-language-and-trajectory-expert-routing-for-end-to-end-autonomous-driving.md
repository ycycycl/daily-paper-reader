## 摘要
端到端自动驾驶要求模型能够理解交通场景、推断驾驶意图，并生成可执行的运动规划。近年来，视觉-语言-动作（VLA）模型继承了大规模视觉-语言预训练中的语义先验，但仍面临一种耦合权衡：完全共享的骨干网络能够保留多模态交互，却可能使语言推理与轨迹预测相互纠缠；而解耦的推理-动作流水线虽能减少任务冲突，却削弱了语义与运动之间的耦合。我们提出 VECTOR-DRIVE，这是一个基于 Qwen2.5-VL-3B 构建的紧耦合 VLA 框架。VECTOR-DRIVE 通过共享自注意力保持所有 token 的耦合，并根据 token 语义对前馈计算进行路由。视觉和语言 token 由视觉-语言专家处理，以保留语义先验；而目标点、自车状态和噪声动作 token 则被路由至轨迹专家，以进行面向运动的计算。在动作 token 通路上，基于流匹配的规划器将噪声动作 token 细化为未来航点和速度曲线。该设计在单一多模态 Transformer 内耦合语义推理与运动规划，同时分离任务特定的 FFN 计算。在 Bench2Drive 上，VECTOR-DRIVE 取得了 88.91 的 Driving Score，并优于具有代表性的端到端和基于 VLA 的基线方法。定性结果和消融实验进一步验证了共享注意力、语义感知专家路由、渐进式训练以及基于流的动作解码的优势。

## Abstract
End-to-end autonomous driving requires models to understand traffic scenes, infer driving intent, and generate executable motion plans. Recent vision-language-action (VLA) models inherit semantic priors from large-scale vision-language pretraining, yet still face a coupling trade-off: fully shared backbones preserve multimodal interaction but may entangle language reasoning and trajectory prediction, whereas decou pled reasoning-action pipelines reduce task conflict but weaken semantic-motion coupling. We propose VECTOR-DRIVE, a tightly coupled VLA framework built on Qwen2.5-VL-3B. VECTOR-DRIVE keeps all tokens coupled through shared self attention and routes feed-forward computation according to token semantics. Vision and language tokens are processed by a Vision-Language Expert to preserve semantic priors, while target-point, ego-state, and noisy action tokens are routed to a Trajectory Expert for motion-specific computation. On the action-token pathway, a flow-matching planner refines noisy action tokens into future waypoints and speed profiles. This design couples semantic reasoning and motion planning within a single multimodal Transformer while separating task-specific FFN computation. On Bench2Drive, VECTOR-DRIVE achieves 88.91 Driving Score and outperforms representative end-to end and VLA-based baselines. Qualitative results and ablations further validate the benefits of shared attention, semantic-aware expert routing, progressive training, and flow-based action de coding.

- **核心问题**  
  现有 VLA 自动驾驶方法主要存在两类设计范式，但各有不足：

  1. **完全共享骨干的 VLA 模型**  
     - 视觉、语言和动作预测共享同一个 Transformer / VLM 主干。
     - 优点是保留了多模态交互。
     - 缺点是语言推理和连续轨迹预测具有不同的优化目标与归纳偏置，容易产生任务干扰，影响运动规划精度。

  2. **解耦式推理-动作流水线**  
     - VLM 负责高层语义理解或推理，另一个动作模块负责轨迹生成。
     - 优点是降低了任务冲突。
     - 缺点是语义信息通常只通过文本、隐变量或高层决策传递给规划器，语义与轨迹生成之间的细粒度耦合不足，限制了复杂交互场景下的规划能力。

- **论文整体含义**  
  本文提出 **VECTOR-Drive**，试图在“完全共享”和“完全解耦”之间取得平衡：  
  - 在 **自注意力层面共享**，让视觉、语言、导航、自车状态和动作 token 持续交互；
  - 在 **前馈网络 FFN 层面分离**，根据 token 类型将其路由到不同专家，减少语言推理与轨迹预测之间的任务干扰；

### 2.2 问题形式化

论文将端到端驾驶建模为条件多模态生成问题。

- 输入：
  - 图像：`I`
  - 全局目标提示：`G`
  - 目标点：`P`
  - 高层命令：`U`
  - 自车状态：`S`

其中自车状态为：

```text
S = [v, a, θ, θ˙]
```

分别表示速度、加速度、航向角和航向角速度。

- 输出：
  - 语言推理与指令：

```text
O = {o1, ..., oK}
```

  - 未来路径：

```text
Yp = {p1, ..., p20}, pk = (xk, yk)
```

即预测 20 米范围内、每米一个 waypoint。

  - 未来速度曲线：

```text
Yv = {v1, ..., v10}
```

整体映射为：

```text
(O, Yp, Yv) = F(I, G, P, U, S)
```

### 2.3 统一 token 表示

论文将所有模态映射到统一隐藏空间：

- 全局目标提示 `G` 和高层命令 `U` 被文本 tokenizer 转换为 token；
- 图像 `I` 被视觉编码器转换为视觉 token；
- 目标点 `P` 和自车状态 `S` 通过轻量 MLP 投影为 token；
- 动作生成部分引入噪声动作状态 `Xτ`，再结合时间嵌入 `e(τ)` 转换为动作 token：

```text
ZA(τ) = φ(Xτ, e(τ))
```

所有 token 被组织为一个交错序列：

```text
Z = Interleave(ZG, ZI, ZP, ZU, ZS, ZA)
```

这样做的目的，是让语义、视觉、导航、自车状态和动作表示能够在同一 Transformer 序列中交互。

### 2.4 共享自注意力 + 语义感知专家路由

这是论文最核心的设计。

#### 共享自注意力

在每一层 Transformer 中，所有 token 首先经过共享自注意力：

```text
H~(l) = H(l-1) + Attn(LN(H(l-1)))
```

这意味着：

- 视觉 token 可以与语言 token 交互；
- 动作 token 可以读取语义、导航和自车状态信息；
- 语义推理和运动规划在注意力层面保持紧耦合。

#### 专家 FFN 路由

经过共享注意力后，token 根据语义类型被路由到两个不同专家：

1. **Vision-Language Expert**
   - 处理：
     - 全局目标 token；
     - 图像 token；
     - 高层命令 token。
   - 主要负责视觉-语言语义理解和语言推理。

2. **Trajectory Expert**
   - 处理：
     - 目标点 token；
     - 自车状态 token；
     - 噪声动作 token。
   - 主要负责运动规划相关计算。

两个专家都采用 gated FFN 结构：

```text
FFN(X) = [SiLU(XWgate) ⊙ (XWup)] Wdown
```

处理完成后，两个专家输出再被合并回原始 token 顺序。

#### 静态路由而非学习路由

- VECTOR-Drive 使用 **静态 token 类型路由**；
- 不使用 learned top-k routing；
- 因此避免了额外 router 参数和负载均衡损失。

论文消融显示，学习式路由与静态路由性能非常接近，因此静态路由已经足够有效且更简单稳定。

### 2.5 注意力 mask 设计

为了防止动作 token 中的未来动作信息泄漏到语言生成路径，论文设计了因果多模态注意力 mask：

- 噪声动作 token 之间允许双向注意力；
- 其他 token 遵循 causal mask；
- 动作 token 放在序列末尾，因此：
  - 动作 token 可以访问前面的语义和驾驶条件；
  - 语言 token 不能访问未来动作状态。

这一设计使轨迹生成能够利用语义上下文，同时避免语言生成受到动作答案泄漏影响。

### 2.6 语言生成分支

语言分支从最终的 Vision-Language hidden states 中生成链式思考和驾驶指令：

```text
p(O | Hvl) = ∏ p(ok | o<k, Hvl)
```

该分支使用 LM Head 进行自回归生成。

语言输出主要用于：

- 场景理解；
- 驾驶意图解释；
- 为运动规划提供语义先验。

### 2.7 流匹配动作规划器

论文使用 **flow matching** 生成未来路径和速度曲线，而不是简单回归。

#### 空间路径生成

给定真实未来路径 `Yp` 和高斯噪声 `ε`，在时间 `τ ∈ [0,1]` 上构造插值状态：

```text
Xτ = (1 - τ)ε + τYp
```

目标向量场为：

```text
v* = Yp - ε
```

模型学习预测从噪声到真实轨迹的向量场。

推理时：

1. 从高斯噪声初始化路径：

```text
X0 ~ N(0, I)
```

2. 通过条件 ODE 逐步演化：

```text
dXτ / dτ = vω(Xτ, τ | Cfm)
```

3. 得到最终路径预测 `Yp`。

论文使用 **Euler ODE solver**，推理时采用 **10 个 Euler steps** 生成最终 20 个 waypoint。

#### 速度曲线生成

速度分支与路径分支类似：

- 输入为噪声速度状态；
- 输出 10 步未来速度曲线；
- 与路径分支共享语义条件，但使用速度专用 Flow Head。

#### 多模态条件

Flow Head 的条件包括：

1. **隐式语义条件**
   - 来自最终 Vision-Language hidden states；
2. **显式驾驶条件**
   - 自车状态；
   - 导航表示；
   - 目标点。

因此，轨迹和速度生成都受到语义理解、导航目标和自车动力状态共同约束。

### 2.8 三阶段训练策略

论文采用三阶段训练以稳定优化。

#### Stage I：视觉-语言学习

- 冻结轨迹专家和运动头；
- 训练 Vision-Language Expert 和 LM Head；
- 目标是学习场景理解、语义推理和驾驶指令生成。

损失为语言交叉熵：

```text
Llang = -Σ log p(ok | o<k, Hvl)
```

#### Stage II：驾驶技能学习

- 冻结 Vision-Language Expert；
- 训练 Trajectory Expert、Flow Heads 和条件投影器；
- 优化路径和速度生成。

路径流匹配损失：

```text
Lpath = E || fpω(Fpτ) - (Yp - εp) ||²
```

平滑正则项：

```text
Lsmooth = Σ ||p(k+1) - 2pk + p(k-1)||²
```

速度流匹配损失：

```text
Lspeed = E || fvη(Fvτ) - (Yv - εv) ||²
```

总驾驶损失：

```text
Ldrive = λpath Lpath + λsmooth Lsmooth + λspeed Lspeed
```

#### Stage III：全参数联合微调

- 解冻所有专家和预测头；
- 联合优化语言和驾驶目标：

```text
L = Llang + Ldrive
```

该训练策略的作用是：

- 先建立语言语义能力；
- 再专门学习运动规划；
- 最后联合对齐语义推理和动作生成。

总体来看，实验较为完整，覆盖了：

- 闭环 benchmark；
- 多种强 baseline；
- 多能力行为指标；
- 关键模块消融；
- 训练策略消融；
- 解码方式对比；
- 定性案例分析。

这些实验能够较好地支撑论文关于“共享注意力 + 专家路由 + 流匹配规划”的主要结论。

### 5.3 公平性与客观性分析

较客观的方面：

- 使用公开 benchmark Bench2Drive；
- 对比方法数量较多；
- 不仅报告 DS 和 SR，也报告细粒度 Multi-Ability；
- 消融实验针对论文核心模块展开，逻辑清晰。

可能不足的方面：

- 训练数据为作者自采集的 CARLA 数据，虽然规模较大，但与 baseline 的训练数据是否完全一致不清楚；
- 部分 baseline 的实现、训练配置、模型规模是否严格对齐未详细说明；
- 只在 CARLA / Bench2Drive 仿真环境验证，没有真实道路实验；
- 没有跨 benchmark 验证，例如 nuScenes、CARLA Leaderboard 或真实车测试；
- 缺少统计显著性分析，例如多随机种子均值与方差。

## 6. 主要结论与发现

- VECTOR-Drive 在 Bench2Drive 上取得：
  - **Driving Score：88.91**
  - **Success Rate：71.82%**
  - **Multi-Ability mean：72.03%**

- 相比强 baseline SimLingo：
  - DS 提升 3.84；
  - SR 提升 4.55 个百分点；
  - Multi-Ability mean 从 67.28% 提升到 72.03%。

- 在细粒度能力上，VECTOR-Drive 在以下任务上表现突出：
  - merging；
  - overtaking；
  - braking；
  - traffic-sign compliance。

- 消融实验表明：
  - 仅使用解耦 VLM 条件规划效果较差；
  - 弱耦合语义-动作交互不如共享注意力紧耦合；
  - 单一 action-oriented expert 会损害视觉语言路径；
  - 静态语义路由与 learned routing 性能接近，但更简单；
  - 三阶段训练比直接联合训练更稳定；
  - flow matching 解码优于回归和 DDPM 解码。

- 论文最终结论是：  
  **在注意力层面保持多模态共享交互，同时在 FFN 层面进行语义感知专家分离，是提升 VLA 端到端自动驾驶闭环规划性能的有效设计。**

## 7. 优点与亮点

### 7.1 方法设计亮点

- **紧耦合与解耦的折中设计清晰**
  - 共享 self-attention 保留语义-动作交互；
  - 专家 FFN 减少语言推理和轨迹预测之间的任务冲突。

- **token 类型静态路由简单有效**
  - 不需要复杂 router；
  - 无需负载均衡损失；
  - 消融显示与 learned routing 性能接近。

- **动作 token 直接参与 Transformer 交互**
  - 轨迹生成不是简单地接收 VLM summary；
  - 而是在每一层 attention 中持续获取语义和驾驶上下文。

- **使用 flow matching 生成轨迹和速度**
  - 相比直接回归，更适合建模连续轨迹生成；
  - 相比扩散模型，推理流程可能更简洁；
  - 支持从噪声逐步细化为可执行轨迹。

- **同时预测空间路径和时间速度曲线**
  - 不仅给出几何路径；
  - 也给出速度规划，更接近可执行运动计划。

- **三阶段训练策略合理**
  - 先学习语言语义；
  - 再学习驾驶技能；
  - 最后联合对齐；
  - 消融结果支持该策略有效性。

### 7.2 实验设计亮点

- 使用 Bench2Drive 闭环评估，比纯 open-loop 轨迹误差更接近真实驾驶性能；
- 对比方法覆盖传统端到端模型、Transformer 模型和 VLA 模型；
- 报告 Multi-Ability，有助于分析模型在不同驾驶行为上的能力；
- 消融实验覆盖架构、训练策略和解码头，针对性较强。

## 8. 不足与局限

### 8.1 实验覆盖局限

- 论文只在 **Bench2Drive / CARLA** 上验证，没有真实道路实验。
- 没有在多个 benchmark 上交叉验证，泛化性仍不充分。
- 没有报告 nuScenes、Waymo、Argoverse 等真实世界数据集上的 open-loop 或 closed-loop 结果。
- 没有展示跨城市、跨天气、跨传感器配置的泛化实验。

### 8.2 数据与公平性风险

- 训练数据为作者自采集的约 300 万帧 CARLA 数据，具体采集分布和数据质量细节有限。
- baseline 是否使用相同数据、相同训练预算、相同输入模态并不完全清楚。
- 如果 VECTOR-Drive 的训练数据规模或质量优于部分 baseline，可能影响对比公平性。

### 8.3 计算成本与部署限制

- 基于 Qwen2.5-VL-3B，模型规模较大。
- 训练需要 4 张 A800 80GB GPU，对资源要求较高。
- 推理阶段每个 ODE 轨迹生成需要 10 个 Euler steps，且每一步都需要将当前噪声状态重新编码为动作 token 并经过 Transformer，这可能带来较高推理延迟。
- 论文未报告实时性、FPS 或车载部署性能，因此实际部署可行性仍不明确。

### 8.4 安全性与可靠性不足

- 没有系统评估极端长尾场景、传感器失效、遮挡、恶劣天气下的鲁棒性。
- 没有给出安全约束或形式化可验证机制。
- Flow matching 输出虽具连续性，但仍可能产生不可行轨迹，论文仅通过平滑项约束，未充分讨论动力学可行性和碰撞约束。

### 8.5 语言推理评估不足

- 虽然模型生成 CoT 和驾驶指令，但论文没有对语言质量进行定量评估。
- CoT 是否真正提升规划，还是仅作为辅助监督，目前主要通过消融间接说明。
- 生成的语言推理可能存在幻觉风险，尤其在复杂交通语义判断中。

### 8.6 方法复杂度较高

- 架构同时包含 VLM、专家路由、动作 token、flow matching、双分支路径/速度头和三阶段训练。
- 虽然每个组件都有消融支持，但整体系统实现和调参复杂度较高。
- 对于资源受限或实时性要求高的自动驾驶系统，可能需要进一步压缩和加速。
